为什么使用聚合体积？

聚合体积是为包含许多个体体积、且这些体积没有任何边界行为的工作流设计的。 由于已知光线不会在体积边界处反射或折射，渲染器就可以针对大量这类体积执行显著优化。

在上面展示的 Luca 云层场景中，几十个独立云元素被分层组合成统一的云景。聚合体积工作流允许这些云元素高效渲染，而不必把它们合并为单个体积；合并可能是渲染器外部一个代价很高的过程。

此外，聚合体积允许创建命名的体积集合，可用于特殊场景，例如把非均匀内部介质绑定到电介质表面的内部。

工作流

聚合体积的常规工作流，是通过把 Attribute "volume" "string aggregate" ["name"] 附加到一个或多个 RiVolumes 图元上来创建一个命名聚合体积，其中 "name" 是所需的聚合体积名称。这些体积图元应使用 PxrVolume 着色器来控制其体积外观。当多个体积图元都用相同名称标记了该属性后，可以把这个命名聚合体积作为 "volumeAggregate" 参数提供给 PxrPathTracer 积分器。这表示场景的 全局体积聚合 就是这个命名聚合体积。这个命名体积聚合现在会出现在渲染中。体积如何积分和采样由 PxrPathTracer 积分器上的参数控制。

如果这些体积的数据源是 OpenVDB 文件，密度信号直接来自文件，并且不会通过着色增加，那么强烈建议将新的 Attribute "volume" "int dsominmax" 设置为 [1]。这样做可以让渲染器利用密度的最小值和最大值，更快地为体积构建加速结构；这些值可以直接从 VDB 文件中读取预烘焙数据，也可以在体积加载时即时生成。这样会显著缩短首个像素时间，因为渲染器不再需要在发射光线之前预先运行密度着色预处理。如果体积密度是在着色器中以程序化方式派生的，那么 dsominmax 应设置为 0。对于需要快速进行密度操作的情况存在一个例外： impl_openvdb 支持新的控制项，可以对密度进行全局缩放和塑形，同时仍然支持 dsominmax 为 1 时的优化。

此外，如果 OpenVDB 文件细节非常丰富，并且预计文件密度可能超出实际需求，可以选择先用 vdbmake 预处理 VDB 文件以生成 mipmap。这样做 可能 降低峰值内存需求，因为渲染器只需要加载 mipmap 金字塔中较低层级的数据。另一方面，加载两个精细层级的 mipmap 金字塔往往也可能比只加载金字塔的基础层级使用更多内存；尤其是 OpenVDB 文件格式，可能会让 mipmap 映射的优势不如其他潜在体积文件格式中那么明显。

虽然 PxrPathTracer 积分器通常只能渲染一个全局体积聚合，但可以创建任意数量的聚合。这些聚合可以作为内部介质绑定到使用 PxrSurface 材质的闭合电介质表面上。与常规体积不同，聚合体积可以作为 非均匀 内部介质供 PxrSurface 完整支持。这是创建复杂效果的最佳方式，例如宝石内部的遮挡、晶球内部旋转的烟雾球，或者 Luca 在带有曙暮光光束的海洋中游泳。

能力与优势

聚合体积受到 PxrPathTracer 和 PxrUnified 积分器完全支持。如上所述，在存在数十个甚至数百个重叠体积的情况下，聚合体积设计上比以前的体积工作流更高效，并且不限制重叠体积的数量。其设计也意味着，在处理重合几何体时（例如重合的体积边界，或与表面重合的体积），聚合体积远不容易出现相关问题。

聚合体积还提供了非聚合体积没有的若干优势。当体积密度直接来自 VDB 文件且不会通过着色增加时，聚合体积通常能在复杂渲染中大幅缩短首次可判断结果时间。聚合体积还支持 mipmap 化的 体积，这可以在体积细节过高时潜在降低内存消耗。聚合体积在复杂光传输场景中也可能收敛更快，因为渲染器可以访问场景中更多体积的信息，从而对采样做出更好的全局决策。

预计聚合体积将成为未来版本 XPU 渲染器中主要支持的体积工作流。

限制

聚合体积的主要限制在于，它会为多个体积构建单个加速结构，而这个加速结构无法存储各个体积的许多属性。这是出于性能原因；如果必须构建一个了解所有这些单独属性的加速结构，就会极大拖慢对这些体积的迭代。除非经过特殊处理，否则会影响光线遍历的单个体积属性无法与聚合体积配合使用，而这种特殊处理会要求渲染器在体积中的每一步执行额外检查。举例来说，把体积分组到 trace memberships（追踪成员关系）并使用 ray subsets（光线子集） 确实可行，但可能不如常规表面渲染情况高效。

主要不起作用的单个体积属性包括：

  • 体积采样标志：PxrVolume 上控制采样技术的参数（例如 minSamples, maxSamples, equiangularWeight）在使用聚合体积时不适用。相反，体积采样控制项设置在主 PxrPathTracer 积分器中。 

若干限制也与可以跨许多不相交体积工作的渲染器优化有关。聚合体积不支持单次散射；多重散射是唯一选择，并且 multiScatter 参数对 PxrVolume 会被忽略。聚合体积也没有针对严格均匀体积进行优化。虽然渲染算法针对接近均匀的体积做了优化，但它们仍然假定聚合体积整体是非均匀的，即使某个单独元素是均匀的，也无法据此完美地对这些体积进行重要性采样。

聚合体积只支持使用 RiVolume 几何图元。

与非聚合体积相比，单个体积的高速变换运动模糊在聚合体积工作流中不会那么高效。

虽然聚合体积可以作为内部介质绑定到 PxrSurface，但不建议在复杂的嵌套电介质场景中使用聚合体积（例如雾气房间中有一块有色玻璃，玻璃中又有浑浊的有色水）。

PxrVCM 目前不支持聚合体积。PxrPathTracer 和 PxrUnified 都完全支持聚合体积。

多重散射优化受到完全支持，但 PxrVolume 上的相关控制项会在聚合体积上被忽略。相反，在聚合体积工作流中，这些控制项现在位于灯光本身。

Mipmap 映射

Mipmap 映射是一种标准技术，它会根据纹理在屏幕上的观察尺寸和角度，从金字塔中自动选择适当的纹理分辨率层级，以改善纹理走样和渲染性能。这项技术可以 选择性地 应用于由体素组成的体积数据：与 2D mipmap 金字塔层级包含由四个更精细纹素（texel）平均得到的纹素类似，3D mipmap 金字塔层级包含的每个体素都由八个更精细体素平均得到。对体积资产进行 mipmap 映射会显著增加其磁盘空间占用；如果最精细体素的大小与相机所需细节层级相匹配，渲染时内存也可能略有增加。不过，如果体素细节远比相机设置所需更精细，mipmap 映射可以节省大量 RAM 和渲染时间，因为渲染器只会加载更粗的平均体素。使用 impl_openvdb 插件作为体积图元数据源的聚合体积，可以启用 mipmap 映射，方法是将 filterWidth（过滤宽度）参数设置为大于 0 的值；该参数是渲染器计算出的标准 filterwidth 的乘数。

vdbmake 是 RenderMan 发行版随附的新工具，用于帮助创建 OpenVDB 格式的这些优化文件。 vdbmake 以包含一组网格的 OpenVDB 文件作为输入，并创建带有辅助 mipmap 网格以及这些网格的预烘焙加速信息的 OpenVDB 文件。

多重散射近似

用于近似深度多重散射体积外观的技术，依赖于根据相机光线深度改变体积的密度属性。对于体积聚合，相关控制项已经移到了灯光上。更多信息请参阅 使用聚合体积渲染云。
