Critters，使用由九个 OSL 着色器组成的着色网络

RenderMan 原生支持 OSL，但存在 下列限制。

已知限制

并非 OSL 的所有操作都受支持。目前不支持以下内容：

  • 材质闭包

  • environment

  • pointcloud_search

  • pointcloud_get

  • pointcloud_write

  • setmessage

  • getmessage

  • surfacearea

另请注意，目前只支持三种标准 OSL 纹理过滤器类型。使用不完整或不受支持功能的 OSL 着色器可能会导致程序终止。

OSL Pattern 目前支持 OSL 的许多功能，但需要特别注意它们不支持使用材质闭包。相反，该插件被设计为创建 Pattern 输出结果；这些结果随后通过把 OSL 输出连接到其他 Pattern，并把 OSL 着色器结果送入 Bxdf 和 Displace 着色器的各种输入来使用。Bxdf 和 Displace 着色器以可扩展方式取代闭包，并且允许比 OSL 闭包提供的线性组合更复杂的分层。

OSL 性能说明

在 RenderMan 中使用 OSL 时，需要注意几个性能方面的考虑。首先，默认情况下，OSL 会先为给定点执行着色器网络中所有已连接节点，然后再转到下一个点，而不是像其他 Pattern 节点那样为所有点逐个执行每个节点。如果网络中访问了大量纹理，这意味着为了维持相同性能，纹理缓存需要一次性容纳所有这些纹理，而不是只缓存单个纹理。对于 C++ Pattern 节点可以在所有点之间复用某些设置的情况，OSL 有时必须为每个点重复这些设置。

默认情况下，OSL 以“垂直”方式执行着色器，而不是像其他 PxrPattern 节点那样跨一组点执行。如果访问许多纹理，并且纹理设置会在一组着色点上连续执行，这可能导致效率下降。

不过，当前 默认 使用 “Batched” 模式。该模式会通过硬件支持的向量化指令运行多个着色点。Batched OSL 的加速效果取决于工作负载：射线发散量高、着色器相干性较低的场景提升较小；而相干性较高的场景，例如单次反弹路径或 PxrVisualizer，以及规模较大的 OSL 着色器，会看到更多提升。我们会谨慎地只在必要时执行优化，以提供整体最佳性能。

OSL 的 SIMD 版本经过优化，可使用 Intel® Advanced Vector Extensions 512 (Intel® AVX-512) 一次执行 16 个样本。它也可以使用 Intel® Advanced Vector Extensions 2 (Intel® AVX2)、Intel® Advanced Vector Extensions (Intel® AVX) 或 Intel® Streaming SIMD Extensions 4.2 (Intel® SSE4.2) 执行，但性能级别会降低。OSL 的 SIMD 版本支持 RenderMan 所支持的全部 OSL 1.8 语言功能和库调用。

它通过以下 user 选项启用

Option "user" "int osl:batched" [1]


另外，在运行时编译着色器和着色器网络的成本可能会变得显著，尤其是因为这个编译阶段会阻止多线程以最高效率运行。通常，对于渲染时间较长的场景，这项成本实际上会被摊销掉。不过，如果存在数十万个唯一着色器实例，这项成本就可能开始影响最终渲染时间。Batched 模式对启动时间的影响可能大于非 Batched 模式，但对于复杂场景，整体收益是值得的。

不过，使用 OSL 的一个好处是，它能够基于着色器的 uniform 输入参数创建优化过的原生机器代码。对于复杂网络以及为利用这些优化而编写的代码，性能提升可能非常显著。这使得你可以编写带有全部所需功能的着色节点；只要你愿意在渲染开始时支付编译成本，就可以针对每种不同用法高效地重新编译，只保留渲染时真正需要的功能。

可以通过输出不同级别的统计信息，更好地理解场景中的 OSL。请注意，更高的详细级别会影响渲染时间，因为渲染期间打印消息存在开销。因此，建议只在分析场景数据时使用，不要在速度很重要的最终渲染或外观开发测试中使用。详细级别与标准 OSL 消息级别对应：0 不输出，1 仅严重消息，2 仅严重和错误，3 严重、错误和普通消息，4 严重、错误、普通消息和警告，5 严重、错误、普通消息和信息。这些统计会写入 RenderMan 的 Advanced XML Stats 的 Plugins 选项卡。3 级及以上的统计输出会向统计中添加计时信息，并且可能比低级别对性能影响更大。4 级及以上应与 RenderMan 支持团队配合使用，并同时提供你对 Batched 模式或默认模式的使用信息。


基础 OSL 着色器编写

你可以在一个或多个文本文件中创建着色器，通常使用 .osl 扩展名。然后使用 OSL 编译器 oslc 将它们编译成带 .oso 扩展名的着色器目标文件。为方便起见，RenderMan 发行版随附 oslc。这些着色器目标文件随后可以直接作为 RenderMan 中的 Pattern 着色器使用。

我们将通过下面这个简单着色器示例，说明编写 OSL 着色器的基础知识，以及该着色器如何集成到 RenderMan 中：

shader myMixShader (color a = color(1,0,0),
                    color b = color(0,1,0),
                    float mixVal = .5 [[ int lockgeom=0 ]],
                    output color resultRGB = color(1,1,1) ) {
    float finalMix = clamp(mixVal,0.0,1.0);
    float doMix = 1.0;
    getattribute("allowMixing", doMix); 
    float mixExponent = 1.0;
    getattribute("builtin", "curvature", mixExponent);

     

    if (doMix != 0) {
        finalMix = pow(finalMix, mixExponent);
        float mixRemainder = 1.0 - finalMix;
        resultRGB = a * finalMix + b * mixRemainder;
    } else {
        resultRGB = a;
    }
}

第一行包含着色器类型、着色器名称以及参数列表的开头。RenderMan 只支持通用的 “shader” 着色器类型，不支持其他渲染器使用的更具体的 “surface”、“displacement”、“light” 和 “volume” 着色器类型。 

接下来是参数列表。每个参数都必须有默认初始化值。可以在双中括号 [[ ]] 中指定元数据，为渲染器和其他内容创作应用程序提供关于这些参数的额外信息。在这个例子中，元数据 int lockgeom = 0 表示 mixVal 的值可以从附着到几何体上的 mixVal 变量插值而来。目前，RenderMan 允许附着 OSL 支持的所有类型变量，但 int 和结构体除外。

着色器的结果，在本例中是 color resultRGB，会通过声明为输出参数，连接到着色网络的其余部分。这些输出可以连接到 Bxdf 和 Displace 着色器输入，也可以连接到其他 OSL 或 C++ Pattern 着色器输入。RenderMan 提供这个接口来替代标准闭包，因为标准闭包在 RenderMan 中不受支持。

这个着色器最先做的事情之一，是声明一个局部变量 finalMix。它将 OSL 函数 clamp() 应用于 mixVal 输入，并把结果存储在 finalMix 中。OSL 中的输入不可写，因此必须把它存到第二个临时值中。不过不必担心这个额外变量，OSL 的渲染时优化步骤很擅长消除这类变量，所以可以放心使用这种临时变量，让着色器代码更清晰。

RenderMan 支持几乎所有 OSL 内置标准函数，例如 clamp()。可以参考“已知限制”查看不受支持的函数列表，并参考 OSL 文档了解受支持函数的用法。你可以用 OSL 编写自己的函数，但目前没有接口可以直接链接用其他语言（例如 C++）编写的函数。为此，需要编写 C++ Pattern，并通过 C++ 与 OSL 之间的参数列表传递值。有关混合 C++ 与 OSL Pattern 的详细信息和限制，请参见下文相应章节。

下一处值得关注的是 getattribute() 调用。它允许按属性名称获取当前几何体片段上设置的属性。它还支持对象名称，用来指明属性应该来自何处。RenderMan 支持使用 “global”、“primvar” 和 “builtin” 对象名称，也支持不提供名称或使用空字符串 “”。不提供名称或使用空字符串会获取当前几何体片段的 RenderMan 属性。使用 “global” 会获取选项。由于 RenderMan 属性和选项被划分为不同类别，因此获取它们时需要在属性名称前加上类别和冒号。例如，Attribute “dice” “float micropolygonlength” 可以使用 getattribute(“dice:micropolygonlength”) 获取。使用 “primvar” 只会获取存储在几何体自身上的值，它们独立于 RenderMan 属性状态。使用 “builtin” 会获取 RIS 提供的额外内置变量，这些变量不是 OSL 默认全局变量的一部分。

XPU 允许 float 与 int 自由互转，也允许从 float[3] 转换为 color 和 point 类型，并在使用 getattribute() 进行查找时返回 1。RIS 不允许这种转换，会返回 0。

混合 C++ 与 OSL Patterns（仅 RIS）

为了让 C++ 与 OSL Pattern 相互交互，需要告诉 OSL Pattern 不要假定它们的输入是常量，因为这些输入可能由 C++ Pattern 输出提供。如果要允许把非 OSL 输出连接到 OSL 输入，必须为该输入设置元数据标志 [[ int lockgeom = 0 ]]。


使用 OSL 的一个优势是，大型 Pattern 网络会在渲染器使用它们时即时整体编译。这意味着编译器可以利用着色器的当前设置，并优化掉任何给定对象未使用的部分。不过，OSL 编译器无法“看到”基于 C++ 的 Pattern 节点内部，因此混入它们可能会削弱一部分优化过程。此外，OSL Pattern 会被分组并一起执行，而 C++ Pattern 会逐个执行。如果一个 C++ Pattern 位于网络中间，RenderMan 可能必须把 OSL Pattern 拆成两组：必须在 C++ Pattern 之前执行的一组，以及必须在其之后执行的一组。遗憾的是，RenderMan 目前无法在这两组节点之间传递隐式导数，因此如果你的纹理过滤或凹凸贴图依赖上游信号的导数，混入 C++ 节点可能会打断这些导数。因此，不建议在网络中间交错插入 C++ 节点。不过，它们可以放在所有 OSL 节点下游而不会造成损害，也可以完全放在所有 OSL 节点上游，通常不会产生多少意外影响（除了纹理过滤和凹凸贴图无法工作）。

OSL 实现说明：

在 PRMan 中将 primvar 绑定到几何体相当简单：只需在着色器中使用 metadata [[int lockgeom = 0]]标记参数即可。这会告诉着色系统将该变量视为会随几何体变化的输入，并且如果该变量存在于几何体上，就会把它绑定到着色器中。另请注意，可以绑定其他 Pattern 节点的输出，OSL Pattern 节点会自动把一个 Pattern 节点的输出绑定到 OSL 着色器的参数。同样，OSL 着色器的输出也可以被其他节点引用，无论这些节点是 Pattern、Bxdf 还是 Displace 节点。只需按名称引用它们即可。

[[int lockgeom = 1]] 大致表示：“允许 OSL JIT 锁定几何绑定，并假定该着色器参数为常量值”。这使常量传播能够更积极地工作，但也会阻止该着色器以随几何体变化的输入执行。如果要接入会变化的 C++ 输入，就需要在 JIT 阶段强制这一行为；不过在这种情况下，prman 此时已经会为你处理。

第二部分表示，如果你已经锁定几何绑定，即 lockgeom=1，那么 primvar 到着色器输入的自动绑定将不会发生。这是默认行为，因此如果某个输入是 'color primColor'，并且部分几何体上有名为 'primColor' 的顶点变量，而你期望它被连入着色器，就需要设置一个 [[int lockgeom=0]] 元数据标签。

有几个选项可以控制 OSL 在 RenderMan 中的行为和使用方式。

  • Option “user” “int osl:verbose” [0-5] - 控制 RenderMan 回显的 OSL 消息详细级别。0 表示关闭，1-5 每级都会向打印内容中增加一类 OSL 消息。这些类别为：1 - SEVERE，2 - ERROR，3 - MESSAGE，4 - WARNING，5 - INFO。

  • Option “user” “int osl:statisticslevel” - 这类似于 “osl:statistics:level” 属性，但它控制 RenderMan 为整个 OSL 着色系统集成所跟踪的统计级别。

请注意，有若干 OSL 属性可在 RIB 文件中设置。支持以下属性：

  • Option "user" "int osl:debug" [0]

  • Option "user" "int osl:optimize" [1]

  • Option "user" "int osl:lockgeom" [1]

  • Option "user" "int osl:debug_nan" [0]

  • Option "user" "int osl:debug_uninit" [0]

  • Option "user" "int osl:compile_report" [0]

  • Option "user" "int osl:statistics:level" [0]

有关其用法的详细信息，请参考 OSL 文档。你也可以访问以下链接，了解当前版本的 OSL： OpenShadingLanguage。

使用 trace() shadeop：

包含了对 trace() shadeop 的有限支持。trace() shadeop 不能在置换、presence 和 opacity 着色期间调用。trace() shadeop 也仅限于取回几何值，因为我们不允许递归着色，所以可选的 “shade” 参数会被忽略，并始终假定为 0。在 trace() 之后，getmessage() 可以取回 “hit”、“hitdist” 和 “geom:name” 值，以及命中几何体上的 primvar 值。

访问渲染器信息

可以通过使用 getattribute 调用并把 “rendererInfo” 作为对象，获取当前正在运行的渲染器版本信息。

string renderer = "unknown"; 
getattribute("rendererInfo", "renderer", renderer);
if (renderer == "PhotoRealistic-RenderMan") ...

在 rendererInfo 对象中支持三个查询： “renderer”, “version”，以及 “versionCode”。

  • “renderer” 会返回字符串 “PhotoRealistic-RenderMan” ，表示正在 RenderMan 内运行。

  • “version” 会返回 渲染器的当前版本字符串。

  • “versionCode” 是一个 int[4] 数组，包含版本信息； versionCode[0] 始终为 3， versionCode[1] 为发行版的主版本号（例如 RenderMan 26 中的 26）， versionCode[2] 为发行版的次版本号，并且 versionCode[3] 为构建号。

访问着色上下文

可以访问 C++ 着色器在 RixShadingContext 上可见的 RenderMan 专用上下文，方法是通过 getattribute 调用并把 “context” 作为对象。 

string mymode = "unknown"; getattribute("context", "shadingMode", mymode);
if (mymode == "displacement") ...


“shadingMode” 的值对应 RixShading.h 中的 RixSCShadingMode：

  • presence

  • opacity

  • scatter

  • volumeTransmission

  • volumeScatter

  • emission

  • bake

  • displacement

如果希望获得类似网格式着色，而不是光线命中式着色，则不要使用 shadingMode，而是可以查询一个返回 1 或 0 的布尔上下文属性，其中包括：

  • eyePath (*)

  • lightPath (*)

  • primaryHit

  • missContext (*)

  • reyesGrid

标记为 (*) 的查询在 XPU 中始终返回 0。

隐式导数

OSL 使用隐式微分按需跟踪变量的导数。RenderMan 将导数的 x 和 y 空间映射到底层几何体的隐式 u 和 v 空间。隐式导数用于控制纹理过滤，也可用于为凹凸贴图创建新的着色法线。不过，只有当来自渲染器的原始值已预置导数时，这才有效，因为导数必须与这些初始值中的变量一起计算，并经过其他数学运算变换，直到在 texture() 或 calculatenormal() 调用中使用。导数主要来自着色器全局变量 P，prman 确实会使用适当导数对这个着色器全局变量进行预置。此外，在几何体上插值的 primitive variable，如果通过 getattribute("primvar","primvarname",value)访问，或者直接通过 getattribute("primvarname", value)  访问，也会提供导数。 

与 C++ 着色上下文中可用内置变量对应的值，可以通过 getattribute("builtin", "builtinname", value) 调用访问，但它们 不会 提供导数，因为底层 C++ 实现并不提供它们。这类值最常见的需求，是访问着色器全局变量 P 的预置换值的导数。C++ 着色器作者可能习惯把这个值作为内置变量 “Po” 获取，但对于 OSL，建议使用 getattribute("P", value) 它会从底层几何体重新插值 P，并返回预置换值及其关联导数。

全局选项、属性、constant/uniform primitive variable 以及 “context” 变量的隐式导数也会平凡地为 0，因为它们不会在表面上变化。

字符串替换

在基于 CPU 的渲染中，着色器内的字符串操作可能开销很高；在基于 GPU 的渲染中，如果字符串操作不能完全在编译时完成，则会被禁止。为帮助处理这一点，RenderMan 支持在纹理文件名中替换以下标记

  • <udim> - 将替换为与提供的纹理坐标对应的 UDIM tile（贴片）

  • <u>,<v>,<U>,<V> - 将分别替换为纹理坐标的整数部分，截断或向上取整

  • <primstr:primvarname> - 将替换为具名 constant string primitive variable 的值，例如在一个带有 "const string model" ["apple"] 的苹果上，"/assets/<primstr:model>/diffuse.tex" 这样的资产名称会扩展为 "/assets/apple/diffuse.tex"

LPE 匹配与 Throughput

自定义 LPE 匹配和吞吐量值可以在 OSL 中通过 getattribute() 访问。该功能允许着色根据自定义 LPE 指定的路径变化，甚至根据这些路径的吞吐量变化。例如，着色器可以根据射线是否折射穿过某个特定对象而作出响应并返回不同值。

getattribute("lpe:match", "<customlpename> | <customlpeexpression>", val);

该 lpe:match 查询会在当前着色事件之前的射线路径 匹配该 LPE 时返回 1，否则返回 0。 val 预期类型为 float。

getattribute("lpe:throughput", "<customlpename> | <customlpeexpression>", val);

该 lpe:throughput 查询会返回射线路径的总路径吞吐量；如果当前着色事件之前的射线路径不匹配该 LPE，则返回 0。 val 必须是 color。

在这两种情况下，传给 getattribute() 的第二个参数可以是自定义 LPE 名称，也可以是自定义 LPE 表达式。一个表达式是典型的 LPE 字符串，例如 "C<TS>.*" 。一个名称（name）是一种方便且对艺术家友好的方式，用来引用一个 表达式，并且需要设置流水线，使 Options 被渲染器知晓，从而建立该名称与表达式的关系。例如，如果向渲染器发出以下 Option 调用：

 Option "lpe" "string customLPE:myDiffuse" ["C<.D>.*"] 
   "string customLPE:myReflection" ["C<RS>.*"] 
   "string customLPE:myRefraction" ["C<TS>.*"]

后续 getattribute() 调用便可以自由使用 myDiffuse, myReflection，以及 myRefraction 作为对应 LPE 表达式的同义名。

请注意，在 RenderMan 中，Pattern 着色是在 光照之前 进行的，因此你的 LPE 表达式需要考虑这一点。特别是，如果你习惯在 AOV LPE 表达式末尾使用 L 标记（用于捕获一条路径直到并包含光源为止的贡献），则需要移除 L，才能将这类表达式用于 lpe:match 或 lpe:throughput。请注意，上面 myDiffuse 的定义没有以 L 结尾。
