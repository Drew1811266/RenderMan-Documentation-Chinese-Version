tractor-spool

Source: https://rmanwiki-27.pixar.com/space/TRA/22184276/tractor-spool
Downloaded: 2026-06-15T03:49:12+00:00

提交器会把作业脚本送入 Tractor 作业队列，以便在渲染农场上处理和分发。作业脚本是文件，通常采用 Tractor 作业脚本格式（也称为 Alfred 脚本）。这些脚本通常由其他应用程序编写，例如 "RenderMan for Maya" 插件，或各工作室的专用作业提交脚本。对于简单任务，tractor-spool 本身可以根据你想发送到渲染农场的简单命令为你创建作业。如果你已经有一组要提交的 RenderMan RIB 文件，tractor-spool 也可以根据这些文件名自动为你创建作业（参见 "--ribs" 示例）。

用法：

tractor-spool [options] [jobfile ...]

tractor-spool [options] -c [/path/]appname appArg1 appArg2 ...

tractor-spool [options] --ribs frm1.rib frm2.rib ...

tractor-spool [options] --rib frm1_prologue.rib frm1.rib...

tractor-spool [options] --jdelete=JOB_ID --user=JOB_OWNER

其中 "jobfiles" 是描述待执行工作的作业脚本。可以给出多个作业文件名；它们会按顺序提交到作业队列。"-c" 和 "--ribs" 变体会根据命令或 RIB 参数为你创建作业脚本，并提交这些脚本。

术语说明：作业提交有时也称为 spooling（提交），这是类比 print spooling（打印假脱机）或传统批处理作业系统的说法。


选项说明
--version显示程序版本信息，然后退出。
-h, --help打印用法摘要，然后退出。
-v添加详细状态消息。
-q

静默模式，不打印状态信息。

--engine=HOST[:port]中央 Tractor Engine 服务的 Hostname[:port]（主机名[:端口]），默认值为 tractor-engine:80。主机名 "tractor-engine" 通常是本地网络管理员添加的 DNS 别名，指向当前运行该服务的实际主机名。这个别名简化了到 tractor-engine 进程的所有连接，因为默认都会尝试该名称。默认端口为 80，并且必须与 tractor-engine 实际监听的端口号匹配（启动 tractor-engine 时选择）。如果未给出此选项，提交器会先查找名为 TRACTOR_ENGINE 的环境变量作为 host:port 规格，然后才使用默认值。

-c
--command

-C

把带有给定参数的单个命令发送到渲染农场（而不是发送预先写好的作业脚本）。它会收集所有剩余命令行参数，然后创建一个单任务 Tractor 作业，把给定命令发送到远程 tractor-blade。 注意：这必须是给出的最后一个选项，因为 -c 后面的每个 shell token（命令行标记）都会成为给定应用程序的单独参数。例如：

tractor-spool --service=required_host_type -c echo hello world

用于启动给定命令的搜索路径和其他环境设置由每台主机上的 tractor-blade 服务器控制。既有默认路径和设置，也有可配置的站点定义运行环境包（"environment packages"），例如某个制作项目（show）的所有锁定设置；这些可用此处描述的其他选项选择，例如下面的 --envkey。

如果命令需要特定类型的远程服务器，请在 -c 之前添加 --service=name 规格。服务名称抽象在 blade.config 中定义；如果目标是某个特定主机，也可以直接使用主机名；如果该类别中的任意主机都可接受，则可使用配置档名称。

有几个内置可移植性别名（"portability aliases"）可用于常见命令，这些命令通常用于测试或诊断场景。这些别名以等号开头，以便和具体的实际可执行文件区分。最有用的可能是：

tractor-spool --service=somehost -c =printenv

使用 -c 生成常规 RemoteCmd 格式，使用 -C 强制生成 local Cmd。

-r, --ribs把剩余文件名参数视为独立 RIB 文件，并作为不同远程 tractor-blade 上的 INDEPENDENT（独立）prman 进程渲染；创建一个多任务 Tractor 作业来处理这些渲染任务。
-R, --rib把剩余文件名参数视为 RIB 文件，并使用远程 tractor-blade 上的 SINGLE（单个）prman 进程渲染，也就是说 prman 会连接所有 RIB 文件进行渲染；创建一个单任务 Tractor 作业来处理该渲染。
--jobcwd=DIRNAME从此作业启动命令时，Blade 会尝试 chdir 到指定目录；默认值就是运行 tractor-spool 时的当前目录。
--projects=PROJECTS项目归属列表，例如 'TheFilm lighting'，可配合 Limit Allocations 管理资源池。
--priority=FLOAT已提交作业的任意正浮点优先级；优先级值较高的作业会先处理。工作室通常会制定如何使用优先级的约定。
--title="words"与 -c 和 -r 一起使用，用于更改自动生成作业的默认作业标题。
--svckey=SVCKEY为已提交作业中的 Cmds 指定额外作业级服务键限制；这些键会与 Cmds 自身已有的任何键做 AND。与 -c 或 --rib 选项一起使用时，它会覆盖 "PixarRender"，作为用于为这些 Cmds 选择匹配 Blade 的唯一服务键。
--cmdsvckey=CMDSVCKEY
--cmdservice=CMDSVCKEY

与 -c 或 -r 选项一起使用，用于为每个命令指定唯一服务键表达式；该表达式会受 RANGE 和 ITER 替换影响。

--remotecleankey=KEY把 "local" clean-up Cmd 转换为 RemoteCmds。在某些上下文中，用户可能无法控制作业生成的细节，因此无法以其他方式强制清理命令远程运行。"local" Cmd 必须在提交该作业的同一主机上运行，例如艺术家工作站，这意味着用户必须在桌面机上保持 tractor-blade 进程运行，以处理这些 Cmd 清理命令。相比之下，RemoteCmd clean-up 会在第一台可用渲染农场机器上运行。给出此选项时，默认也会把 Job -whenerror 和 -whendone 块转换为 RemoteCmds，而不是 local Cmds。此提交选项还要求给出 Blade 服务键名，用于指定运行清理命令的合适 Blade 类型。这个转换选项是一种变通方法，适用于作业脚本生成器本身无法更新为在清理块中直接使用 RemoteCmd、并且需要把 Job -whendone/error 块转换为更通用 -postscript 块的情况。
--envkey=ENVKEY与 -c 和 -r 一起使用，用于更改环境键，该键会配置应用到这些自动生成作业脚本的环境变量；默认值：None。
--range=RANGE

为给定范围中的每个整数创建一个带独立任务的作业。范围可写成 '1-10' 或 '1-4,7,9,16-20'。 仅适用于通过 --command (-c) 选项指定的命令模板。每个新任务都包含一份命令模板副本，其中单词 'RANGE' 会被替换为范围序列中的下一个整数。也支持并推荐使用 Python 字符串 format() 语法，字段名为 {RANGE}。 若要插入类似 00041、00042 的前导零填充，请使用 {RANGE:0>NN}，其中 NN 是数字位数。 （另请参阅 --ribs，可轻松提交一系列已有 RIB 文件。）

例如：

tractor-spool --no-spool --range 11-13 -c cp "/src/foo.{RANGE:0>5}.rib" /dst/bar.RANGE.rib

会显示以下作业文件：

Job -title {cp ...} -subtasks {
  Task -title {cp} -service pixarRender -cmds {
     RemoteCmd {cp /src/foo.00011.rib /dst/bar.11.rib}
  }
  Task -title {cp} -cmds {
     RemoteCmd {cp /src/foo.00012.rib /dst/bar.12.rib}
  }
  Task -title {cp} -cmds {
     RemoteCmd {cp /src/foo.00013.rib /dst/bar.13.rib}
  }
}
 

注意：{ 和 } 字符对许多命令行 shell 有特殊意义。上面使用双引号，是为了确保 { 和 } 会被传入 tractor-spool；这对大多数 shell 都是必需的，包括 cmd.exe 和 bash，以及 csh 等。

--range-formatting=both|sfmt|text

默认设置 'both' 会尝试把范围 整数替换进 -c 命令参数：先使用 Python 字符串 format() 风格的 {RANGE} 表达式，然后对任何剩余的 出现位置中的 单词 RANGE 做简单替换。此处使用值 'sfmt' 表示只处理字符串格式化 {RANGE} 表达式；使用值 'text' 表示只做简单 文本替换。

--itervalues=LIST 

为列表中的每一项创建一个任务；该列表以逗号分隔项指定，并在命令字符串或服务键表达式中把 ITER 替换为当前值；例如 red,green,blue 或 'red house,green lawn,blue sky'。

--iterfile=FILENAME

为文件中的每一项创建一个任务；该文件指定按行分隔的项列表。

--task-title=TTITLE任务标题模式，用于使用 RANGE 或 ITER 构造的自动生成作业。
-T [MINSECS,]MAXSEC
--runtime-bounds=[MINSECS,]MAXSECS

与 -c 一起使用，为命令在 Blade 上的耗时添加最小和最大边界。以 'min,max' 或 'min-max' 形式给出秒数范围。如果已启动命令的运行耗时短于或长于给定边界，则标记为 Error（错误）状态。
超过最大时间的命令会被终止。如果只给出一个值，则它指定最大运行时间。最大时间为 0（零）表示无限制。

--limit-tags=TAGS

指定要添加到每个命令的限制计数标签；会受 RANGE 和 ITER 替换影响。

--tier=TIER分派层级分配，用于特殊作业。
--paused以暂停状态提交该作业，表示在以后把优先级从负数改为正数之前，不会从该作业启动任何任务。
--aftertime='MM DD HH:MM'把作业开始延迟到给定日期，格式为 'MM DD HH:MM'。
--afterjid=JID

把作业开始延迟到给定作业完成，指定为一个或多个 JID。

--nrm让自动生成的 --ribs 作业在本地 Blade 上使用 netrender，而不是在 Blade 上用 prman 直接渲染；当指定的 RIB 文件无法被远程 Blade 直接访问时使用。
--spool-wait

阻塞，直到作业完全提交（到 tractor-engine 2.0+）。

--status-json

在 stdout 上以 JSON 格式字典打印提交确认消息（或拒绝消息），而不是纯文本。

--status-plain

以人类可读纯文本打印提交确认消息；这是默认设置。

-A, --in-alfred（Alfred 输入格式选项）

表示正在提交的作业文件是 Alfred (tcl) 格式，这是默认值。

-J, --in-json

表示正在提交的作业文件格式为 Tractor 兼容 JSON。

--alfescape

处理作业文件时假定它们期望向后兼容的 Alfred 风格两级去引号/替换。

--alf-argv-subst

只对 Cmd 可执行文件参数应用第二次替换轮次。

--maxactive=MAXACTIVE

限制该作业中并发活动命令的最大数量。

--user=LOGIN与此作业关联的用户（login）；默认值为执行提交脚本的用户名称。站点配置策略可能禁止这样设置。
--jdelete=JDEL_ID
--jretire=JDEL_ID		从活动队列中删除请求的作业。
 --haddr=HADDR 提交主机的主机地址选择，通常不需要。只有在提交主机有多个网络接口，或者 tractor-engine 可能看到提交脚本与本地运行的 tractor-blade 节点的不同地址时，才可能需要它，且通常只针对 "local Cmd" 变体作业。

-p PASSWD
--passwd=PASSWD

提交作业的用户的 Tractor 登录密码（如果启用了 Engine 密码）。

--configfile=CONFIGFILE包含运行 tractor-spool 的用户的缓存登录名和密码数据的文件。
--parse-debug

解析入站作业文本并报告错误；该作业不会提交给 Engine 处理。

-o
--review
--print-alfscript

打印作业 alfscript 而不是提交它；通常用于查看或保存 tractor-spool 自身基于其他参数生成的作业文本，而不是读取现有作业文件时使用。
