配置文件说明

Tractor 的设置位于 .config 文件中。Tractor 安装区域的 "config" 目录保存所有带默认设置的文件。针对工作室的这些文件覆盖应放在安装区域外部的站点专用目录中，并通过 --configdir (path) 启动选项将其位置传给 tractor-engine。记得根据站点需求为这些文件设置适当的读写权限。最常自定义的文件包括：

tractor.config：tractor.config 包含 Tractor 的基础配置和策略设置，例如许可证服务器位置、作业数据库位置以及基本调度模式。详情请参阅文件自身注释。crews.config：crews.config 文件包含 crew（用户组）定义和基本作业队列编辑权限。Crews 只是命名的 Tractor 用户列表。权限可以授予单个用户，也可以授予指定用户组的所有成员。详情请参阅文件自身注释。blade.config：blade.config 文件控制渲染农场上主机的基本调度属性。主机配置使用 blade profile（blade 配置档）描述，每个配置档可应用到多个 blade 服务器。limits.config：limits.config 文件指定 Tractor 在渲染农场上启动的并发应用数量限制。Limit "tags" 是与每个应用程序关联的任意关键字，Tractor 会为每个 tag 维护计数。例如，如果整个工作室中某个应用程序只有有限数量许可证可用，则 limits.config 中的条目可以确保 Tractor 不会超过该程序的同时启动数量。
 

站点专用配置覆盖

在 安装期间，engine 启动服务会被调整为将 engine 指向你维护自定义配置文件的目录。最好将修改内容保存在某个目录中，且该目录位于 Tractor 特定版本的安装目录 之外，这样安装新版本 Tractor 时可重复使用这些修改，并且重新安装产品时不会丢失更改。

从 Tractor 2.0 开始，你的配置目录可以更加 稀疏， 只包含你实际想修改的文件副本。不在配置目录中的文件将从 Tractor-2.0/config 安装区域中的原始默认文件读取。

此外，tractor.config 和 db.config 这两个文件会应用特殊处理 -- 你在 --configdir 目录中的这些文件副本可以缩减为 仅包含 你实际想修改的少数键值对。对于这两个文件，engine 会先从已安装发行版的 config 目录加载默认文件，然后加载 --configdir 中的版本，并将“增量”覆盖到结果字典中。其他 config 文件 不会 以这种方式处理，以避免删除或部分替换 嵌套库存条目 所带来的复杂性。
 

选择一个用于 "EngineOwner" 的登录账号

重要 -- 如果你打算以 root 身份启动 tractor-engine 进程（这是一个有用的默认做法，因为它可以获取用于连接请求的 "protected" 端口 80），那么你必须编辑 tractor.config 中名为 "EngineOwner" 的条目，并指定一个非 root 账号，由它在启动后拥有引擎和数据库进程。数据库应用不允许自己以 root 运行，因此引擎会负责以你在 tractor.config 中指定的登录账号启动它。许多工作室会专门创建一个账号来拥有这类服务进程。
 

"@merge" 便捷函数

engine 的配置文件解析器支持一个简单的 "@merge" 机制，用于把 JSON “子文件”加载到参数块中。例如，在 blade.config 中，如果多个 profile 共享相同的 environment handler 定义，可以把公共文本放到单独文件中，然后“合并”到每个 profile 定义中。

解析器会检查值字符串中类似 "@merge('file.json')" 的文本，并将该字符串替换为指定文件的解析内容。注意： @merge 调用必须位于常规 JSON 字符串内部；也就是说 @merge 调用必须位于常规 JSON 字符串内部， 也就是必须用双引号包起来。
 

Tractor 网络端口使用

下表列出了各个 Tractor 组件使用的网络服务端口。这些 监听端口 允许位于不同主机上的 Tractor 各个部分相互通信。网络 防火墙 默认可能阻止这些连接，因此你可能需要为这些端口添加“防火墙例外”（有时称为 "pinholes"）。并不是每个端口都需要在每台机器上设置例外！

主机端口用途
engine80/tcp主作业队列在此端口上监听来自所有客户端组件的作业提交和 Dashboard 连接。可以用 "tractor-engine --port=NNNN" 或 tractor.config 中的 ListenerPort 指定备用端口（以及接口）。
engine80/udpengine 的 UDP 监听器接收来自 blade 的进度更新，随后收集这些更新并发送到 Dashboard 会话，用于绘制各个任务的进度条。此端口号始终设置为与上方主 TCP 监听器相同的数值。
engine9180/tcp命令输出的默认日志机制会将每个命令的输出流发送到 engine 主机上运行的集中式日志归档进程。拥有大量 blade 或繁重日志需求的工作室应改用 直接写入方式（direct writes）。
engine9876/tcpPostgreSQL 数据库服务器端口。注意：正常使用中，此端口只在 "loopback" 接口上供 tractor-engine 自身的私有连接使用。通常不需要来自外部主机的任何连接，因此也无需为此服务创建防火墙 pinhole。
blade9005/tcpTractor-blade 在此端口上监听多种异步连接。对于简单调度，这不是严格必需的，因为 blade 只会向 engine 发起出站请求来请求任务分配。不过，中断正在运行的作业需要异步连接，某些 blade 状态报告也需要。使用 "tractor-blade -L NNNN" 参数可指定备用端口，blade 会将该值注册到 engine。使用 "-L 0"（零）可让操作系统为此用途分配一个随机未使用端口；但当需要防火墙 pinhole 时，这种做法不合适。
engine 及其他主机239.255.255.250 1900/udp这个 "engine discovery" 组播方案（SSDP）使用给定组播地址和端口，既用于请求当前 engine 的可用性通告，也用于监听这些通告。这些广播是可选的，在大型工作室中通常并不受欢迎。
许可证服务器9010/tcpPixarLicenseServer 进程在此端口上监听许可证签出请求。可以使用其他端口号，但需要从 客户支持获取更新后的许可证。
