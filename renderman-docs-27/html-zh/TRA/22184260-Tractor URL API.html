Tractor URL API

Source: https://rmanwiki-27.pixar.com/space/TRA/22184260/Tractor+URL+API
Downloaded: 2026-06-15T03:49:12+00:00

各个 Tractor 组件使用 HTTP 请求进行通信。大多数查询响应都是 JSON 格式的数据字典。

下面汇总了查询 URL 及其响应：

  • ENGINE（引擎） 下面的 "hostname:port" 表示 tractor-engine 的主机名和端口。
  • BLADE（执行节点） 下面的 "hostname:port" 表示命令所指向 blade 的主机名和端口。每个 tractor-blade 进程都会在一个特定且可能唯一的端口号上监听管理控制请求。
  • NOTE（注意） - 许多 edit（编辑） 请求都需要使用 Tractor 当前的简单认证机制。这些请求 URL（或其表单数据）必须包含形如 "&tsid=123abc-456" 的登录会话标识符。实际的会话专用 tsid 值由 monitor 在初始登录交换期间生成。
  • JSON 回复格式 - 请注意，许多发往 Engine 的 URL 指令会让 Engine 生成 JSON 格式的 HTTP 响应正文。默认情况下，响应 HTTP 头会按照典型 JSON 约定包含 "Content-Type: application/json"，这也是许多 JavaScript 代理或其他请求方所期望的格式。不过，如果你手动把下面某些查询输入到 Web 浏览器的 URL 字段中，浏览器可能会 显示 返回的 JSON 会显示为难以阅读、没有结构的纯文本块。在这些手动、非脚本化场景中，可以向 URL 请求添加一个额外参数，使响应在浏览器窗口中以美化格式显示：在请求 URL 中加入 "&fmt=pre" 会把响应 Content-Type 设为 "text/html"，并把整个响应放入 HTML "<pre>" 块中，从而保留结构化格式，便于人工阅读。请注意，这种格式不适合用于需要实际 JSON 文本的脚本化场景。
     

Tractor Dashboard 用户界面

要启动 Dashboard 会话，请将 Web 浏览器指向： 

http://ENGINE/tractor/dashboard/
 

队列控制

登录并创建认证会话；某些类型的控制操作需要该会话：

http://ENGINE/Tractor/monitor?q=login&user=UUUU
（注意：在使用登录密码的站点，还需要额外参数 "c=CCCCC"，用于提供站点指定的密码哈希或加密值。）

停用一个作业，将其从活动分配队列和 Dashboard 中删除：

http://ENGINE/Tractor/queue?q=jretire&jid=1234&tsid=XXX
（注意：tsid 值是先前 q=login 请求返回的认证令牌。）

如果底层作业数据仍然存在，则恢复先前删除的作业：

http://ENGINE/Tractor/queue?q=jrestore&jid=1234&tsid=sss

重试某个作业中所有出错的任务：

http://ENGINE/Tractor/queue?q=jretry&jid=1234&tsubset=error&tsid=sss

停止并重新排队某个作业中当前所有活动任务：

http://ENGINE/Tractor/queue?q=jretry&jid=1234&&tsubset=active&tsid=sss

重试作业中的某个指定任务，例如 tid=22：

http://ENGINE/Tractor/queue?q=jretry&jid=1234&tsubset=22&tsid=sss

停止并重新排队某个 Blade 上当前所有活动任务：

http://ENGINE/Tractor/queue?q=ejectall&blade=host/addr&tsid=sss

跳过作业中的某个指定任务，允许其父级继续执行：

http://ENGINE/Tractor/queue?q=tskip&jid=1234&tid=22&tsid=sss
（注意：使用 "tid=error" 可以跳过该作业中的所有错误任务，使用 "tid=active" 可以中断所有活动任务并越过它们继续执行。）

完全重新启动（重新提交）整个作业：

http://ENGINE/Tractor/queue?q=jrestart&jid=1234&&tsid=sss

中断正在运行的作业，终止活动命令并暂停该作业以阻止进一步调度：

http://ENGINE/Tractor/queue?q=jinterrupt&jid=1234&tsid=sss

中断某个指定任务，使该任务进入错误状态，同时让其他任务继续运行：

http://ENGINE/Tractor/queue?q=jinterrupt&jid=1234&tid=9876&tsid=sss
（注意：tid 参数也可以指定以逗号分隔的任务 ID 列表。）

作业队列信息查询

列出先前提交过作业的用户：

http://ENGINE/Tractor/monitor?q=users

列出某个特定用户当前排队的作业：

http://ENGINE/Tractor/monitor?q=jobs&owner=name&filter=filtername&tsid=sss
（注意：owner 和 filter 限制条件是可选的，但未过滤的
列表可能很大。只有指定 filter 时才需要 tsid 参数。
添加 "&metadata=0" 还可以抑制 metadata 块。）

获取某个特定作业的任务树：

http://ENGINE/Tractor/monitor?q=jtree&jid=1234

列出与某个任务关联的命令：

http://ENGINE/Tractor/monitor?q=taskdetails&jid=1234&tid=11

获取某个特定任务的日志 URL：

http://ENGINE/Tractor/monitor?q=tasklogs&owner=name&jid=1234&tid=11

编辑作业属性

常规作业属性编辑 - 作业的

jobinfo.json 字典中列出的属性条目可以使用以下形式的 HTTP URL 修改：

/Tractor/queue?q=jattr&set_ATTRNAME=VALUE&jid=NNNN&tsid=sss

类似地，命令级别的属性可以使用 "q=cattr&cid=..." 变体修改（注意：目前只支持 set_service）：

/Tractor/queue?q=cattr&set_ATTRNAME=VALUE&jid=NNNN&cid=CCCC&tsid=sss

请注意，crews.config 可以指定权限，将特定属性的编辑限制给特定 crew，例如只允许 Wranglers。

更改作业级别的 Service Key 需求：

http://ENGINE/Tractor/queue?q=jattr&jid=5678&set_service=someKey&tsid=sss

重新设置作业优先级：

http://ENGINE/Tractor/queue?q=jattr&jid=5678&set_priority=12.34&tsid=sss

暂停作业中的新调度：
 http://ENGINE/Tractor/queue?q=jattr&jid=5678&set_pause=1&tsid=sss

使用 "&set_pause=0" 可以取消作业暂停。这种 "pause" 会暂停新任务的调度；在取消暂停之前，该作业会在 Blade 分配遍历中被跳过。该作业中任何已经运行的任务都会继续运行。
 

会话查询

用户专用信息的多个文件存储在 Tractor spool 文件夹下方的 user 区域中。以下查询允许 UI 保存和检索这些文件。

列出指定类型用户的所有已存储过滤器：

http://ENGINE/Tractor/monitor?q=filters&user=name&type=filtertype

检索用户的命名过滤器：

http://ENGINE/Tractor/monitor?q=getfilter&user=name&key=filtername

存储用户的命名过滤器：

http://ENGINE/Tractor/monitor?q=putfilter&user=name&key=filtername

检索用户的偏好信息：

http://ENGINE/Tractor/monitor?q=getpreference&user=name&key=filename

存储用户的偏好信息：

http://ENGINE/Tractor/monitor?q=putpreference&user=name&key=filename

删除用户的偏好信息：

http://ENGINE/Tractor/monitor?q=delpreference&user=name&key=filename

检索用户的会话文件：

http://ENGINE/Tractor/monitor?q=getsession&user=name&key=filename

存储用户的会话文件：

http://ENGINE/Tractor/monitor?q=putsession&user=name&key=filename

删除用户的会话文件：

http://ENGINE/Tractor/monitor?q=delsession&user=name&key=filename

检索可用文件规则定义列表：

http://ENGINE/Tractor/monitor?q=filterrules&type=filtertype

管理员控制

重新加载配置文件（blade、crews、limits）：

http://ENGINE/Tractor/ctrl?q=reconfigure

重新加载单个配置文件，例如 limits.config：

http://ENGINE/Tractor/ctrl?q=reconfigure&file=limits.config

查询基本 Engine 状态：

http://ENGINE/Tractor/ctrl?q=status

查询当前活动 limit 计数：
 http://ENGINE/Tractor/queue?q=limits

注意：此查询开销相对较大，会锁定所有线程。添加 &full=1 还会包含 limits.config 中未定义且值为零的“临时”limits：**

查询展开后的 crew 定义：

http://ENGINE/Tractor/monitor?q=crewdetails

查询当前调度层级定义：

http://ENGINE/Tractor/monitor?q=enumtierlist

查询展开后的 Blade profile 定义：

http://ENGINE/Tractor/config?q=get&file=blade.config

列出最近连接过的 Dashboard 用户（会话 mailbox）：

http://ENGINE/Tractor/monitor?q=mboxes

跟踪 Engine 分配决策：
 http://ENGINE/Tractor/ctrl?q=tracer&fmt=plain -- 下一次分配遍历

添加 &t=bbbb -- 针对名为 bbbb 的 Blade；添加 &t=jjjj -- 针对 jid 为 jjjj 的作业

从显示的活动 Blade 列表中移除旧条目：
 http://ENGINE/Tractor/btrack?q=delist&id=hhhhh/aa.aa.aa.aa

其中 hhhhh 是 Blade 的主机名，aa.aa.aa.aa 是该 Blade 当前的 TCP/IP 地址。

诊断：查询 Engine 内部消息队列长度：
 http://ENGINE/Tractor/ctrl?q=status&qlen=1

注意：此查询开销相对较大，会锁定所有线程。

Engine 日志级别： 更改 Engine 自身诊断日志的日志阈值级别：
http://ENGINE/Tractor/ctrl?q=loglevel&v=debug

可识别的级别名称包括：severe、notice、info、debug、trace

Engine 数据库上下文： 重新建立 Engine 自身到数据库服务器的内部客户端连接：

http://ENGINE/Tractor/ctrl?q=dbreconnect
 

Blade 查询和控制

列出当前所有已连接 Blade，并包含最近的状态快照：

http://ENGINE/Tractor/monitor?q=blades

停止并重新排队某个特定 Blade 上当前所有活动任务：

http://ENGINE/Tractor/queue?q=ejectall&blade=host/addr&tsid=sss

向 Engine 查询某个指定 Blade 的信息：

http://ENGINE/Tractor/monitor?q=bdetails&b=NAME

向 Engine 查询 Blade 信息，并包含对 Blade 本身进行一次（开销较高的）直接探测所得结果： http://ENGINE/Tractor/monitor?q=bdetails&b=NAME&probe=1

请谨慎使用此查询，尤其是在脚本中，因为 Engine 等待 Blade 响应时可能产生处理延迟，特别是探测响应较慢的 Blade 时。在这种查询形式中，NAME 参数可以是 "hostname" 或 "hostname/address"。address 部分是历史遗留内容，会被忽略；Engine 会从 Blade 数据库的最后一个条目推导 Blade 地址。

直接查询某个 Blade 的当前状态：

http://BLADE/blade/status

通过 Engine 更改 Blade 的 NIMBY 状态：

首先发送登录请求以接收会话 ID（tsid），然后：

http://ENGINE/Tractor/ctrl?q=battribute&tsid=SSS&b=BBB&nimby=NNN

其中：

SSS = 登录响应中的 tsid
BBB = 标识 Blade，格式如 "blade_hname/192.168.0.1"
NNN = 期望的新 nimby 状态：
nimby=1 -- Blade 只接受来自该主机提交作业中的本地 Cmd
nimby=0 -- 将 nimby 状态重置为不受限制
nimby=jobOwner1,owner2,... -- Blade 只接受指定用户拥有的作业

通过与 Blade 直接建立 HTTP 连接来更改 Blade 的 NIMBY 状态：
 http://BLADE/blade/ctrl?nimby=NNN

其中 NNN 与上面的 Engine 代理情形相同。请注意，这种直接连接类型可能会被 blade.config 中的 NimbyConnectPolicy 设置禁止。
 

Engine 指标和统计

获取最新的 Engine 健康统计：
 http://ENGINE/Tractor/monitor?q=statistics

该查询返回各种 tractor-engine 性能和状态指标。目前提供的值旨在为管理员提供一组 Engine“关键体征”，便于在查看整体 Tractor 健康状况或 Engine 热点时快速检查。

统计报告组织为 JSON 字典。字典中的每个 key 表示一种指标类型。每个 key 的 value 字段是该指标近期样本组成的数组。

当前字典指标名称列在这里。该列表可能会随时间演变。

  • "dt" - 每个采样间隔的实际长度
  • "enla" - Engine 归一化平均负载，即 Engine 主机上的 CPU 负载
  • "rtot" - 该间隔内 Engine 入站查询总数
  • "asn" - 该间隔内完成的分配数量
  • "nrun" - 当前在 farm 上调度并运行的 cmd 数量
  • "nctot" - 所有等待作业中未执行命令的总积压量
  • "ncrdy" - 所有等待作业中就绪命令的总积压量
  • "nslot" - Blade 报告的 farm 总 slot 数
  • "nsin" - farm 上正在使用的总 slot 数
  • "tecpu" - 当前 tractor-engine 造成的 CPU 利用率估计值
  • "temem" - 当前 tractor-engine 造成的 RAM 利用率估计值
  • "dbcpu" - 当前作业数据库造成的 CPU 利用率估计值
  • "dbmem" - 当前作业数据库造成的 RAM 利用率估计值
  • "dbcqn" - 待处理作业状态数据库提交的积压量
  • "iqn" - 主接收队列上等待请求的积压量
  • "sqn" - 发送池队列上等待请求的积压量
  • "aqn" - 分配器队列上等待请求的积压量
  • "qqn" - 作业归档队列上等待请求的积压量
  • "jqn" - 作业提交队列上等待请求的积压量
  • "mqn" - 监控 UI 队列上等待请求的积压量
  • "nmbox" - 已订阅的 UI 会话

获取 Engine 统计的近期历史记录：
 http://ENGINE/Tractor/monitor?q=statslog

此查询类似于上面的 q=statistics，但它会返回每个指标数分钟的样本。

注意：对于持续监控，请在周期性 q=subscribe 请求中添加 "&stats=1"；统计信息会作为 's' mbox 消息到达。

每个数组中的最后一个条目是最新样本。Engine 会维护这些样本的一个简单环形缓冲区，因此在下一次更新时，第一个数组元素会被丢弃，所有其他样本看起来会“左移”，并在末尾出现新值。statslog 查询返回缓冲区中的所有样本，使 UI 能够从单个查询生成完整图表。
