Tractor 2.x 功能

Tractor 2 中的新功能扩展了核心 Tractor system（Tractor 系统） 这些核心能力建立在 1.x 系列发布版本之上。新增和改进范围很广，从面向 wrangler 的高效新命令行和脚本接口，到简单的用户界面变更都有涵盖。内部升级包括新的高性能、高能力 job database，以及新的全工作室资源共享和分配控制。请参考 Upgrading（升级说明文档）中描述的指导原则。

下面是一些亮点：

  • Tractor 产品布局 -- 单一发布目录、每个平台单一下载包、捆绑的子系统更新 -- Tractor 2.x 的打包和安装布局在一个下载包中包含一套匹配的 Tractor 组件：engine、blade、spooler、用户界面和脚本工具。它们会一起安装到一个带版本号的区域中，并且只包含一份匹配的共享资源，其中包括若干第三方子系统的预构建版本。

  • Tractor 查询工具 -- 引入 tq，即 tractor query（Tractor 查询） 命令行工具和模块。tq 基于 Pixar 工作室中经过验证的工具构建，是从 terminal shell、Python 脚本或 Dashboard 新标签页查询实时或历史 Tractor 数据的最佳方式。

  • 自适应 Farm 分配 -- 一种使用 Tractor 灵活的 Limits（限制） system 在人员或项目之间动态分配抽象资源的方法。如果有两部影片正在制作，可以把 farm 的 60% 分配给其中一部，30% 分配给另一部，剩余 10% 留给其他项目。如果某个 show 处于空闲状态，其他项目可以临时扩大自己的份额，然后在所有项目都处于 active 状态时缩回名义水平。

  • Dispatching Tiers（分派层级功能） -- 一种简单方式，用于把大范围 job 组织成站点定义的降序 priority group。默认 tier 名称为： 默认层级名称为 admin、rush、default、batch。 也可以创建你自己的 tier。

  • Custom Menu Actions（自定义菜单动作） -- 添加站点定义的 Dashboard 菜单项，调用你自己的集中式脚本，并用用户当前列表选择作为参数。

  • Job Authoring API（作业编写 API） -- 新的 tractor.api.author 模块允许 Python 脚本轻松创建 Job、Task 和 Command 对象，并按依赖需求把它们链接在一起。随后 Job.spool() 方法会把生成的 job 发送到 tractor-engine job queue。

  • 简单 Engine Discovery（引擎发现） -- 面向小型工作室的简单 “zero-config” 公告能力，允许 tractor-blade 和其他 Tractor 工具在本地网络中找到 tractor-engine，而不需要手动修改 nameserver（DNS/LDAP）配置。在站点 nameserver 数据库中已由管理员创建 hostname alias “tractor-engine” 的工作室，tractor-engine 会自动禁用这种 SSDP 风格流量。

  • Checkpoint-Resume（检查点恢复）支持 -- 对 job scripting、dispatching 和 Dashboard 的扩展增加了与增量计算相关的新能力。Tractor 也支持通用的 “retry from last checkpoint” 方案。这两项功能都与新的 RenderMan 19 RIS checkpoint 和 incremental rendering 功能集成。

  • Blade Auto-Update（Blade 自动更新） -- 一个简单的 tractor-blade 补丁管理系统，允许管理员把 farm “pin” 到某个特定 blade patch version，并把新版本自动推送到整个 farm。过期 blade 会使用新的模块版本自行重启。

  • Pluggable Authentication Module（PAM）支持 -- engine 可选的新内置 PAM 支持会把密码验证直接委托给 engine host 上的操作系统。在 LAN 已经提供足够 credential transport security 的工作室，这种替代方式可以简单启用密码支持。

  • Privilege Insulation（权限隔离） -- 在 tractor.config 配置文件中 EngineOwner 设置指定 tractor-engine 应在其下运行的 login identity。此设置很重要，因为它允许 engine 在获取任何需要的受保护 network port 后立即放弃 root 权限。之后 engine 日常操作会在与指定 login name 关联的限制下进行。

  • 动态 Service Key（服务键）公布 -- 新增了若干 blade profile “Provides” mode，以支持一些高级 service key 使用场景。例如，blade 可以根据之前已启动 command “consumed” 的 key，动态公布不同的 service key 能力集合。

  • Resource Usage Tracking（资源用量跟踪） -- 操作系统 rusage process metrics CPU、RSS 和 VSZ 现在会针对每个已启动 command 记录到 job database 中。目前在 Linux 和 OSX tractor-blade 上支持。

  • Command Retry History（命令重试历史） -- 现在每次 command 启动尝试都会在 job database 中创建唯一 tracking record。因此，可以使用 tq 工具查看给定 task 上的重试历史。

  • Configuration File Loading（配置文件加载） -- 简化的 override system 可以帮助减少杂乱，并提高哪些文件已从工作室原始 “factory settings” 修改过的清晰度。

  • Task Concurrency Throttle（任务并发节流） -- 每个 job 都可以指定 “maxActive” attribute，以约束来自该 job 的并发运行 task 数量。当不适合修改完整站点级 Limits 设置时，这种对 job 在 farm 上 “footprint” 大小的快速 wrangling 控制会很有用。

  • Automatic Blade Error Throttle（自动 Blade 错误节流） -- 如果 blade 在给定时间间隔内遇到过多错误，此 blade profile 设置会阻止 blade 接收新工作。

  • Job Spooling（作业提交）改进 -- Job processing 升级包括更快处理、更好的错误检查，以及所需子系统的捆绑。并行化的 job intake 和 database staging 方案可以在许多 job 同时 spooled 或许多 “expand” task 并行运行时显著减少 backlog。spooler 中捆绑的自包含 Tcl interpreter 简化了站点安装需求，并可在 job 送达 engine 之前执行 client-side error checking。也支持新的 JSON job spooling 格式（但在 beta-1 之前不可用，等待变更）。

Bug fixes and improvements（Bug 修复和改进）

  • 修复了先前 2.4 发布中的一个 regression，该问题可能导致 service key 报告不正确。

2.4 2069290 中的变更

Tractor 2.4 是一个聚焦整体性能、处理大规模 farm size、job size 和并发连接 user session 数量的更新。变更包括：

  • 一次重要的 code refactoring 解决了若干内部 thread contention bottleneck，包括与频繁 password check 和 identity management 相关的问题。新逻辑提升了 throughput，尤其是在超大型 farm（5000+ blade）上，许多 dashboard session 和 automated script（1000+）访问 job status 时更明显。

  • 为 “expand” task 产生的新 ready command 排序提供了更稳健的 job-global system。此变更解决了 “Cmd not Ready?” 错误问题，该问题源于大型递归展开 job 上的 sorting key collision（precision）。
  • 优化了 limit checking 操作，尤其包括重复 tokenization、trace message 构造，以及当 limit 达到最大容量并临时 throttle 新 dispatching 时的高效处理。
  • 在 queue/backlog 诊断（“status&qlen=1” 查询）中添加了新的 “connecting” 条目。该计数可明确显示已连接但尚未完成的 inbound request 数量，也就是仍在等待完整 http request body 内容的 network i/o 的请求。

  • 用于运行 custom menu item backend script 的 handler thread 数量现在会根据 tractor.config 设置与其他 thread pool 一起扩展。

  • 修复了一个 string handling 问题，该问题可能导致间歇性丢失对 task log output 的访问。

  • 修复了 tractor-engine 在为 “expand task” job graph extension 中不可解析的 control character 构造错误消息时崩溃的问题。
  • 修复了 Dashboard custom job filter 对 job Project name 等 array type 的匹配。此前这些条目只有 exact match 能工作，“starts with” 或 “contains” 比较不能工作。
  • 更新了 tractor-spool job submission utility 的命令行选项。直接从命令行参数构造新 job 时使用的 “range” 值现在支持若干新的 formatting option，例如 zero padding；这不同于提交先前构造好的完整 job file。
  • 修复了 Windows 上 “tractor-spool --ribs *.rib” wildcard expansion 处理问题。

  • 修复了一些 system service start up script 中不匹配的 version string。

2.2 1715407 中的变更

  •  当 Katana（renderboot）把 RenderMan（prman）progress message 包裹在额外 logging text 中时，tractor-blade 现在能正确检测这些消息。
  • 从使用 Python job authoring API 方法 EngineClient.spool() 的 client 提交到 tractor-engine 的 job，现在会正确遵守当前 tractor.config 设置 AllowJobOwnerOverride。请注意，此修复要求 client 重新导入新的 tractor.api.author 模块。
  • 解决了一个 tractor.engine threading 问题。在不常见情况下，tq scripting client 正常向 engine 发起请求，但站点 network routing 问题阻止 engine reply buffer 送回少数 client 时，该问题可能造成很大的 “config” backlog 并减慢 dispatching。现在这些 stalled delivery 会 time-out，而不会影响其他 transaction。
  • 增加了对 “assigner Cmd not Ready” state inconsistency 条件的内部处理；该条件可能出现在同时 task retry 和 job restart 的某些情况下。
  • 修复了 task 完成后、display refresh 之前 Dashboard 中有时显示负 elapsed time 的问题。

2.2 1677499 中的变更

  • 如果 Custom menu item 有任何 script output，就会打开新窗口，即使该 menu item 配置为通常抑制新窗口。这使问题脚本更容易被发现和调试。
  • Custom menu item 支持在 “values” list 中使用 “login” 作为特殊条目，这会让 Dashboard user name 成为发送给 menu item script 的 payload 的一部分。
  • Custom menu item 现在会对 job 和 task custom menu item 的 crews attribute 中的 user name 和 “@owner” 生效。
  • 修复了 Dashboard task list 中按 state 选择 next task 时不会自动滚动到该 task 的 bug。
  • 修复了 Dashboard job graph rollover 中 skipped task 的 elapsed time 计算。
  • 已修正 Dashboard task list 中 task 的排序。
  • Dashboard preview command 现在可以针对 archived job 显示和运行。
  • tq 和 query API 现在支持 kill 操作。它用于 kill 正在运行的 command，并让 task 保持 error state。
  • tq 新增 reloadconfig command，用于从命令行触发配置文件重新加载。
  • query API 只有在 archive flag 设置为 True 时才会返回未注册 blade。
  • tractor-dbctl --exec-sql 现在会输出错误消息。
  • blade 会缓存 user database entry，以便在 LDAP server 短暂故障时更具韧性。
  • systemd 配置目录默认指向 Tractor installation config/ directory，与 sysvinit 设置保持一致。
  • systemd 现在默认以 root 启动 tractor blade，与 sysvinit 设置保持一致。

2.2 1641321 中的变更

  • 修复了最近引入的 “rfm” environment key pattern（例如 “rfm-RRR-maya-MMMM”）在 Mac OSX 上用于 Maya render 时的默认 MAYA_LOCATION，这与类似的 “rms-” key 不同。environment key 会让 tractor-blade 在启动每个 command 前设置各种 environment variable。这些变量可包括 MAYA_LOCATION 和 PATH。系统有内置 key handler，也支持每个工作室定义的 custom handler。在此情况下，像 “rfm-21.0-maya-2016.5” 这样的 key 应该让 shared.macosx.envkeys 扩展 PATH，并包含正确的 Maya location。

    /Applications/Autodesk/maya2016.5/Maya.app/Contents/bin

  • 修复了某些情况下单个 blade 并发运行大量 command 时的 blade status probe 问题。

  • 修复了 tractor-blade 问题：当提交 job 的用户 login name 中包含 “@” 字符时，task launch 会失败。（这里不是 email address，而是 unix login user name。）

  • 调整了 “limit max set to zero?” diagnostic 的 log severity level。它现在只在 TRACE level 作为 debugging hint 出现。limit 的 maximum 有时设为 zero 是可接受的。

2.2 1625934 中的变更

  • 新增了基于 “total physical RAM” 进行 blade selection 的 job service key expression 支持。例如，该 expression

    RemoteCmd {prman my.rib} -service "PixarRender && @.totalmem > 24"

    选择提供 “PixarRender” service（blade.config）且安装至少 24 GB RAM 的 blade。此前支持的用于 “available free RAM” 的 “@.mem” key 也仍然可用。

  • Enabled access（已启用访问）：在 “tq blades” 查询中启用了对 BladeUse attribute 的访问：taskcount、slotsinuse 和 owners。

  • 为 logcleaner utility script 添加了 --user option，使其可以使用不同用户来查询 job，而不是使用执行文件删除的 process owner。

  • 修复了 tq jattr 和 cattr command 中用于对 job 和 command list attribute 执行相对变更的 --add 和 --remove 操作。

  • 解决了 Linux 上 tractor-engine 的 socket exception handling 问题；当 tractor-blade host（operating system）无响应时，例如 GPU driver、OOM 或 kernel panic 情况，tractor-engine process 有时会在 socket handler 中空转并表现出高 CPU load。

  • 修复了 retry、skip 和 job interrupt URL request 的 JSON response 中的 access-denied advisory text。

  • RHEL6 PAM 相关 file descriptor leak 的建议 workaround：

    On Linux RHEL 6.x era releases, the pam_fprintd.so module
contains a bug causing it to leak file descriptors on every call from
tractor-engine.  Since PAM modules are loaded into the tractor-engine
process, and it performs many authentications over time, the unclosed
"pipe" descriptors will accumulate, unknown to the main tractor-engine
code and will eventually exhaust the available file descriptor limit
for that engine process.  While many studios do not depend on
fingerprint validation, especially for scripted API access to a system
service, the "fprint" module is called indirectly from many common
RHEL6 PAM policies, including "login" and "su".  It has been removed
from the common policies in RHEL 7 era distributions.  A workaround
for RHEL6 is to create your own "tractor" policy that doesn't include
system-auth, or perhaps to specify a less general policy in crews.config,
such as password-auth.

2.2 1572682 中的变更

  • Dashboard job list 中新的 “Serial Elapsed” 列会显示每个 job 中 task 的 aggregate wall-clock runtime。当 task 并发运行时，此值会长于该 job 作为整体在 farm 上花费的 real time。例如，如果一个 job 有 10 个 task，每个运行 2 分钟，且 10 个 task 同时在不同 blade 上启动，则 job 会在 2 分钟内完成，而 “Serial Elapsed” 会显示 20 分钟。
  • Dashboard 中先前的 job “Elapsed” 列已重命名为 “Activity Span”，以更准确描述其显示的值。这是从该 job 第一次 task dispatch 到该 job 最后一次 task activity 或 user action 之间的 wall-clock time span。它不一定表示在 farm 上 actively executing 的时间，因为给定 job 的 task 之间可能存在 scheduling gap（例如其他更高 priority job），或 attribute edit、retry 等 user action 可能发生在上一次 task activity 之后很久。
  • Dashboard command info view 中的 “Elapsed”、“App Secs” 和 “Sys Secs” 字段已重命名为 “Real Elapsed”、“Thread Elapsed” 和 “Sys Elapsed”，以更准确描述这些值。值同时以 H:M:S 和秒数表示。
  • 修复了 Dashboard 问题，该问题可能导致 job elapsed time 列（现在为 Activity Span）在手动选择 job 之前停留在 0:00。
  • 修复了滚动 job graph 时 preview indicator 消失的问题。
  • 修复了 “tq blades” 行为，使其默认一致地列出并操作 “registered” blade。这些 blade 是在其 visibility 最近一次从 dashboard 或 tq 中 “cleared” 之后重新向 engine registered 的 blade。使用 “tq blades up” 可以列出最近发送 heartbeat 的 blade。某些 blade 会 register，但随后离线；它们不会有最近 heartbeat，但会保持 registered，以便管理员看到它们已离线。
  • 修复了 Python job authoring API，使其在创建 “local” command 而不是更常见的默认 RemoteCmd 时，能正确处理 Command(local=true, ...) keyword。
  • 修复了与 task retry 相关的若干 state update 问题。

2.2 1564321 中的变更

功能

  • 现在可以在 blade.config 的 “ProfileDefaults” block 中指定 GPU Exclusion Patterns。此设置用于在 service matching 和 counting consideration 期间排除某些 GPU type。请注意，此 keyword 只在 ProfileDefaults dictionary 中有效，并且 GPU filtering 会在 prior（之前） 任何 per-profile match testing 之前执行。背景：给定 profile 可以基于 “Hosts” clause 中的若干 criteria 匹配特定 host，这些 criteria 可包括该 host 上可用 GPU 的 type 和 count。有些 host 包含多个 GPU，包括 “uninteresting” virtual GPU 或性能不足的 GPU，它们应始终在 profile matching pass prior（之前） 就从考虑范围中排除。使用 “GPUExclusionPatterns” 枚举应在 count 和 match 中跳过的 GPU make/model。列表中的每一项都是简单 “glob-style” wildcard pattern，不包含 ‘*’ 或 ‘?’ 的 pattern 会被当作给出了 “TEXT” 来处理。示例见新的 stock blade.config：

    "GPUExclusionPatterns": ["QX?", "paravirtual", "RV3*"]

  • 新的 blade.config 设置 “SubstituteJobCWD” 可以要求在使用给定 profile 的 blade 上，从所有 job 启动的所有 command 都使用单一 working directory。

  • 修复了 engine 对 job database 中 task “retry” state 的 update，避免各种 dashboard display inconsistency。
  • 修复了 job 被 reload 到 engine 时对 prior job elapsed time 的处理。time estimation summary 有时会被置零，导致 tq 和 dashboard 中显示不正确的值。

  • 修复了 dashboard task graph “instance” node rendering，使其立即应用 task state change color 并避免不必要 decoration。
  • 更新了 dashboard query pane，以支持显示最近添加的 database field。
  • 更新 RMS/RFM environment handler，在 path 中包含捆绑的 “rmantree” directory（用于 RMS），或即将发布的 RfM-21 对应 RPS tree。

    RemoteCmd {sleep 15} -service PixarRender -minrunsecs 5 -maxrunsecs 20

    "CmdAutoRetryStopCodes": [-9, -15, 10110, 10111],

  • Engine statistics query -- 新增 URL request（Tractor/monitor?q=statistics），帮助把 tractor-engine performance metric 与其他 site-wide monitoring system 集成。返回的 JSON object 包含 engine 收集的若干自身 statistic 的最新 sample。例如，这些数据可用于填充外部站点监控系统。有些监控系统能直接发起此 URL request 获取数据，而其他系统可能需要编写一个小 data source script，请求 JSON statistics report 后把每个感兴趣的 value 分别转发给 monitoring system。

  • Concurrent Expand Chunks（并发扩展分块）-- 这个高级 expand task variant 为避免包含长时间运行单一 command 的 job 中出现 serial delay 提供了一种方法，这些 command 会生成其他 task 所需的一系列 result。此新扩展允许 pipeline integrator 构建如下 job：启动一个 long running command（如 fluid simulation），并在第一个 command 依次生成每个 output file 时并发启动另一个 command（如 render）。因此，render 可以在不等待所有 simulation step 完成的情况下继续。此方法特别适合 simulation app 正在创建事先不知道 filename 的 output file 的情况，因此后续 render command line argument 必须动态生成。simulation 或 wrapper script 检测下一步完成后，把适当的 rendering Task description 写入 temporary file，然后通过在 stdout 上输出新的 ‘TR_EXPAND_CHUNK “filename”n’ 行来通知 tractor-blade。Tractor-blade 会在 application stdout stream 中检测到该 directive，并把文件内容发送给 engine。新的 render task 会插入到 running job 中，并可立即在 farm 上其他位置 dispatch。blade 会在 temporary file 送达后自动删除它。

  • TR_EXIT_STATUS auto-terminate policy change -- TR_EXIT_STATUS handler 的 default behavior 现在恢复为 1.x 和更早 2.x 的行为，即只记录 status value，并在 command 实际退出时稍后报告。较新的行为（blade 收到 TR_EXIT_STATUS 后主动 kill app）仍然可用，但必须使用 profile setting 在 blade.config 中显式启用：

    "TR_EXIT_STATUS_terminate": 1,

  • Blade record visibility flag -- Dashboard blade list display 基于 database record 创建，这些 record 描述过去曾连接到 engine 的每个 tractor-blade instance。即使 blade host 不再部署，这些 record 也会保留，以便把以前执行的 command 与运行它们的机器关联起来。dashboard blade list menu item “Clear prior blade data” 不再移除给定 blade 的实际 database record，而只是设置一个 flag，在 dashboard 中隐藏该 record。record（及其新的 unique id field）现在会保留，用于与旧 task record 关联。如果 blade data item 确实不再需要，可以手动完全移除。

  • Cookie-based Dashboard relogin -- 新 policy 允许根据保存的 session cookie 自动重新登录到新的 Dashboard window，即使站点启用了 password。cookie 只包含由 engine 验证的 session ID，本身不包含任何 password data。若要恢复旧 policy（需要 password 时拒绝 auto-login），可向 crews.config SitePasswordValidator 设置添加 “_nocookie” modifier。

  • 新增 tractor-dbctl --set-job-counter option，用于设置新 job database 中的初始 job ID value。默认情况下 Job ID 从 1 开始，因此在 fresh Tractor install 中指定不同 starting value 会很有帮助，可防止新安装的 job ID 与旧 job 重叠。复用先前 job database 的 Tractor upgrade install 会继续保持 job ID continuity。

  • job database upgrade procedure 做了若干内部改进。新 release 中许多 code-related change 现在可以不进行重大 database alteration 就应用，只需要 engine restart。涉及新 database schema definition 的变更现在会通过一个更好处理跨多个版本 upgrade 的系统来应用。

  • Overall throughput optimizations -- 本 release 中做了各种 performance improvement，尤其是处理大量 simultaneous update 方面，例如许多 job 同时完成或删除时。
