Tractor 2.x 功能

Tractor 2 中的新功能扩展了核心 Tractor 系统。这些核心能力建立在 1.x 系列发布版本之上。新增和改进范围很广，从面向调度员的高效新命令行和脚本接口，到简单的用户界面变更都有涵盖。内部升级包括新的高性能、高能力作业数据库，以及新的全工作室资源共享和分配控制。请参考 Upgrading（升级说明文档）中描述的指导原则。

下面是一些亮点：

  • Tractor 产品布局 -- 单一发布目录、每个平台单一下载包、捆绑的子系统更新 -- Tractor 2.x 的打包和安装布局在一个下载包中包含一套匹配的 Tractor 组件：Engine、Blade、提交器、用户界面和脚本工具。它们会一起安装到一个带版本号的区域中，并且只包含一份匹配的共享资源，其中包括若干第三方子系统的预构建版本。

  • Tractor 查询工具 -- 引入 tq，即 Tractor 查询命令行工具和模块。tq 基于 Pixar 工作室中经过验证的工具构建，是从终端 shell、Python 脚本或 Dashboard 新标签页查询实时或历史 Tractor 数据的最佳方式。

  • 自适应渲染农场分配 -- 一种使用 Tractor 灵活的 Limits（限制）系统在人员或项目之间动态分配抽象资源的方法。如果有两部影片正在制作，可以把渲染农场的 60% 分配给其中一部，30% 分配给另一部，剩余 10% 留给其他项目。如果某个项目处于空闲状态，其他项目可以临时扩大自己的份额，然后在所有项目都处于活动状态时缩回名义水平。

  • Dispatching Tiers（分派层级功能） -- 一种简单方式，用于把大批作业组织成站点定义的降序优先级组。默认层级名称为：admin、rush、default、batch。 也可以创建你自己的层级。

  • Custom Menu Actions（自定义菜单动作） -- 添加站点定义的 Dashboard 菜单项，调用你自己的集中式脚本，并用用户当前列表选择作为参数。

  • Job Authoring API（作业编写 API） -- 新的 tractor.api.author 模块允许 Python 脚本轻松创建 Job、Task 和 Command 对象，并按依赖需求把它们链接在一起。随后 Job.spool() 方法会把生成的作业发送到 tractor-engine 作业队列。

  • Simple Engine Discovery（简单 Engine 发现） -- 面向小型工作室的简单“零配置”公告能力，允许 tractor-blade 和其他 Tractor 工具在本地网络中找到 tractor-engine，而不需要手动修改名称服务（DNS/LDAP）配置。在站点名称服务数据库中已由管理员创建主机名别名 “tractor-engine” 的工作室，tractor-engine 会自动禁用这种 SSDP 风格流量。

  • Checkpoint-Resume（检查点恢复）支持 -- 对作业脚本、分派和 Dashboard 的扩展增加了与增量计算相关的新能力。Tractor 也支持通用的 “从上一个检查点重试” 方案。这两项功能都与新的 RenderMan 19 RIS 检查点和增量渲染功能集成。

  • Blade Auto-Update（Blade 自动更新） -- 一个简单的 tractor-blade 补丁管理系统，允许管理员把渲染农场“pin”到某个特定 Blade 补丁版本，并把新版本自动推送到整个渲染农场。过期 Blade 会使用新的模块版本自行重启。

  • Pluggable Authentication Module（PAM）支持 -- Engine 可选的新内置 PAM 支持会把密码验证直接委托给 Engine 主机上的操作系统。在 LAN 已经提供足够凭据传输安全的工作室，这种替代方式可以简单启用密码支持。

  • Privilege Insulation（权限隔离） -- 在 tractor.config 配置文件中 EngineOwner 设置指定 tractor-engine 应在其下运行的登录身份。此设置很重要，因为它允许 Engine 在获取任何需要的受保护的网络端口后立即放弃 root 权限。之后 Engine 日常操作会在与指定登录名关联的限制下进行。

  • 动态 Service Key（服务键）公布 -- 新增了若干 Blade 配置档 “Provides” 模式，以支持一些高级服务键使用场景。例如，Blade 可以根据之前已启动命令已经“消耗”的键，动态公布不同的服务键能力集合。

  • Resource Usage Tracking（资源用量跟踪） -- 操作系统 rusage 进程指标 CPU、RSS 和 VSZ 现在会针对每个已启动命令记录到作业数据库中。目前在 Linux 和 OSX tractor-blade 上支持。

  • Command Retry History（命令重试历史） -- 现在每次命令启动尝试都会在作业数据库中创建唯一跟踪记录。因此，可以使用 tq 工具查看给定任务上的重试历史。

  • Configuration File Loading（配置文件加载） -- 简化的覆盖系统可以帮助减少杂乱，并提高哪些文件已从工作室原始 “出厂设置” 修改过的清晰度。

  • Task Concurrency Throttle（任务并发节流） -- 每个作业都可以指定 “maxActive” 属性，以约束来自该作业的并发运行任务数量。当不适合修改完整站点级 Limits 设置时，这种对作业在渲染农场上 “占用规模”的快速调度控制会很有用。

  • Automatic Blade Error Throttle（自动 Blade 错误节流） -- 如果 Blade 在给定时间间隔内遇到过多错误，此 Blade 配置档设置会阻止 Blade 接收新工作。

  • Job Spooling（作业提交）改进 -- 作业处理升级包括更快处理、更好的错误检查，以及所需子系统的捆绑。并行化的作业接收和数据库暂存方案可以在许多作业同时提交或许多 “expand” 任务并行运行时显著减少积压。提交器中捆绑的自包含 Tcl 解释器简化了站点安装需求，并可在作业送达 Engine 之前执行客户端错误检查。也支持新的 JSON 作业提交格式（但在 beta-1 之前不可用，等待变更）。

2.4 2069290 中的变更

Tractor 2.4 是一个聚焦整体性能、处理大规模渲染农场、大型作业和大量并发用户会话的更新。变更包括：

  • 一次重要的代码重构解决了若干内部线程争用瓶颈，包括与频繁密码检查和身份管理相关的问题。新逻辑提升了吞吐量，尤其是在超大型渲染农场（5000+ 个 Blade）上，许多 Dashboard 会话和自动化脚本（1000+）访问作业状态时更明显。

  • 为 “expand” 任务产生的新版 ready（就绪）状态命令排序提供了更稳健的作业全局系统。此变更解决了 “Cmd not Ready?” 错误问题，该问题源于大型递归展开作业中的排序键冲突（precision）。
  • 优化了限制检查操作，尤其包括重复分词、追踪消息构造，以及当限制达到最大容量并临时限制新的分派时的高效处理。
  • 在队列/积压诊断（“status&qlen=1” 查询）中添加了新的 “connecting”（连接中）条目。该计数可明确显示已连接但尚未完成的入站请求数量，也就是仍在等待完整 HTTP 请求体内容的网络 I/O 请求。

  • 用于运行自定义菜单项后端脚本的处理器线程数量现在会根据 tractor.config 设置与其他线程池一起扩展。

  • 修复了一个字符串处理问题，该问题可能导致间歇性丢失对任务日志输出的访问。

  • 修复了 tractor-engine 在为 “expand 任务” 作业图扩展中不可解析的控制字符构造错误消息时崩溃的问题。
  • 修复了 Dashboard 自定义作业过滤器对作业项目名称等数组类型的匹配。此前这些条目只有精确匹配能工作，“starts with”（开头为）或 “contains”（包含）比较不能工作。
  • 更新了 tractor-spool 作业提交工具的命令行选项。直接从命令行参数构造新作业时使用的 “range” 值现在支持若干新的格式化选项，例如补零；这不同于提交先前构造好的完整作业文件。
  • 修复了 Windows 上 “tractor-spool --ribs *.rib” 通配符展开处理问题。

  • 修复了一些系统服务启动脚本中不匹配的版本字符串。

2.2 1715407 中的变更

  •  当 Katana（renderboot）把 RenderMan（prman）进度消息包裹在额外日志文本中时，tractor-blade 现在能正确检测这些消息。
  • 从使用 Python 作业编写 API 方法 EngineClient.spool() 的客户端提交到 tractor-engine 的作业，现在会正确遵守当前 tractor.config 设置 AllowJobOwnerOverride。请注意，此修复要求客户端重新导入新的 tractor.api.author 模块。
  • 解决了一个 tractor.engine 线程问题。在不常见情况下，tq 脚本客户端正常向 Engine 发起请求，但站点网络路由问题阻止 Engine 回复缓冲区送回少数客户端时，该问题可能造成很大的配置积压并减慢分派。现在这些停滞的传递会超时，而不会影响其他事务。
  • 增加了对 “assigner Cmd not Ready” 状态不一致条件的内部处理；该条件可能出现在同时任务重试和作业重启的某些情况下。
  • 修复了任务完成后、显示刷新之前 Dashboard 中有时显示负运行时长的问题。

  • 修复了 Dashboard 作业图 rollover 中被跳过任务的运行时长计算。

  • Dashboard 预览命令现在可以针对已归档作业显示和运行。

2.2 1641321 中的变更

  • 修复了最近引入的 “rfm” 环境键模式（例如 “rfm-RRR-maya-MMMM”）在 Mac OSX 上用于 Maya 渲染时的默认 MAYA_LOCATION，这与类似的 “rms-” 键不同。环境键会让 tractor-blade 在启动每个命令前设置各种环境变量。这些变量可包括 MAYA_LOCATION 和 PATH。系统有内置键处理器，也支持每个工作室定义的自定义处理器。在此情况下，像 “rfm-21.0-maya-2016.5” 这样的键应该让 shared.macosx.envkeys 扩展 PATH，并包含正确的 Maya 位置。

    /Applications/Autodesk/maya2016.5/Maya.app/Contents/bin

  • 修复了某些情况下单个 Blade 并发运行大量命令时的 Blade 状态探测问题。

  • 修复了 tractor-blade 问题：当提交作业的用户登录名中包含 “@” 字符时，任务启动会失败。（这里不是电子邮件地址，而是 Unix 登录用户名。）

  • 调整了 “限制 max set to zero?” 诊断的日志严重级别。它现在只在 TRACE 级别作为调试提示出现。限制的 maximum 有时设为 0（zero）是可接受的。

2.2 1625934 中的变更

  • 新增了基于 “total physical RAM”（总物理内存）进行 Blade 选择的作业服务键表达式支持。例如，该表达式

    RemoteCmd {prman my.rib} -service "PixarRender && @.totalmem > 24"

    选择提供 “PixarRender” 服务（blade.config）且安装至少 24 GB RAM 的 Blade。此前支持的用于 “available free RAM”（可用空闲 RAM）的 “@.mem” 键也仍然可用。

  • 已启用访问：在 “tq Blade” 查询中启用了对 BladeUse 属性的访问：taskcount、slotsinuse 和所有者。

  • 为 logcleaner 工具脚本添加了 --user 选项，使其可以使用不同用户来查询作业，而不是使用执行文件删除的进程所有者。

  • 修复了 tq jattr 和 cattr 命令中用于对作业和命令列表属性执行相对变更的 --add 和 --remove 操作。

  • 解决了 Linux 上 tractor-engine 的套接字异常处理问题；当 tractor-blade 主机（操作系统）无响应时，例如 GPU 驱动、OOM 或内核崩溃情况，tractor-engine 进程有时会在套接字处理器中空转并表现出高 CPU 负载。

  • 修复了 retry（重试）、skip（跳过）和作业 interrupt（中断）URL 请求的 JSON 响应中的访问被拒绝提示文本。

  • RHEL6 PAM 相关文件描述符泄漏的建议解决办法：

    On Linux RHEL 6.x era releases, the pam_fprintd.so module
contains a bug causing it to leak file descriptors on every call from
tractor-engine.  Since PAM modules are loaded into the tractor-engine
process, and it performs many authentications over time, the unclosed
"pipe" descriptors will accumulate, unknown to the main tractor-engine
code and will eventually exhaust the available file descriptor limit
for that engine process.  While many studios do not depend on
fingerprint validation, especially for scripted API access to a system
service, the "fprint" module is called indirectly from many common
RHEL6 PAM policies, including "login" and "su".  It has been removed
from the common policies in RHEL 7 era distributions.  A 解决办法
for RHEL6 is to create your own "tractor" policy that doesn't include
system-auth, or perhaps to specify a less general policy in crews.config,
such as password-auth.

2.2 1614082 中的变更

  • 在与作业删除以及 skip（跳过）、kill（终止）、recall（召回）、重试等任务操作相关的日志消息中添加了更好的用户归属信息。

  • 修复了作业属性编辑策略的执行，使 Dashboard 编辑不能应用到正在被移入归档（也就是已删除）的作业。
  • 修复了 tractor-blade 状态检查点问题，该问题有时会在 Blade 重启后导致状态报告延迟。
  • 修复了与处理故意畸形 URL 查询相关的 Engine 崩溃。
  • 在 tractor-blade 中增加了额外异常处理器保护，以防用户提供的自定义 TractorSiteStatusFilter 扩展中出现错误。
  • 修复了处理很长限制标签名的问题。
  • 改进了 Dashboard 在自动刷新作业和 Blade 列表方面的效率，减少了许多用户连接时 Engine 和数据库的负载。
  • 修复了一个 RPM 规格问题，该问题可能导致类似 “Transaction check error: file /opt from install ...” 的 RPM 安装错误消息。
  • 修复了 tractor init.d 脚本，使其在使用 “status” 查询检查服务是否运行时返回非零状态码（3）。
  • 绕过了一个 init.d 内置函数问题，该问题有时会在 Blade 服务启动时导致 “dirname” 错误。
  • 引入了可选的新 systemd 启动脚本，用于 tractor-engine 和 tractor-blade，例如供 RHEL7 系统使用。这些脚本位于 /opt/pixar/Tractor-2.2/lib/SystemServices/systemd/。请参见文档章节 Linux - with systemd（Linux 使用 systemd 的服务配置）

  • Task Elapsed Time Bounds（任务运行时长边界）-- 作业作者现在可以为给定已启动命令指定可接受的运行时长范围。运行时长超出可接受范围的命令会被标记为错误。运行超过最大时长边界的命令会被 kill（终止）。作业脚本语法示例：

    RemoteCmd {sleep 15} -service PixarRender -minrunsecs 5 -maxrunsecs 20

  • Per-Tier Scheduling（按层级调度）-- tractor.config 中 DispatchTiers 规范的新扩展允许每个已定义层级拥有自己的调度模式。例如，“rush” 层级可以按严格 FIFO 顺序调度，而默认模式可以是偏向 shared-access 的模式之一（如 P+ATCL+RR）。层级可以被分配新的 “P+CHKPT” 模式，以利用 Tractor 2.0 中的 partial-graph looping 功能；使用该模式的层级应放在接收 “classic” non-checkpoint 作业的层级之前。

    "CmdAutoRetryStopCodes": [-9, -15, 10110, 10111],

  • Concurrent Expand Chunks（并发扩展分块）-- 这个高级 expand（扩展）任务 变体为避免包含长时间运行单一命令的作业中出现串行延迟提供了一种方法，这些命令会生成其他任务所需的一系列结果。此新扩展允许流水线集成者构建如下作业：启动一个长时间运行的命令（如流体模拟），并在第一个命令依次生成每个输出文件时并发启动另一个命令（如渲染）。因此，渲染可以在不等待所有模拟步骤完成的情况下继续。此方法特别适合模拟应用正在创建事先不知道文件名的输出文件的情况，因此后续渲染命令行参数必须动态生成。模拟或包装器脚本检测下一步完成后，把适当的渲染任务描述写入临时文件，然后通过在 stdout 上输出新的 ‘TR_EXPAND_CHUNK “filename”n’ 行来通知 tractor-blade。tractor-blade 会在应用 stdout 流中检测到该指令，并把文件内容发送给 Engine。新的渲染任务会插入到运行中作业中，并可立即在渲染农场上其他位置分派。Blade 会在临时文件送达后自动删除它。
