Engine（调度引擎）

Source: https://rmanwiki-27.pixar.com/space/TRA/22184316/Engine
Downloaded: 2026-06-15T03:49:12+00:00

tractor.config 文件包含配置参数，用于控制中央 tractor-engine 进程本身的基本启动行为，以及许多基本调度策略设置。通常，此文件只需要极少数覆盖变更，而且只需要把这些少量变更作为一个小型 tractor.config 文件复制到 站点覆盖目录 中。

 

EngineOwner

EngineOwner 是登录名，会成为 tractor-engine 进程、其数据文件以及其子进程（例如作业数据库服务器）的所有者。如果该参数缺失或设置为空字符串，那么 tractor-engine 会以启动它的用户 ID 运行。这种继承用户模式对测试很有用；当 Engine 作为系统服务运行且登录身份预期通过服务定义从外部设置时，这种方式也合理。如果这里给出非空值，则必须由该用户启动 Engine，或者由 root 启动 Engine，并在获取监听端口后由 Engine setuid 到给定所有者名称。出于安全原因，这里给出的值不能是 root；如果设置为空，Engine 也不会继续以 root 身份运行（数据库服务器也会单独强制执行此策略）。建议工作室专门创建一个登录账户，用作 Engine 进程的所有者 ID，主要用于封装所需权限模型。某些情况下也可以使用已有 “service” 账户或 nobody，但应避免这样做，以免产生冲突和混淆。

"EngineOwner": "tractor",

 

ListenerPort

ListenerPort 是 Engine 用于接受来自 Web 浏览器、Blade 和提交器 的客户端请求的 TCP/UDP 服务端口。指定的端口值不得被其他应用程序（例如 Web 服务器）占用，必须能被站点网络上其他主机访问，并且 Engine 进程的所有者必须有权限使用它。

80 是客户端连接 Engine 时使用的内置默认值。如果你在这里选择不同值，则你 必须 同时在 TRACTOR_ENGINE 环境变量或 Blade 和提交器的 --engine 参数、tq 会话，以及 Dashboard 浏览器和管理脚本的每个 URL 连接中提供该值。关于小型网络上的替代方式，另请参见下面关于 EngineDiscovery 的说明；关于大型网络，请参见 ListenBacklog 设置。

通常，ListenerPort 是一个简单整数端口号，例如 80。

或者，也可以指定完整的 “interface and port”（接口和端口）对，使 Engine 只监听给定网络接口，而不是监听所有接口。使用带引号字符串（例如 localhost:8181 或 10.0.0.1:80）指定接口+端口对。接口部分可用点分四组表示法，也可以是可解析主机名。

"ListenerPort": 80,

 

EngineDiscovery

EngineDiscovery 控制 Engine 在 LAN 上对其监听地址公告，用于替代可用 DNS 或其他名称服务器。Tractor 组件可以使用轻量级组播发现方案（UPnP SSDP 功能的一个子集）相互发现。可能的设置包括：

  • 空 JSON 字符串 "" 会告诉 Engine 禁用组播公告；对于希望减少网络杂讯的大型站点，这可能是理想设置。

  • @ 会告诉 Engine 始终组播，并广告 Engine 自己通过命令行参数和自省获得的 IP 地址与端口；此设置适合希望尽量减少配置工作，并确保发现服务始终开启的小型工作室。

  • tractor-engine 或其他主机名会启用 “multicast if unknown” 模式。只有在给定主机名 UNKNOWN 时，Engine 才会发送公告。也就是说，如果你的名称服务器（DNS、AD、LDAP、/etc/hosts）已经能解析给定名称，就不会使用这些组播发现包。此设置会为已经有 "tractor-engine" 主机名别名的工作室避免组播流量，同时为没有其他配置变更的小型站点提供发现。

默认值为：

"EngineDiscovery": "tractor-engine",

 

LicenseLocation

Tractor 会通过检查与其他 Pixar 应用程序相同的许可证文件位置来查找许可证信息。它期望在主 Pixar 应用程序目录的根目录中找到 pixar.license 文件，该目录被假定为应用程序特定安装位置的父目录。如果站点需要除此之外的特殊行为，管理员可以使用环境变量 PIXAR_LICENSE_FILE 指定许可证信息，或使用 LicenseLocation 显式命名文件或许可证服务器。只有当共享位置不可用时，才使用这里的某个设置。

"LicenseLocation":         "${TractorInstallDirectory}/../pixar.license",
"LicenseLocation":         "${PIXAR_LICENSE_FILE}",
"LicenseLocation":         "9010@pixarserver",

 

MaxConcurrentDispatch

默认情况下，Engine 会消耗所有可用 Tractor 许可证席位，并用该值决定渲染农场上可启动的并发进程数量。你可以在这里指定最大值，限制给定 tractor-engine 使用的并发分派数量（以及许可证席位数量）。这通常只在测试时有用，例如测试 Engine 应相对于主生产 Engine 限制许可证使用量。使用数字 0 表示 “use all available”（使用全部可用席位）。

"MaxConcurrentDispatch": 255,

 

EngineWorkerThreads

EngineWorkerThreads 设置线程总数，这些线程会在 Engine 进程内的若干角色之间分配。它们通常用于高延迟 I/O 操作，例如写入客户端事务套接字。线程并行性允许分派和 UI 操作在对先前请求的回复送达期间继续进行。由于这些线程主要用于隐藏延迟，它们通常不会消耗大量 CPU。注意，除了这里给出的 “worker” 池计数，Engine 还会分配若干其他必需子系统线程。这个计数会预先分配给几个主要内部 worker 线程池，典型运行值为 10 + (number_of_blades / 100)。使用值 0 可让 Engine 只根据 Engine 核心计数选择默认。

"EngineWorkerThreads": 32,

 

TractorDataDirectory

Tractor 数据目录至少包含两类不同子目录。psql/ 子目录保存用户提交进来的作业，以及有关其 进度的检查点信息。user/ 子目录保存与每个 Tractor 用户相关的文件，主要是 Dashboard 配置设置。

数据目录中的所有文件都应视为重要资产。它们应纳入定期备份。Tractor Engine 在能以高带宽、低延迟访问该目录中的文件时性能最好。

默认值是 /var/spool/tractor。

"TractorDataDirectory": "/ssd/tractor",

 

EngineLogFile

EngineLogFile 指定 Engine 自身诊断日志使用的日志文件名。使用空字符串 "" 可让日志打印到终端。默认是记录到数据目录中的 engine.log。${TractorDataDirectory} 用于指定数据目录路径；如果数据目录变更，它也会相应调整。

"EngineLogFile": "${TractorDataDirectory}/engine.log",
"EngineLogFile": "/tmp/tractor.log",
"EngineLogFile": "",

 

EngineLogVerbosity

EngineLogVerbosity 指定日志严重级别阈值。可识别值按详细程度递增顺序为：SEVERE、NOTICE、INFO、DEBUG 和 TRACE。默认值为 NOTICE。

"EngineLogVerbosity": "DEBUG",


Command Logging（命令输出日志）

下面的一些参数用于存储和检索 tractor 启动的命令输出。某些值可以指定一个 模板字符串，其中可包含如下替换模式：

  • %u 表示作业所有者的登录名（用户 ID，字符串）

  • %j 表示作业 ID（整数）

  • %t 表示任务 ID（整数）

 

SiteCmdLogServerStartup

SiteCmdLogServerStartup 指定 Engine 启动时应如何自动启动 Python 日志服务器。使用空字符串表示 Engine 不应启动日志服务器。默认值为空字符串。

"SiteCmdLogServerStartup": "'${TractorInstallDirectory}/bin/tractor-cmdlogger' --filetemplate='${TractorDataDirectory}/cmd-logs/%u/J%j/T%t.log' --port=9180",

 

SiteCmdLogRetrievalURL

SiteCmdLogRetrievalURL 会发送给 UI，为其提供一个用于检索命令输出日志的 URL。

某些站点配置允许 UIs 直接从存储日志的文件服务器获取这些日志，例如另一个 Web 服务器（如 Apache）可直接访问日志文件时。默认情况下，tractor-engine 本身充当 Web 服务器，把日志文本交付给 UI 浏览器，前提是它能访问实际输出日志。Blade 在 blade.config 中被告知日志放在哪里。默认情况下，Engine 会启动一个简单 Python 日志接收子进程（见上面的 SiteCmdLogServerStartup），默认 blade.config 会指示 Blade 连接到它进行日志记录。在这种场景中，如果 Engine 提供日志，则下面给出的 relative（相对路径形式） URL 就足够；该 URL 不提及主机。已经连接到 Engine 进行 UI 服务的 UI 会获得该 URL，并使用同一个被查询的主机来获取 URL。

你可以通过包含 http://HOST 前缀来显式指定不同主机。默认值为 /tractor/cmd-logs/%u/J%j/T%t.log。

"SiteCmdLogRetrievalURL": "http://HOST:PORT/tractor/cmd-logs/%u/J%j/T%t.log", "SiteCmdLogRetrievalURL": "/tractor/cmd-logs/%u/J%j/T%t.log",

 

SiteURLMap

SiteURLMap 是入站 URL 资源前缀到磁盘资产的映射。这是一个简单 JSON 列表，由字符串对组成；每个字符串对的第一项是前缀，第二项是位置。 顺序很重要，因为前缀会按给定顺序测试，并使用第一个匹配前缀。未列出前缀的资产会被假定在 /tractor/ 下面。

"SiteURLMap": [
   "/tractor/cmd-logs/",   "${TractorDataDirectory}/cmd-logs/",
   "/tractor/dashboard/",  "${TractorDashboardDirectory}/tv/",
   "/tractor/tv/",         "${TractorDashboardDirectory}/tv/",
],

 

SiteContentTypeMap

SiteContentTypeMap 定义当加载非 Tractor 文件（如图像预览或日志文件）时，tractor 发送给 Web 浏览器的 HTTP Content-Type 头。某些站点可能有特殊浏览器插件或已注册辅助应用程序；如果 Content-Type 头设置恰当，它们会自动显示特定文件类型。反之，如果类型设置为浏览器不识别的内容，通常会允许用户在下载文件或启动自己选择的应用程序之间选择。Tractor 还内置一个用于常见类型的映射表；如果文件扩展名没有在这里显式列出，就会使用它。

"SiteContentTypeMap": {
   ".tif":  "image/tiff",
   ".exr":  "image/x-exr",
   ".png":  "image/png",
   ".jpg":  "image/jpeg",
   ".log":  "text/plain"
},

 

SiteMaxListReplyCount

SiteMaxListReplyCount 会截断发送给 Dashboard 的作业或 Blade 数量，以限制 Engine 创建回复所花处理时间，并避免某些 Web 浏览器在解析很大的回复列表时可能发生 JavaScript 崩溃。默认值为 2500。

"SiteMaxListReplyCount": 1000,

 

JobScriptQuotingMode

Tractor 可以接受 Alfred 作业脚本格式的作业描述。Alfred 对作业脚本中嵌套字符串要求某些引用和转义约定（需要额外一层转义字符）。Tractor 可以为了旧版作业生成器而遵循相同约定，也可以为较新作业接受更简单引用风格。使用值 alfred 表示 Alfred 兼容性，使用值 tractor 表示更简单方案。默认值为 tractor。

"JobScriptQuotingMode": "alfred",

 

JobSchedulingMode

JobSchedulingMode 决定给 Blade 分配时考虑任务的顺序。

P+FIFO 是把可用 Blade 分配给作业的默认方案；它总是选择优先级值最高的作业，如果多个作业具有相同优先级，则选择最早提交的那个。如果可用 Blade 多于第一个作业中可运行任务的数量，就会考虑下一个作业。

P+RR 是一种简单的 "Round-Robin（轮转）" 方案，用于考虑相同优先级的作业。多个具有许多 ready（就绪）任务的作业可以并发运行，不过它倾向于偏向带有长时间运行任务的作业，因为这些作业会在每个后续轮次中 收集额外 Blade。生产设置中通常更推荐 “ATCL” 模式。

P+ATCL, Active Task Count Leveling，首先按严格优先级对作业排序，然后在具有相同优先级的作业块内，优先把可用 Blade 分配给 active（活动）任务最少的作业。此模式倾向于给每个作业分配大致相同数量的 Blade，同时偏向较旧作业而不是新提交作业。由于它保持大致相等 Blade 计数，带有短时间运行任务的作业会比拥有相同数量长时间运行任务的作业更早完成。

P+ATCL+RR 与上面的 P+ATCL 类似，但当等待任务多于 Blade 时，它处理作业之间 Blade 均匀分布的方式不同。它添加了一种轮转组件，倾向于随时间在优先级块中的 所有 作业之间均衡 Blade 计数。通常，P+ATCL 偏向最旧作业，P+ATCL+RR 偏向等待 Blade 时间最长的作业。

注意：除 P+FIFO 以外的任何调度模式都会在 Blade 分配期间产生额外计算开销，因此会在一定程度上降低系统吞吐量。观察到的影响幅度会随整体作业负载、渲染农场规模、任务组合和 Engine 主机而变化。默认模式为 P+FIFO。

"JobSchedulingMode": "P+ACTL",

 

DispatchTiers

队列上的每个作业都关联到一个分派层级，通常只是默认层级。除了通常的作业优先级和当前调度策略模式之外，层级提供了一个额外控制层，用于控制作业分派到渲染农场的顺序。具体来说，来自高值层级的作业会先于低值层级的作业被考虑。这种抽象分类和排序在若干情况下可能有用，但层级通常在某种调度管理“危机”（无论大小）期间最有帮助。例如，可以把几个重要作业移到更高层级，同时保持其作业优先级不变，从而保留其相对排序。每个层级也可以在 Dashboard 中单独 paused（暂停），因此可临时暂停或启用整组作业的分派。例如，可以暂停常规分派，同时允许一系列管理作业继续执行。目前可配置层级属性非常简单，只有组名以及相对于其他层级的优先级。未来版本中可能会提供附加属性和行为。Dashboard 也提供一些控件，用于基于层级过滤和排序作业。下面显示默认层级配置。

"DispatchTiers": {
   "admin":   {"priority": 111.0},
   "rush":    {"priority":  99.0},
   "default": {"priority":  50.0},
   "batch":   {"priority":  10.0}
},

 

SiteHttpOriginAccess

SiteHttpOriginAccess 控制 Access-Control-Allow-Origin 头行的出站 HTTP 协议回复值。此设置允许 Engine 作为 Web 服务器在站点的 CORS 策略限制内工作。为方便使用，默认值是 *（asterisk）。使用空字符串可 禁用 CORS 风格共享。此设置主要影响 Tractor Dashboard 在指定日志服务器主机名与加载 Dashboard 本身所用 URL 中的主机名不同时 获取命令日志文件的能力。默认值是 *。

"SiteHttpOriginAccess": "*",

 

CmdAutoRetryAttempts 和 CmdRetryrcRetryAttempts

CmdAutoRetryAttempts 和 CmdRetryrcRetryAttempts 控制 tractor-engine 对以非零退出状态退出的命令进行自动重试的尝试数量。这些自动重试 可能 对经常因临时瞬态条件遇到命令失败的站点而言是有用的变通方法。

在某些作业脚本中，某些 RemoteCmds 会用 -retryc {5 17 84} 选项标注，其中给出的实际数字代表已启动应用程序的已知退出码，这些条件值得重试。此情况下重试尝试的数量由下面的 CmdRetryrcRetryAttempts 控制。设置为 0 可禁用 retryrc 重试，或设置为较小的整数以允许这些重试尝试。

CmdAutoRetryAttempts 用于控制未列在 Cmd 的 -retryrc 选项中的退出代码的自动重试，或者在未指定 -retryrc 时，对任何非零退出的自动重试。设置为 0 可禁用这些自动重试，或设置为较小的整数以全局强制这些重试。

在 Unix 风格平台上，因 SIGINT、SIGTERM 或 SIGKILL 而退出的命令 不会 重试，因为这些信号几乎总是有意的外部操作，意在实际停止给定任务继续执行。在 Windows 上，中断正在运行的子进程的请求由 TerminateProcess 调用处理；Windows Blade 会使用代码 "-15" 报告这些退出，它类似于 SIGTERM，即不会触发 Engine 中的自动重试。

默认设置如下所示。

"CmdAutoRetryAttempts": 0, "CmdRetryrcRetryAttempts": 1,

 

InitialJobLoadProgressLog

初始启动时，Engine 会从作业数据库 加载先前提交的作业，以继续 分派未完成作业。加载这些作业时，Engine 可以在每加载 N 个作业后向其日志打印一条 DEBUG 级诊断信息。InitialJobLoadProgressLog 控制报告该诊断信息的作业数量。将此值设置为 0 可 抑制这些日志条目。默认值为 1000。

"InitialJobLoadProgressLog": 5,


LogLongRunningRequests

此设置控制一些 额外 Engine debug 级诊断信息，相关对象是长时间等待和长时间运行的 HTTP 请求。两个参数值是阈值 [WAIT, RUN]，当 入站（或内部）HTTP 请求超过给定等待时间或运行时间（单位秒）时，会触发 Engine 日志中的诊断条目。使用 0 可禁用 任一 check。

"LogLongRunningRequests": [0.0, 5.0],


ListenBacklog

此设置是连接计数，用于控制为保存来自 TCP 客户端的未处理的入站 “connect” 请求而分配的存储量。这些通常是来自 tractor-blade 和 Dashboards 的 HTTP 请求。传统 SOMAXCONN 编译时值对负载下的现代 Web 服务器来说太小，因此需要运行时控制。使用值 0 会使 tractor-engine 向操作系统查询当前已配置的最大监听积压。

"ListenBacklog": 0,

这些操作系统值通常由系统管理员在针对特定工作室使用场景仔细调优 Engine 主机时配置。例如，

Linux: /proc/sys/net/core/somaxconn macOS: sysctl kern.ipc.somaxconn

注意，必须 重启 Engine 才能获取此文件中值的变更，或已变更的系统配置值。此外，通常还需要其他 协调的操作系统调优变更，例如 Linux 上的 /etc/sysctl.conf：

net.ipv4.tcp_max_syn_backlog = 2048
net.core.somaxconn = 1024
net.ipv4.tcp_syncookies = 1

然后使用以下命令应用这些变更： sysctl -f /etc/sysctl.conf

 

SiteSMTPServer 和 SiteDefaultMainDomain

站点 SMTP 服务器可供 Tractor 用于向你站点的管理员发送重要消息，或在某些情况下完成用户账户创建。它会通过 SMTP 协议连接到 SiteSMTPServer 指定的主机来投递消息。它会把 SiteDefaultMainDomain 用作简单用户名的电子邮件地址后缀。例如，给定用户 ID 为 eve 且域为默认空字符串，则邮件投递会简单尝试在 SMTP 服务器上使用 eve；如果域指定为 buyandlarge.com，则收件人将指定为 eve@buyandlarge.com。

默认值如下：

"SiteSMTPServer": "smtp", "SiteDefaultMailDomain": ""
