Python 中的作业规范与作业编写

注意 关于从你自己的 Python 脚本创建和提交作业的详情，请参见 Python Job Authoring API 相关讨论。

对于简单的单命令作业，例如渲染一个网络挂载的 RIB 文件，也可以使用 tractor-spool 命令行选项让它为你创建作业：

tractor-spool -c prman -Progress /net/myfiles/frame1.rib 或 tractor-spool -c /usr/bin/printenv

下方章节描述的是一个 不同 概念：用于静态定义单个作业结构的“传统”作业文件的格式和语法，也就是作为 数据文件 的格式。此类作业文件通常由其他程序写出，然后使用 tractor-spool 程序提交到 Tractor 作业队列。
 

静态作业规范概要

Tractor 作业文件通常表示为 Tcl 脚本，其中使用 Tractor 作业解析器可识别的特殊作业定义操作符。此格式最初为 Alfred 系统引入，Tractor 仍兼容几乎所有 Alfred 构造。

Tractor 还支持一组持续演进的 JSON 作业规范格式，这些格式在 单独的说明文档

中描述。Tractor 作业输入脚本（alfscript 格式）由使用以下 Tcl 操作符的表达式组成：
 

Job [options]

Task {title} [options]

RemoteCmd {launch_expr} [options]

Cmd {launch_expr} [options]

Instance {task_title_to_reference}

Iterate varname -from n -to m -by i -template {script} [options]

Assign varname {value_string}

有关选项和操作符语法的详细描述，请参见 操作符详情。


介绍

作业脚本描述要由 Tractor（或 Alfred）分派和监控的工作。作业脚本通常由应用程序在提交新的渲染作业时创建。也就是说，下面的语法讨论大概只对开发兼容 Tractor 的应用程序的人有用。

作业脚本支持一种简单的层级结构方案，可编码依赖关系，而这些依赖关系会进一步决定任务执行顺序；从这个角度看，作业脚本有些类似 makefiles。预定义的作业脚本操作符包括 Task、Cmd 和 Instance。还有一些特殊约定，用于把某些运行时值嵌入命令表达式。在作业初始化期间解析作业脚本时，Tractor 会同时构建 Engine/Blade 的内部执行层级，以及 Dashboard 的显示层级。随后 Engine 会持续遍历这个内部表示，寻找启动新工作的机会。

表达式 语法 由一些简单的列表构造规则组成。具体来说，每个操作符接受一个参数字符串列表，其中有些参数可以是可选的。字符串使用双引号或大括号分隔；字符串可以包含换行，也允许为空。大小写通常很重要，而空白通常不重要。当换行和分号不在字符串中时，它们是表达式结束标记。反斜杠用于像 C 一样生成具有语法意义字符的字面形式以及其他特殊字符代码（也用于禁用换行处理）。开头的井号（#）引入单行注释。更多信息请参见 特殊字符 相关讨论。

层级嵌套通过在父 Task 操作符的子任务字段中嵌入子 Task 描述来完成。只有使用大括号分隔符才能创建嵌套字符串。
 

一个简单示例

下面是一个定义示例任务层级的简单脚本。假设我们要计算两个阴影贴图，以及一个引用它们的最终帧；在这种情况下，必须先完成阴影贴图，才能开始渲染最终帧。“保留词”（保留字） 是 Job, Task，以及 RemoteCmd，这些操作符可以接受选项，例如以前导连字符开头的 -subtasks：

Job -title {A Simple Job} -subtasks {

    Task {Frame One} -subtasks {

        Task {Shadow A} -cmds {
            RemoteCmd {render light.1.rib} -service {PixarRender}
        }

        Task {Shadow B} -cmds {
            RemoteCmd {render light.2.rib} -service {PixarRender}
        }

    } -cmds {
        RemoteCmd {render beauty.1.rib} -service {PixarRender}
    }
}

此作业脚本描述了一个两级执行树，其中名为 Scene One 的任务会在它依赖的两个任务成功完成之后启动自己的命令（渲染 frame.1.rib）（这些依赖任务是 Shadow）。嵌套任务描述定义了深度优先的执行顺序；因此，light.1.rib 的渲染命令会先启动，然后是 light.2.rib。随后，当二者都完成后，才会渲染 frame.1.rib。命令会作为独立子进程按它们在脚本中出现的顺序启动。

注意，两个阴影命令彼此并不依赖，也就是说，它们不是相互嵌套的；因此它们代表一个并行处理机会。在资源充足时，Tractor 通常会启动两个命令，并允许它们并发执行。因此，脚本定义的是 启动顺序，某些命令在另一个命令启动时可能仍在运行，而这些并发命令可能以任意顺序完成。不过，在任何情况下，Task 都不会启动自己的命令，直到 所有 子任务都已成功完成各自命令。

此示例中的假设是：

  • 存在名为 render 的可执行文件，并且它位于当前搜索路径中。

  • 两个阴影贴图图像会在磁盘上创建到 frame.1.rib 期望找到它们的位置，也就是这些 RIB 文件内的文件系统引用保持一致。

Tractor 不会 尝试确认这类假设；这是作业生成器的职责。如果确实发生错误，该任务会变为 blocked（阻塞）状态，对应的 UI 控件会切换到 error（错误）状态，所有依赖该故障任务的任务都不会执行；不过，与它没有依赖关系的其他任务仍可继续。

关于 Task 需要注意的事项：

  • 标题字符串会显示在 Dashboard 中，因此最好选择简短但描述性强的标题。

  • 子任务列在 -subtasks 字段中；如果没有依赖关系，该字段可以为空或缺失。

  • -cmds 字段中的 RemoteCmd 列表描述依赖关系满足后要执行的实际动作；它可以为空或缺失；也可以包含多个用换行分隔的 RemoteCmd，这些 RemoteCmd 会按顺序处理。


进程启动与跟踪


远程服务器、启动表达式与运行时替换

传给 启动表达式 参数由 RemoteCmd 操作符使用，用于指定 Tractor 应启动的实际可执行文件。如上面的简单示例所示，它们可以是直接指定可执行文件及其参数的普通命令字符串。很多时候，命名的可执行文件只是一个本地编写的 shell 脚本。

Tractor 与 Alfred 一样，除层级启动顺序外，还提供其他关键分派功能，最重要的是 定位 可用远程服务器。考虑一个可能在终端中输入的命令：

prman -Progress /net/my_ribs/a.rib

如果希望 Tractor 在渲染农场中的一台可用机器上执行同一个命令，可以创建如下 Tractor 作业：

Job -title {a render test} -subtasks {
  Task -title {render a.rib} -cmds {
    RemoteCmd {prman -Progress /net/my_ribs/a.rib} -service {PixarRender}
  }
}

服务键

上例中的 RemoteCmd 选项 -service {PixarRender} 称为 服务键表达式，用于描述哪种 类型 的 Blade 可接受运行该命令。在此示例中，prman 可执行文件应在公布服务键 "PixarRender" 的 Blade 上启动。Tractor 在把命令匹配到 Blade 时，只是把这些键当作过滤器。比较命令键和 Blade 能力时会忽略大小写。

创建指定特定服务键的作业脚本后，还需要把匹配的服务键绑定到渲染农场中合适的 Blade，也就是你希望运行该类型命令的 Blade。例如，选择实际安装了 “prman” 的主机可能很重要。对于 Tractor，这意味着前往 blade.config 文件，并把适当的键添加到每种 Blade 配置档的 “Provides” 条目中。文件顶部有所有配置档共享的默认值。

在作业中使用哪些关键字是任意的；只需要为作业脚本和 Blade 配置档采用一致的约定，让二者同意某个已公布关键字对特定 Blade 能力的含义。按惯例，Pixar 应用程序例如 RenderMan for Maya 会生成使用特定服务关键字（例如 PixarRender）的 Tractor 作业。这些关键字旨在表达 Blade 上运行某些命令所需的同级 Pixar 软件类型。下面简要列出一些 RfM 生成的作业中常见服务键，以及它们想要表达的要求：

PixarRender

命令需要 Blade 上安装 RPS（prman）

PixarNRM

命令需要安装 RPS，并且需要正在运行的 netrender 服务器（tractor-blade.py 也是 netrender 服务器，因此实践中它通常等同于 PixarRender，但有助于区分 netrender 命令）

RfMRender

命令需要 Blade 上安装合适版本的 Maya，并且还必须安装 RfM 或 RMS；该命令会消耗渲染许可证

RfMRibGen

命令需要 Maya 和 RfM/RMS，不需要渲染许可证

注意： 除了 blade.config 中指定的关键字，每个服务器 Blade 始终还会提供三个额外隐式关键字：Blade 的 主机名，Blade 的 IP 地址（点分四组格式），以及它正在使用的 blade.config 配置档 名称。此功能很少使用，但通过把配置档或主机名指定为 RemoteCmd 的 --service 键，脚本可以把特定命令的执行限制到特定主机，或限制到名称匹配特定配置档的某类主机。通常最好引用更抽象的服务键类型。

Tractor 中还有一种高级模式，其中 Blade 的站点定义的 Python 插件可以按请求动态更改该 Blade 公布的服务。例如，站点插件可以在做这类动态决策时引用外部生产数据库。

服务键表达式

服务关键字可以使用简单语法组合，以进一步限制匹配给定命令的 Blade。例如：

RemoteCmd {prman -Progress vast.rib} -service {PixarRender,BigIron}

在此示例中，命令只会匹配配置档中同时提供 PixarRender AND BigIron 关键字的 Blade。

也可以在单个键名前加上感叹号，例如 "!xyz"，表示否定，意思是给定键必须 NOT 存在，Blade 才可接受。例如，为了避开提供名为 "Irix" 键的 Blade，作业脚本可以包含 "RemoteCmd -service {PixarRender,!Irix}"。或者，可以通过把该键添加到顶层 Job 规范、Dashboard 作业属性编辑器，或在 tractor-spool 命令行提交时指定，把同样效果应用到作业中的每个命令：

tractor-spool --service="\!Irix" myjob.alf

服务键通常是站点定义的抽象关键字，例如 blade.config 中 Blade 配置档的 “Provides” 规范所给出的键。它们也可以是特定主机名或整个配置档的名称。因此，如果要避免在名为 "darth" 的主机上渲染，可以把服务键 "!darth" 添加到 Job 的服务键。

不常见的 netrender 风格的客户端/服务器命令

某些命令，例如 netrender，是运行在 本地 提交主机上的客户端应用程序，但它们也要求将一个或多个 名称 作为命令行选项的一部分提供给 远程 服务器。scp 或 ftp 等应用程序同样是远程服务器的客户端。例如，考虑在终端中对 netrender 的典型调用：

netrender -h antigone -h percival /net/my_ribs/a.rib

在这种情况下，单个 RIB 文件会分配给两个渲染服务器，这些服务器假定可用，并且已经在运行 netrender 服务器（也就是说，每个服务器都有正在运行的 tractor-blade，因为它内置 netrender 支持）。本地 netrender 客户端应用程序会联系其命令行中列出的每个主机，并使用自定义 netrender 协议在那里启动渲染。通常希望 Tractor 从池中查找可用 netrender 服务器，而不是输入具体主机名（例如上面的 antigone 和 percival）。请求这组服务器的示例作业脚本可能如下：

Cmd {netrender %H /net/my_ribs/a.rib} -service {PixarNRM} -atleast 2

注意： 必须记住，Cmd 指令 并不会 实际把工作发送到通过这种方式选择的远程主机。它会启动本地应用程序，例如 netrender 或 ssh，随后这些应用程序自行管理与远程服务器的交互。 RemoteCmd 应当用于直接把命令发送到远程主机，而不经过本地客户端。
 

替换

当以下符号出现在启动表达式或消息表达式中时，会在启动时展开：

~

主目录展开，与 csh(1) 中相同

%h

替换为通过动态 -service 机制绑定到当前 Cmd 的主机名。这是一个以空格分隔的简单主机名列表（对 rsh 等很有用）。

%H

类似 %h，但格式化为 -h 主机名对（netrender 要求的格式）。

%n

转换为已绑定槽位的计数；这是一个整数，表示绑定到此命令的槽位数。

%j

展开为当前作业的内部调度器 Job 标识符（也称 "jid"）。Tractor Engine 会为其队列中的作业创建唯一 ID。

%t

展开为当前任务的 Task 标识符（也称 "tid"），它只在每个作业内唯一。

%c

展开为当前命令的 Command 标识符（也称 "cid"），它只在每个作业内唯一。

%r

恢复模式；当任务全新启动时展开为 0；当用户或系统请求从检查点恢复时，展开为大于 0 的整数。

%R

展开为因 -resumewhile 构造而重新启动的命令的 "循环计数"。

%q

质量提示；当任务因其所依赖子任务的最终质量运行而执行时展开为 1.0；如果某个子任务在当前 resumewhile 循环中只达到检查点，则该值会小于 1.0。

% idref （主机）

类似 %h，但使用其 -id 值为 idref的命令中的主机名。

% idref （-host）

与上面相同，但格式化为 -h 主机名 对（主机名对）。

%D (路径)

DirMaps使用站点定义的映射表对路径应用按架构重映射。

%%

替换为单个百分号。


作业上下文环境变量

tractor-blade 在启动每个命令之前会设置以下环境变量。在应用程序本身需要了解某些 Tractor 相关上下文信息的情况下，它们可能有用。有关全局或按项目/命令控制自定义环境变量的详情，请参见 自定义环境处理器 中关于自定义环境变量控制的相关讨论。

TR_ENV_JID

派发该命令的作业的作业 ID 编号。

TR_ENV_TID

作业内对应任务的 ID 编号。

TR_ENV_CID

作业内对应命令的 ID 编号。

TR_ENV_JOB_PROJECT

作业的项目归属。名称（或多个名称）在作业提交时指定，通常由作业创建应用程序指定。

TR_ENV_TRACTOR_ROOT

正在运行的 Blade 的 Tractor 安装树顶部路径。


在多个命令之间共享一个服务器签出

有时获取一个远程服务器，并在该服务器上按顺序运行多个命令会很有用。这称为 共享服务器 场景。实现此目标的机制是向外层 Task 添加服务器签出规范，然后在每个需要该服务器槽位的 Cmd 或 RemoteCmd 上引用任务 ID。

例如：

Task {SharedServer example} -id {bob} -service {pixarRender} -cmds {
      Cmd {rsh %h mkdir /tmp/somedir} -refersto {bob}
      Cmd {rsh %h render -Progress some.rib} -refersto {bob}
} -cleanup {
      # Clean-up commands go here.
      # This example uses RemoteCmd just to illustrate its use as an
      # alternative to rsh above.  Note that there is an implicit '%h'
      # used to determine where the command should be run.
      #
      RemoteCmd {/bin/rm -rf /tmp/somedir} -refersto bob
} -subtasks {
      # the description of any nested dependent tasks go here
      [...]
}

签出的生命周期由两个因素管理。初始任务级签出是惰性的，也就是说，只有当引用它的某个命令实际成为下一个要执行的命令时才会发生。引用计数用于判断何时可以安全地归还槽位；最后一个引用它的命令完成或出错后，它会被释放。

语法说明： 允许用于 idref 名称的字符包括字母、数字、句点（.）和下划线（_）。对于上方所有 % 替换，可以使用大括号来分隔名称，方式类似 csh 或 Tcl；例如，如果当前调度为名为 "percival" 的主机定义了类型为 "renderserver" 的服务，那么包含以下命令的 Task：

RemoteCmd {renderer -use %{h}.settings} -service {renderserver}

可能会导致启动以下命令：

renderer -use percival.settings

在这种情况下，大括号是必需的，因为更简单的字符串 "%h.settings" 会使替换机制查找带有 "-id h.settings" 的 Task 或 Cmd，并使用其当前值。 

Tractor 脚本中的特殊字符与转义

Tractor 会对提交给它的脚本做两遍处理。第一遍是提交时 解析 遍，用于构建作业形态，也就是设置任务层级。第二遍（以及可能后续相同的遍）是分派、运行时遍，在此期间命令会被启动，并进行运行时替换等。

脚本的初始解析使用围绕 John Ousterhaut 的 Tcl 解释器构建的系统完成。因此，Tcl 语法 规则会应用于命令参数和字符串构造。脚本由对 Job, Task, Cmd，以及 Instance 命令（操作符）的调用组成，而这些操作符又接受字符串作为参数。对于 Task，-subtasks 选项可以包含附加脚本，并递归解析。总体上，Tcl 规则在应用于操作符参数时，比 Bourne shell（sh）或 csh(1) 的规则简单得多。shell 程序员应注意以下几点：

  • 启动表达式会被标记化并直接传给 execvp(2)；因此，分号分隔的命令序列、命令管线、环境变量引用和文件名元字符（例如 *、?、[]）不会直接支持；应改用适当的 子 shell).

  • Tractor（以及 Alfred）会像 csh 一样执行波浪号展开。

  • 单引号没有特殊含义，也就是说它们不会把词组合在一起；请使用大括号 - {} - 进行分组。

  • 每个参数字符串只会经过一次词法切分遍。

例如，考虑使用 find(1) 从目录树中删除以 "preview" 开头的文件。在 csh(1) 脚本中，该命令可能如下所示：<

find ~bob -type f -name 'preview.*' -exec /bin/rm {} \;

也就是说，星号必须从 shell 文件名展开中转义，因为它会由 find 命令直接使用；exec 表达式后的分号也是如此。大括号不会被 shell 触碰（因为它们不是变量表达式的一部分）。如果类似命令是 Tractor/Alfred 脚本的一部分（例如作为 Task 的清理命令），它会如下所示：

Cmd {find ~bob -type f -name preview.* -exec /bin/rm \{\} ;}

Shell 管线和其他表达式
 

尽管上面描述了对 Cmd 启动表达式的限制，但有时 shell 构造（例如命令管线或运行时文件名展开）仍然非常有用。在这些情况下，一个简单方案是把适当的 shell 作为 Cmd 启动，并通过命令行参数将管线表达式传给该 shell：

Cmd {/bin/sh -e "cd /tmp/frames; ls -1 | xargs -I+ cp + /DDR"}

脚本作者应阅读自己所选 shell 的文档，以理解各种调用选项的影响。例如，必须决定调度器启动上述命令时，是否应执行用户的 .cshrc 或 .profile。

另一种方法是使用 Cmd -msg 选项把任意表达式传给已启动 shell。这本质上等同于上面的方法，但不那么紧凑；不过，如果需要，它确实允许持久复用 shell。考虑下面这些发送邮件的示例，这在作业结束时有时很方便。回想一下，Mail 的 -s 选项会把下一个“词”当作主题，因此空格需要转义（这次使用 csh(1) 语法）：

Cmd {/bin/csh -fc "/usr/sbin/Mail -s 'job done' jean &lt; ~/errlog"} Cmd {/bin/csh -fet} -msg {/usr/sbin/Mail -s 'job done' jean &lt; ~/errlog}

通常 最简单的方案 是用你喜欢的语言编写一个简短 shell 脚本，并从 Tractor 启动该脚本，以处理复杂表达式：

Cmd {myscript}

如果需要远程服务器，可以把运行时主机选择作为参数传给脚本：

Cmd {myscript %h} -service {someServerType}


进度或完成百分比指示

Tractor 还会扫描它启动的应用程序的 stdout，查找如下形式的字符串：

TR_PROGRESS nnn% ALF_PROGRESS nnn%

整数 nnn，范围为 0-100，会发送到 UI，并用于控制活动任务节点上绘制的完成百分比进度条。


Exit Status（退出状态）

Tractor 还会扫描此指令：

TR_EXIT_STATUS nnn ALF_EXIT_STATUS nnn

整数 nnn 用于确定成功或失败，而不是使用实际程序退出状态。此覆盖最常用于表达库内部发生了阻塞作业的错误，即使已知调用应用程序会以状态码 0 退出。（0 表示成功完成，任何非零值表示失败，并导致阻塞任务错误）。相反，有些包装器脚本会在已知已启动应用程序以非零代码退出但属于非致命情况时，以这种方式发出退出码 0。所有合法应用程序退出代码都是 0 到 255（含）之间的整数。Tractor 会使用负数报告因中断信号导致的退出，也就是负信号值。当 tractor.config 被配置为在 error（错误）状态时启用自动任务重试，除 -2、-15 和 -9 之外，所有非零退出代码都会导致重试；这些例外值被认为是由用户操作导致的有意终止。注意，如果应用程序发出 TR_EXIT_STATUS 但没有在两秒内实际退出，则 tractor-blade 会自动开始关闭应用程序进程组；可以在 blade.config 中添加此配置档条目来禁用自动清扫："TR_EXIT_STATUS_termination": 0
