Python 中的 Job Specification 与 Job Authoring

注意 关于从你自己的 Python script 创建和 spool job 的详情，请参见 Python Job Authoring API 相关讨论。

对于简单的 single-command job，例如 rendering 一个 network mounted RIB file，也可以使用 tractor-spool command-line option 让它为你创建 job：

tractor-spool -c prman -Progress /net/myfiles/frame1.rib 或 tractor-spool -c /usr/bin/printenv

下方 section 描述的是一个 不同 概念：用于静态定义单个 job structure 的 "classic" job file 的 format 和 syntax，也就是作为 data file 的格式。此类 job file 通常由其他 program 写出，然后使用 tractor-spool program spool 到 Tractor job queue。
 

静态作业规范概要（Job Specification）

Tractor Job file 通常表示为 TCL script，其中使用 Tractor job parser 可识别的特殊 job definition operator。此 format 最初为 Alfred system 引入，Tractor 仍兼容几乎所有 Alfred construct。

Tractor 还支持一组持续演进的 JSON job specification format，这些 format 在 单独的说明文档 document

中描述。Tractor job input script（alfscript format）由使用以下 Tcl operator 的 expression 组成：
 

Job [options]

Task {title} [options]

RemoteCmd {launch_expr} [options]

Cmd {launch_expr} [options]

Instance {task_title_to_reference}

Iterate varname -from n -to m -by i -template {script} [options]

Assign varname {value_string}

有关 option 和 operator syntax 的详细描述，请参见 Operator Details。


介绍

job script 描述要由 Tractor（或 Alfred）dispatch 和 monitor 的 work。job script 通常由 application program 在 spool 新 rendering job 时创建。也就是说，下面的 syntax 讨论大概只对开发 Tractor-compliant application 的人有用。

job script 支持一种简单的 hierarchical structuring scheme，可编码 dependency，而这些 dependency 会进一步决定 task execution 的顺序；从这个角度看，job script 有些类似 makefiles。预定义的 job script operator 包括 Task、Cmd 和 Instance。还有一些特殊 convention，用于把某些 run-time value 嵌入 command expression。在 job initialization 期间解析 job script 时，Tractor 会同时构建 engine/blade 的 internal execution hierarchy，以及 Dashboard 的 display hierarchy。随后 engine 会持续遍历这个 internal representation，寻找 launch 新 work 的机会。

expression syntax 由一些简单的 list-formation rule 组成。具体来说，每个 operator 接受一个 argument string list，其中有些 argument 可以是 optional。string 使用 double-quote 或 curly-brace 分隔；string 可以包含 newline，也允许为空。case 通常很重要，而 whitespace 通常不重要。当 newline 和 semicolon 不在 string 中时，它们是 end-of-expression marker。backslash 用于像 C 一样生成 syntactically significant character 的 literal version 以及其他 special character code（也用于禁用 newline processing）。开头的 hash sign（#）引入 one-line comment。更多信息请参见 special characters 相关讨论。

hierarchical nesting 通过在 parent Task operator 的 subtask field 中嵌入 child Task description 来完成。只有使用 curly-brace delimiter 才能创建 nested string。
 

一个简单示例

下面是一个定义示例 task hierarchy 的简单 script。假设我们要计算两个 shadow map，以及一个引用它们的 final frame；在这种情况下，必须先完成 shadow map，才能开始 rendering final frame。"reserved words" 是 Job, Task，以及 RemoteCmd，这些 operator 可以接受 option，例如以 leading hyphen 开头的 -subtasks：

Job -title {A Simple Job} -subtasks {

    Task {Frame One} -subtasks {

        Task {Shadow A} -cmds {
            RemoteCmd {render light.1.rib} -service {PixarRender}
        }

        Task {Shadow B} -cmds {
            RemoteCmd {render light.2.rib} -service {PixarRender}
        }

    } -cmds {
        RemoteCmd {render beauty.1.rib} -service {PixarRender}
    }
}

此 job script 描述了一个 two-level execution tree，其中名为 Scene One 的 task 会在它依赖的两个 task 成功完成之后 launch 自己的 command（render frame.1.rib）（这些依赖 task 是 Shadow）。nested task description 定义了 depth-first 的 execution order；因此，light.1.rib 的 render command 会先 launch，然后是 light.2.rib。随后，当二者都完成后，才会 render frame.1.rib。command 会作为独立 subprocess 按它们在 script 中出现的顺序 launch。

注意，两个 shadow command 彼此并不依赖，也就是说，它们不是相互嵌套的；因此它们代表一个 parallel processing opportunity。在资源充足时，Tractor 通常会 launch 两个 command，并允许它们 concurrent execute。因此，script 定义的是 launching order，某些 command 在另一个 command launch 时可能仍在运行，而这些 concurrent command 可能以任意顺序完成。不过，在任何情况下，Task 都不会 launch 自己的 command，直到 所有 sub-task 都已成功完成各自 command。

此示例中的假设是：

  • 存在名为 render 的 executable，且它位于当前 search path 中。

  • 两个 shadow map image 会在 disk 上创建到 frame.1.rib 期望找到它们的位置，也就是这些 RIB file 内的 file system reference 保持一致。

Tractor 不会 尝试确认这类假设；这是 job generator 的职责。如果确实发生 error，该 task 会变为 blocked，对应的 UI widget 会切换到 error state，所有依赖该 broken task 的 task 都不会执行；不过，与它没有 dependency 的其他 task 仍可继续。

关于 Task 需要注意的事项：

  • title string 会显示在 Dashboard 中，因此最好选择简短但描述性强的 title。

  • child task 列在 -subtasks field 中；如果没有 dependency，该 field 可以为空或缺失。

  • -cmds field 中的 RemoteCmd list 描述 dependency 满足后要执行的实际 action；它可以为空或缺失；也可以包含多个用 newline 分隔的 RemoteCmd，这些 RemoteCmd 会按顺序处理。


Process Launch 和 Tracking


远程服务器、启动表达式与运行时替换（Remote Server、Launch Expression、Runtime Substitution）

传给 launch expression 使用的参数 argument 由 RemoteCmd operator 使用，用于指定 Tractor 应 launch 的实际 executable。如上面的 simple example 所示，它们可以是直接指定 executable 及其 argument 的普通 command string。很多时候，命名的 executable 只是一个本地编写的 shell script。

Tractor 与 Alfred 一样，除 hierarchical launch ordering 外，还提供其他关键 dispatching feature，最重要的是 定位 可用 remote server。考虑一个可能在 terminal 中输入的 command：

prman -Progress /net/my_ribs/a.rib

如果希望 Tractor 在 render farm 中一台可用 machine 上执行同一个 command，可以创建如下 Tractor job：

Job -title {a render test} -subtasks {
  Task -title {render a.rib} -cmds {
    RemoteCmd {prman -Progress /net/my_ribs/a.rib} -service {PixarRender}
  }
}

Service Keys

上例中的 RemoteCmd option -service {PixarRender} 称为 service key expression，用于描述哪种 类型 的 blade 可接受运行该 command。在此示例中，prman executable 应在 advertise service key "PixarRender" 的 blade 上 launch。Tractor 在把 command 匹配到 blade 时，只是把这些 key 当作 "filter"。比较 command key 和 blade capability 时会忽略 capitalization。

创建指定特定 service key 的 job script 后，还需要把 matching service key "attach" 到 farm 中合适的 blade，也就是你希望运行该类型 command 的 blade。例如，选择实际安装了 "prman" 的 host 可能很重要。对于 Tractor，这意味着前往 blade.config file，并把适当的 key 添加到每种 blade profile 的 "Provides" entry 中。file 顶部有所有 profile 共享的 default。

在 job 中使用哪些 keyword 是任意的；只需要为 job script 和 blade profile 采用一致的 convention，让二者同意某个 advertised keyword 对特定 blade capability 的含义。按惯例，Pixar application 例如 RenderMan for Maya 会生成使用特定 service keyword（例如 PixarRender）的 Tractor job。这些 keyword 旨在表达 blade 上运行某些 command 所需的 sibling Pixar software 类型。下面简要列出一些 RfM-generated job 中常见 service key，以及它们想要表达的 requirement：

PixarRender

cmd 需要 blade 上安装 RPS（prman）

PixarNRM

cmd 需要安装 RPS，并且需要正在运行的 netrender server（tractor-blade.py 也是 netrender server，因此实践中它通常等同于 PixarRender，但有助于区分 netrender command）

RfMRender

cmd 需要 blade 上有合适的已安装 Maya，并且还必须安装 RfM 或 RMS；该 cmd 会消耗 render license

RfMRibGen

cmd 需要 Maya 和 RfM/RMS，不需要 render license

注意： 除了 blade.config 中指定的 keyword，每个 server blade 始终还会 "provide" 三个额外 implicit keyword：blade 的 host name，blade 的 IP address（dotted quad），以及它正在使用的 blade.config profile 名称。此 feature 很少使用，但通过把 profile 或 host name 指定为 RemoteCmd 的 --service key，script 可以把特定 command 的 execution 限制到特定 host，或限制到名称匹配特定 profile 的某类 host。通常最好引用更抽象的 service key 类型。

Tractor 中还有一种 "advanced" mode，其中 blade 的 site-defined python plug-in 可以按 request 动态更改该 blade advertise 的 service。例如，site plug-in 可以在做这类 dynamic decision 时引用 external production database。

Service Key Expression

service keyword 可以使用简单 syntax 组合，以进一步限制匹配给定 command 的 blade。例如：

RemoteCmd {prman -Progress vast.rib} -service {PixarRender,BigIron}

在此示例中，command 只会匹配 profile 中同时提供 PixarRender AND BigIron keyword 作为关键词的 blade。

也可以在单个 key name 前加上 exclamation point，例如 "!xyz"，表示 negation，意思是给定 key 必须 NOT 存在，blade 才可接受。例如，为了避开提供名为 "Irix" key 的 blade，job script 可以包含 "RemoteCmd -service {PixarRender,!Irix}"。或者，可以通过把该 key 添加到 top-level Job specification、Dashboard job attributes editor，或在 tractor-spool command line 的 spool time 指定，把同样效果应用到 job 中的每个 command：

tractor-spool --service="\!Irix" myjob.alf

Service key 通常是 site-defined abstract keyword，例如 blade.config 中 blade profile 的 "Provides" specification 所给出的 key。它们也可以是特定 hostname 或整个 profile 的名称。因此，如果要避免在名为 "darth" 的 host 上 rendering，可以把 service key "!darth" 添加到 Job 的 service key。

不常见的 netrender-style client/server command

某些 command，例如 netrender，是运行在 local spooling host 上的 client application，但它们也要求将一个或多个 名称 作为 command-line option 的一部分提供给 remote server。scp 或 ftp 等 application 同样是 remote server 的 client。例如，考虑在 terminal 中对 netrender 的典型 invocation：

netrender -h antigone -h percival /net/my_ribs/a.rib

在这种情况下，单个 RIB file 会分配给两个 rendering server，这些 server 假定可用，并且已经在运行 netrender server（也就是说，每个 server 都有正在运行的 tractor-blade，因为它内置 netrender support）。local netrender client application 会联系其 command line 中列出的每个 host，并使用 custom netrender protocol 在那里启动 rendering。通常希望 Tractor 从 pool 中查找可用 netrender server，而不是输入具体 hostname（例如上面的 antigone 和 percival）。请求这组 server 的示例 job script 可能如下：

Cmd {netrender %H /net/my_ribs/a.rib} -service {PixarNRM} -atleast 2

注意： 必须记住，Cmd directive 指令 并不会 实际把 work 发送到通过这种方式选择的 remote host。它会 launch local application，例如 netrender 或 ssh，随后这些 application 自行管理与 remote server 的 interaction。 RemoteCmd 应当用于直接把 command 发送到 remote host，而不经过 local client。
 

Substitutions

当以下 symbol 出现在 launch expression 或 message expression 中时，会在 launch time 展开：

~

主目录展开 home directory expansion，与 csh(1) 中相同

%h

替换为通过 dynamic -service 机制绑定到当前 Cmd 的 hostname。这是一个以空格分隔的简单 hostname list（对 rsh 等很有用）。

%H

类似 %h，但格式化为 -h hostname pair（netrender 要求的格式）。

%n

转换为 bound slot 的 count；这是一个 integer，表示绑定到此 command 的 slot 数。

%j

展开为当前 job 的 internal dispatcher Job identifier（也称 "jid"）。Tractor engine 会为其 queue 中的 job 创建 unique id。

%t

展开为当前 task 的 Task identifier（也称 "tid"），它只在每个 job 内唯一。

%c

展开为当前 command 的 Command identifier（也称 "cid"），它只在每个 job 内唯一。

%r

recover mode；当 task 是 fresh start 时展开为 0；当 user 或 system 请求从 checkpoint recover 时，展开为大于 0 的 integer。

%R

展开为因 -resumewhile construct 而 restarted 的 command 的 "loop count"。

%q

quality hint；当 task 因其所依赖 subtask 的 final quality run 而执行时展开为 1.0；如果某个 subtask 在当前 resumewhile loop 中只达到 checkpoint，则该值会小于 1.0。

% idref （host）

类似 %h，但使用其 -id value 为 idref的 command 命令中的 hostname 主机名。

% idref （-host）

与上面相同，但格式化为 -h hostname pair。

%D (path)

DirMaps使用 site-defined mapping table 对 path 应用 per-architecture remapping。

%%

替换为单个 percent-sign。


Job Context Environment Variables

tractor-blade 在 launch 每个 command 之前会设置以下 environment variable。在 application 本身需要了解某些 Tractor-related context information 的情况下，它们可能有用。有关全局或按 project/command 控制 custom environment variable 的详情，请参见 Custom Environment Handlers 中关于 custom environment variable 控制的相关讨论。

TR_ENV_JID

派发该 command 的 job 的 job ID number。

TR_ENV_TID

job 作业内对应 task 的 ID number。

TR_ENV_CID

job 作业内对应 command 的 ID number。

TR_ENV_JOB_PROJECT

job 的 project affiliation。名称（或多个名称）在 job submission time 指定，通常由 job creation application 指定。

TR_ENV_TRACTOR_ROOT

正在运行的 blade 的 Tractor install tree 顶部 path。


在多个 Command 之间共享一个 Server Check-Out

有时获取一个 remote server，并在该 server 上按顺序运行多个 command 会很有用。这称为 shared server scenario。实现此目标的机制是向外层 Task 添加 server check-out specification，然后在每个需要该 server slot 的 Cmd 或 RemoteCmd 上引用 task-id。

例如：

Task {SharedServer example} -id {bob} -service {pixarRender} -cmds {
      Cmd {rsh %h mkdir /tmp/somedir} -refersto {bob}
      Cmd {rsh %h render -Progress some.rib} -refersto {bob}
} -cleanup {
      # Clean-up commands go here.
      # This example uses RemoteCmd just to illustrate its use as an
      # alternative to rsh above.  Note that there is an implicit '%h'
      # used to determine where the command should be run.
      #
      RemoteCmd {/bin/rm -rf /tmp/somedir} -refersto bob
} -subtasks {
      # the description of any nested dependent tasks go here
      [...]
}

check-out 的 lifetime 由两个 factor 管理。initial task-level check-out 是 lazy 的，也就是说，只有当引用它的某个 command 实际成为下一个要执行的 command 时才会发生。reference count 用于判断何时可以安全地把 slot check back in；最后一个引用它的 command 完成或出错后，它会被释放。

Syntax Note： 允许用于 idref 名称的 character 包括 letter、number、period（.）和 underbar（_）。对于上方所有 % substitution，可以使用 brace 来分隔 name，方式类似 csh 或 Tcl；例如，如果当前 schedule 为名为 "percival" 的 host 定义了 type 为 "renderserver" 的 service，那么包含以下 command 的 Task：

RemoteCmd {renderer -use %{h}.settings} -service {renderserver}

可能会导致 launch 以下 command：

renderer -use percival.settings

在这种情况下，brace 是必需的，因为更简单的 string "%h.settings" 会使 substitution mechanism 查找带有 "-id h.settings" 的 Task 或 Cmd，并使用其 current value。 

Tractor Script 中的 Special Character 和 Escape

Tractor 会对提交给它的 script 做两遍处理。第一遍是 spool-time parsing pass，用于构建 job 的 "shape"，也就是设置 task hierarchy。第二遍（以及可能后续相同的 pass）是 dispatching、run-time pass，在此期间 command 会被 launch，并进行 run-time substitution 等。

script 的 initial parsing 使用围绕 John Ousterhaut 的 TCL interpreter构建的 system 完成。因此，TCL syntax rule 会应用于 command argument 和 string formation。script 由对 Job, Task, Cmd，以及 Instance command（operator）的 call 组成，而这些 operator 又接受 string 作为 argument。对于 Task，-subtasks option 可以包含 additional script，并递归解析。总体上，TCL rule 在应用于 operator argument 时，比 Bourne shell（sh）或 csh(1) 的 rule 简单得多。shell programmer 应注意以下几点：

  • launch expression 会被 tokenized 并直接传给 execvp(2)；因此，semicolon-separated command sequence、command pipeline、environment variable reference 和 filename meta-character（例如 *、?、[]）不会直接支持；应改用适当的 sub-shell).

  • Tractor（以及 Alfred）会像 csh 一样执行 tilde-expansion。

  • single-quote 没有特殊含义，也就是说它们不会把 word 组合在一起；请使用 brace - {} - 进行 grouping。

  • 每个 argument string 只会经过一次 tokenizing pass。

例如，考虑使用 find(1) 从 directory tree 中删除以 "preview" 开头的 file。在 csh(1) script 中，该 command 可能如下所示：<

find ~bob -type f -name 'preview.*' -exec /bin/rm {} \;

也就是说，asterisk 必须从 shell filename expansion 中 escape 出来，因为它会由 find command 直接使用；exec expression 后的 semicolon 也是如此。curly brace 不会被 shell 触碰（因为它们不是 variable expression 的一部分）。如果类似 command 是 Tractor/Alfred script 的一部分（例如作为 Task 的 clean-up command），它会如下所示：

Cmd {find ~bob -type f -name preview.* -exec /bin/rm \{\} ;}

Shell Pipeline 和其他 Expression
 

尽管上面描述了对 Cmd launch expression 的 restriction，但有时 shell construct（例如 command pipeline 或 run-time filename expansion）仍然非常有用。在这些情况下，一个简单方案是把适当的 shell 作为 Cmd launch，并通过 command-line argument 将 pipeline expression 传给该 shell：

Cmd {/bin/sh -e "cd /tmp/frames; ls -1 | xargs -I+ cp + /DDR"}

Script author 应阅读自己所选 shell 的 documentation，以理解各种 invocation option 的 implication。例如，必须决定 dispatcher launch 上述 command 时，是否应执行用户的 .cshrc 或 .profile。

另一种方法是使用 Cmd -msg option 把 arbitrary expression 传给 launched shell。这本质上等同于上面的方法，但不那么 compact；不过，如果需要，它确实允许 persistent reuse shell。考虑下面这些发送 mail 的 example，这在 job 结束时有时很方便。回想一下，Mail 的 -s option 会把下一个 "word" 当作 subject，因此 space 需要 escape（这次使用 csh(1) syntax）：

Cmd {/bin/csh -fc "/usr/sbin/Mail -s 'job done' jean &lt; ~/errlog"} Cmd {/bin/csh -fet} -msg {/usr/sbin/Mail -s 'job done' jean &lt; ~/errlog}

通常 最简单的方案 是用你喜欢的 language 编写一个 short shell script，并从 Tractor launch 该 script，以处理 complex expression：

Cmd {myscript}

如果需要 remote server，可以把 run-time host selection 作为 argument 传给 script：

Cmd {myscript %h} -service {someServerType}


Progress 或 Percent-done Indication

Tractor 还会扫描它 launch 的 app 的 stdout，查找如下形式的 string：

TR_PROGRESS nnn% ALF_PROGRESS nnn%

integer nnn，范围为 0-100，会发送到 UI，并用于控制 active task node 上绘制的 percent-done bar。


Exit Status

Tractor 还会扫描此 directive：

TR_EXIT_STATUS nnn ALF_EXIT_STATUS nnn

integer nnn 用于确定 success 或 failure，而不是使用实际 program exit status。此 override 最常用于表达 library 内部发生了 job-blocking error，即使已知 calling application 会以 status code zero 退出。（0 表示 successful completion，任何 non-zero value 表示 failure，并导致 blocking task error）。相反，有些 wrapper script 会在已知 launched app 以 non-zero code 退出但属于 non-fatal case 时，以这种方式 emit zero code。所有合法 application exit code 都是 0 到 255（含）之间的 integer。Tractor 会使用 negative number 报告因 interrupting signal 导致的 exit，也就是 negative signal value。当 tractor.config 被配置为在 error 时启用 automatic task retry，除 -2、-15 和 -9 之外，所有 non-zero exit code 都会导致 retry；这些例外值被认为是由 user action 导致的 intentional terminal。注意，如果 application 发出 TR_EXIT_STATUS 但没有在两秒内实际退出，则 tractor-blade 会自动开始关闭 application process group；可以在 blade.config 中添加此 profile entry 来禁用 automatic sweep："TR_EXIT_STATUS_termination": 0
