Custom Menu Items

Source: https://rmanwiki-27.pixar.com/space/TRA/22184304/Custom+Menu+Items
Downloaded: 2026-06-15T03:49:12+00:00

Dashboard 的上下文菜单可以通过自定义项扩展，用来调用站点指定脚本。要扩展上下文菜单，必须在站点配置目录中名为 menus.config 的 JSON 配置文件里添加一个条目。

示例：触发求助邮件

下面的示例演示如何通过自定义菜单项创建一封求助邮件。

配置条目

{
  "job":
  [
    {
      "comment": "artists use this to request technical assistance for selected jobs",
      "title": "ask for help",
      "crews": ["lighting", "animation"],
      "suppress": true,
      "exec": ["menu-scripts/askForHelp"],
      "values": ["jid", "owner"],
      "enabled": true
    }
  ]
}

在此示例中，菜单项会出现在作业列表的上下文菜单中，因为它定义在顶层字典中以 job 为键的列表里。task 和 blade 是自定义菜单项可以出现的另外两类菜单。

其中 comment 属性用于记录菜单项的用途。它目前在 Dashboard 中不承担功能性角色。

其中 title 属性定义菜单项在菜单中显示的名称。在此示例中，作业列表的上下文菜单会显示 ask for help。

其中 crews 属性将脚本运行限制为指定 crew 的用户。如果未列出 crew，任何用户都可以调用该菜单项。在此示例中，lighting 和 animation crew 中的用户被允许调用该菜单项。

其中 suppress 属性表示 Dashboard 是否会在新窗口中显示脚本输出。在本例中，没有有用的输出，因此把它设置为 false 有助于避免窗口膨胀。不过，一些脚本（例如报告生成器或图像预览器）可以处理选中项，并以文本或图形响应返回它们。如果存在任何脚本输出，则 suppress 设置会被忽略；这对于捕获异常并显示 traceback 很有用，如本页末尾的示例所示。

其中 exec 属性指定将由 Engine 执行的参数向量。如果第一个参数以 / 开头，则视为绝对路径；否则视为相对于站点配置目录的路径。在此示例中，Engine 会运行站点配置目录的 menu-scripts 子目录中的 askForHelp 脚本。

其中 values 属性表示选中项的哪些属性会发送到已执行脚本的标准输入 stdin。在此示例中，选中作业的 jid 和 owner 会发送到 askForHelp 的 stdin；脚本可使用这些值创建一条消息，说明哪些用户正在请求帮助，以及他们需要帮助的具体作业。所有属性参考可在 这里找到。对于任务菜单项，也可以列出 jid、priority 和 owner 等作业属性。

其中 enabled 属性表示是否显示该菜单项。这对于临时隐藏菜单项，或在配置文件中保留示例以备将来使用很有用。默认情况下，所有菜单项都是启用的，因此只有在需要隐藏时才需要该设置。

JSON 载荷

发送到脚本标准输入 stdin 的 JSON 载荷将是字典列表，每个字典表示一个选中项。在上面的示例中，values 设置为 ["jid", "owner"]，对三个选中作业发送给 askForHelp 的 JSON 载荷大致如下：

[
  {"jid": 1056, "owner": "harry"},
  {"jid": 1058, "owner": "sally"},
  {"jid": 1042, "owner": "sally"}
]

脚本

下面是 askForHelp 的一个示例实现：

#!/usr/bin/python

import sys, json, smtplib
from email.mime.text import MIMEText

def askForHelp(jobs):
    # email wranglers for help with selected jobs

    # the list of job ids is used informatively in the title and body
    jidsStr = " ".join([str(job["jid"]) for job in jobs])

    # Create a text/plain message
    text = "Please inspect the following jobs:\n%s" % jidsStr
    msg = MIMEText(text)

    msg['Subject'] = "Help Request for %s" % jidsStr

    # site would customize senders and recipients
    # sender could be set to login username of dashboard through env var
    sender = "tractor@example.com"
    # establish list of distinct job owners
    owners = set([job["owner"] + "@example.com" for job in jobs])
    recip = ["wranglers@example.com"] + list(owners)

    msg['From'] = sender
    msg['To'] = ",".join(recip)

    s = smtplib.SMTP('localhost')
    s.sendmail(sender, recip, msg.as_string())
    s.quit()

if __name__=="__main__":
    # determine selected jobs
    jsonData = sys.stdin.read()
    jobs = json.loads(jsonData)
    askForHelp(jobs)

    # in this case, no response is expected because the script was
    # configured with suppress set to true.
    # but if one was expected, we could emit "Message sent."
    # as follows:
    #
    # sys.stdout.write("Content-type:text/html\r\n\r\n")
    # sys.stdout.write("<html><body>")
    # sys.stdout.write("Message sent.")
    # sys.stdout.write("</body></html>")

示例：图像输出

下面的示例演示如何返回图像，并在浏览器的新窗口中显示。

配置条目

{
  ...
  "task":
  [
    ...
    {
      "comment": "display an image for the selected task",
      "title": "show image",
      "exec": ["menu-scripts/showImage"],
      "values": ["jid", "tid"]
    }
    ...
  ]
  ...
}

suppress 默认值为 False，因此如果需要，可以在菜单定义中省略它。

JSON 载荷

单个选中任务的 JSON 载荷大致如下：

[
  {"jid": 1098, "tid": 5}
]

脚本

通过让脚本在响应头中适当地设置 Content-type，浏览器随后就能知道如何处理并显示返回的图像数据。

要让此示例正常工作，需要在 Engine 的 /tmp 目录中有一个名为 eye.png 的图像文件，并且 Engine 进程所有者可读取它。

#!/usr/bin/python

import sys, json

if __name__=="__main__":
    # determine selected tasks
    jsonData = sys.stdin.read()
    if jsonData:
        tasks = json.loads(jsonData)

    # something could be done to determine which image to return
    # for now just return a test image in /tmp of the engine host
    filename = "/tmp/eye.png"

    # return the image; it will be displayed in a new browser window
    with open(filename, "rb") as content_file:
        content = content_file.read()
    sys.stdout.write("Content-type: image/png\n\n")
    sys.stdout.write(content)

故障排查

设置或编辑自定义菜单项时，需要重新加载 Dashboard 才能反映更改。

缺失菜单项

如果看不到自定义菜单项，请确认自己属于该菜单项的 crew（如果列出了任何 crew）。自定义菜单项只会显示给被授权调用它们的用户。

另一个导致自定义菜单项缺失的原因是配置文件中存在 JSON 语法错误。可以通过检查浏览器控制台中的相关错误消息，或通过 JSON 校验器（例如 http://jsonlint.com）运行 menus.config 文件来检测。

从命令行测试

如果菜单项出现但未按预期运行，请尝试以下步骤。

检查脚本是否可由 Engine 所有者读取和执行。

检查 menu-scripts 目录是否可由 Engine 所有者读取和执行。

尝试以 Engine 所有者身份从命令行运行脚本；如果有助于覆盖相关代码路径，可以通过管道传入一个测试 JSON 字符串。下面的示例模拟选择了名为 hostA 和 hostB 的两个 Blade，且菜单项的 values 列表表示应发送每个 Blade 的 name 时所运行的命令。

% echo '[{"name": "hostA"}, {"name": "hostB"}]' | ./test脚本

在 Windows 中编辑

如果脚本在 Windows 上编辑，或在编辑器的 Windows 模式中编辑，解释器名称后可能留下尾随 ^M 字符，导致脚本失败。^M 在某些编辑器或编辑模式中不可见。Engine 日志和输出窗口都不会显示此错误，因此很难检测。

从命令行测试脚本会暴露此错误。Python 脚本中的此类错误会显示为：

/usr/bin/python^M: bad interpreter: No such file or directory

捕获异常

调试 Python 脚本的一种方法是捕获异常，并将它们作为响应发送给浏览器。这很有用，因为 stderr 不会发送给浏览器。

下面的 Python 示例展示如何捕获任何异常，并向用户或脚本编写者提供有用反馈。

#!/usr/bin/python

import sys
import json

def test脚本(bladeinfo):
    # intentional error: bar is not defined
    foo = bar

if __name__ == "__main__":
    try:
        jsonData = sys.stdin.read()
        bladeinfo = json.loads(jsonData)
        test脚本(bladeinfo)
    except:
        import traceback
        sys.stdout.write("Content-type:text/html\r\n\r\n")
        sys.stdout.write("<html><body><pre>")
        sys.stdout.write("menu script error:\n")
        sys.stdout.write(traceback.format_exc())
        sys.stdout.write("<pre></body></html>")

一个边界情况是，目前无法捕获语法错误，因为脚本会在进入 main block 之前失败。从命令行测试脚本会暴露这类语法错误。
