"NIMBY" -- "Not In My Back Yard" -- 控制用户管理的桌面 Blade 访问。

Tractor 允许把用户桌面机器作为全站渲染农场的一部分使用，方法很简单：像更典型的服务器主机一样，在它们上面运行 tractor-blade 服务器程序。Blade 进程可以作为系统服务自动启动，也可以由用户自己手动启动；随后 Blade 会自动向 tractor-engine 注册。

不过，并不总是希望允许站点作业队列中的所有作业在每台机器上运行，尤其是桌面机器。Tractor 提供了多个层级的控制，用来决定哪些作业允许在给定 Blade 上运行。

在某些情况下，主机并不适合运行某些 类型 的作业。有些作业会要求目标 Blade 上安装特定软件，或要求一定的最低已安装 RAM 或磁盘容量。这类情况下，会使用 "service key" 作业脚本系统，把作业需求与服务器能力匹配起来。关于这种限制的更多细节，请参阅作业脚本中对 "RemoteCmd -service" 键的讨论，以及 Blade 配置档中对 "Provides" 列表的讨论。

blade.config 中的 Blade 配置档定义也可以定义哪些 作业所有者 可以访问给定的一组 Blade。Blade 配置档中的 "Access" 参数会为该配置档的 所有 成员 Blade 定义这些用户限制。对于典型渲染农场机器，这通常是合适的控制层级。

相比之下，给定 Blade 的 NIMBY 设置通常用于进一步限制会在该特定主机上运行的作业集合。另外，与 blade.config 配置档设置不同，NIMBY 设置可以由用户或调度管理员通过主 Dashboard 或桌面 nimby 应用程序动态更改。桌面 Blade 最常见的 NIMBY 限制形式，是限制它们只接受从同一主机 提交出来的作业，也就是坐在该桌面机前的用户提交的作业。



运行 NIMBY 应用程序

  • macOS

    该 nimby 应用程序位于 /Applications/Pixar/tractor-blade-$version 目录中。启动该应用程序的方式是运行 /Applications/Pixar/tractor-blade-$version/nimby``。

  • Linux

    该 nimby 应用程序位于 /opt/tractor-blade-$version 目录中。启动该应用程序的方式是运行 /opt/pixar/tractor-blade-$version/nimby``。

  • Windows

    该 nimby.bat 批处理文件位于 C:Program FilesPixartractor-blade-$version 目录中。启动该应用程序的方式是运行： C:\Program Files\Pixar\tractor-blade-$version\nimby.bat

运行模式

NIMBY UI 支持 4 种运行模式：

  • Always allow remote commands（始终允许远程命令）

    选中后，本地 Blade 会接受来自任意作业的远程或本地命令，只要该作业匹配正常的服务键和基于 crew 的要求。

  • Private use for:（专供以下用户使用：） username

    选中后，本地 Blade 会接受指定用户的远程或本地命令。

  • Local command use only - no remote jobs（仅本地命令使用 - 不接受远程作业）

    选中后，本地 Blade 只接受本地命令。即使远程命令来自本地用户，也会被拒绝。

  • Remote Use during screen saver only（仅屏幕保护期间远程使用）

    与上面的 local command use only 类似，但此模式会在屏幕保护程序运行时切换为接受所有远程命令。这样当机器被认为处于空闲状态时，Blade 就可以用于工作。

    在 Linux 上，最后一个复选框会显示为 Remote use after 5 minutes idle（空闲 5 分钟后允许远程使用）。在 Linux 上很难判断屏幕保护程序是否正在运行，因此改为查询 X11 服务器的空闲时间。默认情况下，如果鼠标已空闲 5 分钟，UI 会把 NIMBY 模式切换为接受所有远程或本地命令。
     

弹出活动作业

如果本地 Blade 上正在运行一个或多个活动命令，它们会在 UI 的 Active Jobs（当前活动作业列表） 框中详细列出。如果上面的运行模式更改为不允许访问的模式，当前作业会被允许完成，但 Blade 不会再拾取新任务。

点击 Eject Active Jobs（弹出活动作业） 按钮会让 Blade 向 Engine 发送消息，终止当前运行任务，并将其重新放回队列。
 

NIMBY 配置档设置

默认情况下，所有 Tractor Blade 都会接受来自任意作业所有者的工作，前提是服务键匹配且其他分派条件满足。配置档定义中可选的 "NIMBY" 条目可用于限制能够到达使用该配置档的 Blade 的作业：

"NIMBY":1
如果向 Blade 的配置档添加 "NIMBY": 1,"，它将只接受 本地 作业，也就是从该 Blade 主机提交的作业。把它设置为 0 会恢复上文描述的默认行为。
"NIMBY":"username"
"NIMBY" 设置也可以接受一个用户名字符串。这样做时，匹配该配置档的所有 Blade 将只接受来自指定用户的作业。

注意：此参数位于配置档定义的 "top" 层级， 不是 放在 "Access" 字典中。还要注意，可通过 Tractor Dashboard 中的 Blade 列表上下文菜单修改单个 Blade 的这个值。
 

设置 NIMBY

除了使用 Blade 配置档外，还有另外两种方式可以设置 NIMBY 状态：

  • 可以在启动时把 --nimby=foo 添加到 tractor-blade 命令行选项。此方法始终会覆盖 Blade 配置档中的任何设置，即使重新加载配置档之后也是如此。

  • 可以向 Engine 发送新的 NIMBY 更改请求 via URL（通过 URL 地址方式），就像 Tractor Dashboard 通过 Blade 上下文菜单所做的那样。Engine 会联系 Blade 来实现该更改。URL 值可以是：

    • 1 - 将只接受从该 Blade 主机提交的作业。
    • username - 将只接受来自指定用户的作业。
    • 0 - 将接受来自任何人的作业。
    • default - 将 Blade 恢复为配置档设置，或恢复为启动时通过命令行选项设置的状态。

    通过 URL 发送的值是 sticky（粘滞）的，会一直保留，直到被另一个 URL 更改，或直到 Blade 完全重启。需要特别注意，URL 设置 会 覆盖命令行选项 以及 配置档设置。
     

处理身份验证

从 Tractor 1.5 版本开始，脚本现在可以通过 Engine 进行身份验证。NIMBY 应用程序支持三种不同方式来处理身份验证所需的密码。

  • 直接提示

    如果 Tractor 系统启用了身份验证，并且没有使用命令行选项提供必要的登录密码，nimby 应用程序会在终端中提示输入一次密码。
     

  • 明文密码

    如果密码安全性不重要，可以使用 --pass 选项在命令行上指定密码。注意，这并不安全，进程搜索或历史记录搜索会暴露此密码。
     

  • 配置文件中的密码

    发送密码的更安全方法，是使用 --configfile 命令行选项。此参数指定的文件是 JSON 文件，至少应在 tractorconfig 字典中包含 user 和 passwd 信息。

    此文件应由系统级文件权限妥善保护。

    文件格式应如下所示：

    {
  "tractorconfig": {
      "monitor": "tractor-engine",
      "port": 8080,
      "user": "buildrobot",
      "passwd": "fixem"
  }
}

注意，配置文件可能包含其他信息，供同样需要身份验证的其他 Tractor 脚本读取。NIMBY 应用程序会读取并使用上文所示 Tractor config 字典中定义的任何信息。
