调度模式

概述

调度（Scheduling） 指 Tractor 用来决定任务分配给 Blade 的顺序的策略和算法；更一般地说，也包括执行管理优先级、权限和能力需求的规则。|
 

调度层级

队列中的每个作业都关联一个调度层级。调度层级提供最高层级的作业调度控制；高取值层级中的作业总是先于低取值层级中的作业被考虑。例如，rush 层级中的作业可以先于 batch 层级处理。在每个层级内，作业会按该层级的调度模式（见下文）排序。

提交新作业时，除非在提交时明确设置（例如 tractor-spool --tier=batch ...），否则会放入 default 层级。作业可以在 Dashboard 中手动移动到其他层级。例如，在危急情况下，把几个重要作业移动到更高层级，同时保持其作业优先级不变，以保留它们彼此之间的相对排序，可能会很有用。每个层级也可以在 Dashboard 中单独 暂停（paused），因此可以临时挂起或启用整个作业组的调度，例如在允许一系列管理作业继续运行时暂停常规调度。Dashboard 还提供一些基于层级过滤和排序作业的控件。

层级成员关系会作为简单的层级名称属性存储在每个作业上。 层级定义 本身位于 tractor.config 中，和 其他策略设置放在一起。

层级名称以及定义的层级数量都是任意的，可以按需要更改。即使没有显式定义，Tractor 也会确保名为 default 的层级始终存在。其他层级名称和定义不是必需的。如果某个作业引用不存在的层级名称，它会按 default 层级设置处理。

注意： 每个层级都可以有不同的调度策略， 如下文所述。如果某个层级没有指定策略，则使用 JobSchedulingMode 回退值。

"JobSchedulingMode": "P+ATCL+RR",

 "DispatchTiers": {
    "admin":   {"priority": 100.0, "scheduling": "P+FIFO"},
    "rush":    {"priority":  75.0},
    "preview": {"priority":  60.0, "scheduling": "P+CHKPT"},
    "default": {"priority":  50.0},
    "batch":   {"priority":  25.0}
 },

调度模式

Tractor 支持几种作业队列排序模式，它们是主任务分配调度系统的核心。大多数工作室会发现 P+FIFO 很适合以批处理风格为主的作业，而 P+ATCL+RR 适合以交互式作业为主的混合场景。 P+ATCL 方案往往是合理的折中方案。在所有这些模式中， 简单作业优先级 是决定作业处理顺序的主要因素。也请务必查看上方 调度层级 一节，用于对整个作业类别进行优先级排序。

在每个层级内，作业会按该层级的调度模式排序。以 P+ 开头的模式总是先按数值型 Priority 值对作业排序，从高到低。具有相同优先级的作业会按所选策略模式产生的顺序考虑：

  • P+FIFO -- 这是将可用 Blade 分配给作业的默认方案。它始终选择优先级值最高（数值最大）的作业；如果有多个作业具有相同优先级，则选择最早提交的那个。该作业中所有可运行任务会在匹配的 Blade 可用时被分配。如果可用 Blade 多于可运行任务，或某个可用 Blade 不匹配队首作业的需求，则检查已排序队列中的下一个作业。

  • P+RR -- 在主要优先级排序之后，使用简单的轮询方案在相同优先级的作业之间选择。在一个层级内，具有相同优先级的作业会在 Blade 可用时以循环方式访问。多个带有许多就绪任务的作业可以并发运行。这个模式倾向于偏好长时间运行任务的作业，因为当它们的下一个轮次再次到来时，这些作业可能仍有正在运行的任务；因此如果作业池较小，它们往往会收集到更多 Blade。因此，这个模式在生产环境中很少使用，建议改用 P+ATCL+RR。

  • P+ATCL -- Active Task Count Leveling（活动任务数均衡），该模式倾向于把可用 Blade 分配给活动任务最少的作业，试图均衡每个作业中的活动任务数量。当队列中就绪任务多于可用 Blade 插槽时，会允许较旧的作业先完成，然后再启动较新的作业。由于它保持大致相等的活动任务数，拥有短时间运行任务的作业会比拥有相同数量长时间运行任务的作业更早完成。总体上倾向于偏好最旧的作业。

  • P+ATCL+RR -- 该模式类似于 P+ATCL，区别在于它如何处理等待任务多于 Blade 的情况。它添加轮询组成部分（在相同优先级作业组内），倾向于 随着时间 在 所有 优先级区块中的作业之间平衡活动任务数。总体上偏好等待 Blade 最久的作业。

    在两种 ATCL 方案中，随着更多作业被提交到系统，渲染农场上使用许多 Blade 的旧作业会获得越来越少的 Blade。这通常是期望行为：工作室选择它，是为了让新提交的作业能够很快获得一些渲染反馈，而不是必须等待所有先前作业先完成。缺点当然是，分配给旧作业的 Blade 总数会随着新作业被提交而减少。

    举例来说，假设有 100 个相同优先级的等待作业，而渲染农场中只有 25 个 Blade。 P+ATCL 方案最终会收敛为给最旧的 25 个作业各分配 1 个 Blade，剩余 75 个较新的作业会保持空闲，直到 25 个 Blade 的窗口随着当前活动集合中最旧作业完成而推进到它们。P+ATCL+RR 模式则会让完成任务的 Blade 更广泛地分配给 所有 这些等待作业。因此，全部 100 个作业都会周期性获得一些 Blade。两种方案都偏好分配给已运行任务最少的作业。当 该 条件相同时，P+ATCL 偏好最旧的作业，而 P+ATCL+RR 偏好等待新 Blade 分配最久的作业。

  • P+CHKPT -- 一种特殊策略，用于启用参与 checkpoint - yield - resume 处理的任务在该层级中所有作业范围内重新执行。它总体上采用 FIFO 顺序，但每个作业在所有活动任务都到达 checkpoint 后会被挂起。当相同优先级组中的所有作业都到达该点后，会再次从队首作业恢复处理。
