Crews: User Name Enumeration（用户名枚举）

Tractor crews.config 文件定义用户名称列表，称为 crews，这些列表在整个系统中用于多种目的。Crew names 可用于限制某些 farm machines 上的 job access，限制 job edit permissions，或限制对 Tractor 本身的 administrative control。通过 Tractor Dashboard 或其他 clients 登录也会受到 crew definitions 的影响。

Crews 只是名称列表，不过它们可以是 recursive 的，也就是说，一个 crew definition 可以把另一个 crew name 作为成员包含进来，从而把该 crew 的所有 users 添加到新的 crew 中。

名为 ValidLogins 的 crew 尤其重要。只有该列表中命名的 users 才会获得 login access。这个 crew 必须在 crews.config 中定义，否则 engine 会声明 configuration error。ValidLogins definition 必须以某种方式列举所有有效 Tractor users。在小型工作室，这可以是简单的 user names 列表。在较大型站点，通常使用 meta-names 来委托给现有外部 username databases。

Special Meta-Names（特殊元名称）： "@syslogins" 和 "@externlogins"

特殊用户名 "@syslogins" 表示 engine host 上可见的所有有效 login names（即 tractor-engine 可通过 getpwnam(3) system call 解析的名称）。这些 login names 通常存储在 LDAP、NIS 或 /etc/passwd 等数据库中。典型 crews.config 用法如下：

"ValidLogins": ["@syslogins"],

另一种方式是，特殊名称 @externlogins 会使 engine 跳过其内部 valid user names 检查，转而委托给站点自己的外部 PAM module 或 trSiteLoginValidator.py 脚本；给定 login name 和 password 是否被接受，将由它们使用外部数据库执行的 validation checks 决定。因此，使用 @externlogins 要求启用 password validation。
 

可选 Password Validation（密码验证）

基于 web 的 Tractor Dashboard 会显示默认 login screen，其中只要求输入 login name，password field 被禁用。该名称用于获取 UI preferences（例如 filtering controls），并记录该 Dashboard session 中采取的 actions。在许多工作室，这种简单的 self-attested login identity 已经足够，尤其是在用户之间已有较高信任程度、许多用户已经拥有 administrative permissions、并且 inbound web access 受到严格控制的环境中。

类似地，对于 Tractor 的 scripted access，也可能不需要 password protection；process owner 的名称足以用于 logging 和 access checks。

Tractor 为管理员提供两种添加 password validation 的方法：engine 上的 PAM backend，以及用于添加站点定义 challenge/response hashing scheme 的 hooks。

请谨慎处理 password exposure 问题 因为用于 Tractor access 的 passwords 往往与访问其他重要系统的密码相同。实际情况可能是：完全不运行任何 password validation，同时配合站点的良好行为策略（以及对不良行为的惩戒），比薄弱的 password protection 更现实；后者会给用户错误的安全感，并可能把重要 system accounts 暴露给恶意人员。

crews.config 中的 SitePasswordValidator 设置定义从 user interfaces 和 scripts 发送到 engine 的 passwords 所采用的 validation scheme。 使用空字符串 "" 可禁用 password checking， 允许任何 valid login name 作为 client 继续。任何非空值都会要求提供 passwords，并指定要应用的 validation 类型。

 

Pluggable Authentication Module（PAM，可插拔认证模块）

Tractor 提供一种内置方式，可把 authentication decisions 委托给站点可配置的 PAM modules，例如那些已经用于验证操作系统桌面用户登录的模块。将 crews.config 中的 SitePasswordValidator 设置改为你想使用的 PAM policy module 名称：

"SitePasswordValidator": "internal:PAM:tractor",

PAM modules 通常定义在 /etc/pam.d 或 /etc/pam.conf 中，你必须根据站点的适当 policies 提供一个 tractor module definition。也可以指定不同的 PAM module name，例如已经支持站点需求的现有模块。比如，要获得 su authentication rules，请使用 "internal:PAM:su" 作为替代。

注意：在 Linux RHEL 6.x 时代的 releases 上， pam_fprintd.so module 包含一个 bug，会导致它在 tractor-engine 每次调用时泄漏 file descriptors。由于 PAM modules 被加载到 tractor-engine process 中，并且 engine 会随时间执行许多 authentications，未关闭的 pipe descriptors 会在主 tractor-engine code 不知情的情况下累积，并最终耗尽该 engine process 可用的 file descriptor limit。虽然许多工作室并不依赖 fingerprint validation，尤其是针对 system service 的 scripted API access，但 fprint module 会被许多常见 RHEL6 PAM policies 间接调用，包括 login 和 su。在 RHEL 7 时代的 distributions 中，它已从 common policies 中移除。RHEL6 的 workaround 是创建自己的 tractor policy，不包含 system-auth；或者在 crews.config 中指定较不通用的 policy，例如 password-auth。

PAM scheme 允许 Tractor 管理员快速启用 password checking，但它只适合 LAN 或 VPN 已经为从 client 到 engine 的 credential transfer 提供足够安全性的站点。在 PAM scheme 中，credentials 由 client app 从用户处收集，但不会在 client host 上验证（例如运行在 web browser 中的平板 Tractor Dashboard）。编码后的 credentials 会发送到 engine，再通过 PAM API 执行验证。编码 credentials 从 client 传输到 engine 的过程容易受到某些攻击，因此前一节中提出了相关注意事项。
 

Tractor Utility Scripts 的认证

Tractor engine（自 1.5 起）支持 challenge token URL，使需要 authentication 的站点可以直接支持外部 tractor scripts。

对于 python scripts，distribution 中的 tractor.base.TrHttpRPC module 包含 login support。使用 TrHttpRPC module 进行 engine connections 很方便，但不是必需的；下文会为使用其他 scripting languages 或希望编写自己的 authenticated scripts 的站点提供细节。

TrHttpRPC module 提供两个与 authentication 相关的方法。PasswordRequired() 方法返回一个 boolean，指示登录 monitor 是否需要 password。Login(user, passwd) 方法接受两个 arguments，即用于 login 的 username 和 password。

与 dashboard UI 类似，Login() 方法会下载一个名为 trSiteFunctions.py 的文件，其中包含 trSitePasswordHash() function。该文件位于 engine configuration directory。trSitePasswordHash() function 定义在发送给 monitor 并由 login validator 处理之前，用于 hash user password 的 function。返回 python None 表示不需要 password checking。

准备支持 password authentication 的 Python scripts 应先调用 PasswordRequired() function，以判断是否需要 password。脚本没有用于请求 password 的内置 function，不过 tractor utility scripts 提供了一些处理方式示例。请参阅 nimby.py 和 jobMonitor.py 的示例。
 

在外部 Scripts 中实现 Authentication

除了 hash 用户 password 之外，登录 tractor monitor 的 external scripts 还需要为 monitor login URL 正确格式化 challenge/authentication string。需要支持 authentication 的 scripts 应按照以下细节准备 challenge/authentication URL。

  • 脚本应首先连接到 tractor monitor 并请求 challenge token。做法是向 monitor 发送 /Tractor/monitor?q=gentoken URL。

  • 该 URL 会返回如下格式的 JSON dictionary：

    JSON 示例：{ "challenge": "23767519241324501130125633560" }

  • 脚本必须要么 know 这个 password hashing function，要么从 monitor 请求它。一种方法是把 hash function 保存在 engine configuration directory 中的文件里，并使用 /Tractor/config?file=configurationfile URL 请求该文件。例如，随附的示例 utility scripts 会使用 URL /Tractor/config?file=trSiteFunctions.py 请求一个 python 文件。该 URL 会把指定文件的内容作为文本交付给脚本。根据所用 scripting language 的不同，这些数据可能需要 eval 或 source 到 scripting interpreter 中。

  • 定义 hashing function 后，应使用 test password 调用它，以判断是否确实需要 password。如果不需要 password，hashing function 应返回 NULL 或 None 类型的 response。

  • 如果需要 authentication，下一步就是为 login URL 格式化 challenge token 和 password hash。

  • 首先需要用竖线 | 连接 challenge token 与用户 password 经过 hashing function 处理后的结果。大致形式为 challengestring|hashedpassword。

  • 然后需要把该字符串转换为 hex representation，其中字符串中的每个 byte 都要转换为 02xformat representation。例如，上面的字符串 示例字符串：challengestring|hashedpassword 会编码为 6368616c6c656e6765737472696e677c68617368656470617373776f7264。

  • 最后，这个 hex coded passwordhash string 需要进行 URL encoded，并通过 login URL 发送给 monitor。该 URL format 为：

    /Tractor/monitor?q=login&user=user&c=urlencodedchallengestring

  • 如果 login 正确通过验证，monitor 会返回如下所示的 JSON dictionary：

    {
  "rc": 0, "login": "ok",
  "host": "192.168.1.1",
  "user": "rdavis",
  "tsid": "4f04f1c9-4ef2488a-0021",
  "crews": ["ValidLogins", "Wranglers"]
}

  • dictionary 中的 tsid entry 应保留下来，并添加到发送给 monitor 的所有 URL 中。以上面的示例为例，每个 URL 现在都应以 ....&tsid=4f04f1c9-4ef2488a-0021结尾。
