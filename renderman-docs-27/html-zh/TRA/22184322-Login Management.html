Crew（用户组）：用户名枚举

Tractor 的 crews.config 文件定义了称为 crew（用户组）的用户名称列表，这些列表在整个系统中用于多种用途。Crew 名称可用于限制某些渲染农场机器上的作业访问、限制作业编辑权限，或限制对 Tractor 本身的管理控制。通过 Tractor Dashboard 或其他客户端登录也会受到 crew 定义的影响。

Crew 本质上只是名称列表，但它们可以递归定义：一个 crew 定义可以把另一个 crew 名称作为成员包含进来，从而把该 crew 的所有用户加入新的 crew。

名为 ValidLogins 的 crew 尤其重要。只有该列表中命名的用户才会获得登录访问权限。这个 crew 必须在 crews.config 中定义，否则 Engine 会报告配置错误。ValidLogins 定义必须以某种方式列举所有有效的 Tractor 用户。在小型工作室，这可以只是一个用户名列表；在较大型站点，通常使用元名称来委托给现有的外部用户名数据库。

特殊元名称： "@syslogins" 和 "@externlogins"

特殊用户名 "@syslogins" 表示 Engine 主机上可见的所有有效登录名（即 tractor-engine 可通过 getpwnam(3) 系统调用解析的名称）。这些登录名通常存储在 LDAP、NIS 或 /etc/passwd 等数据库中。典型 crews.config 用法如下：

"ValidLogins": ["@syslogins"],

另一种方式是，特殊名称 @externlogins 会使 Engine 跳过内部有效用户名检查，改为委托给站点自己的外部 PAM 模块或 trSiteLoginValidator.py 脚本；给定登录名和密码是否被接受，将由它们通过外部数据库执行的验证检查决定。因此，使用 @externlogins 需要启用密码验证。

可选 Password Validation（密码验证）

基于 Web 的 Tractor Dashboard 会显示默认登录界面，其中只要求输入登录名，密码字段处于禁用状态。该名称用于获取过滤控件等 UI 偏好设置，并记录该 Dashboard 会话中执行的操作。在许多工作室，这种简单的自声明登录身份已经足够，尤其是在用户之间已有较高信任程度、许多用户已经拥有管理权限、并且入站 Web 访问受到严格控制的环境中。

类似地，对于脚本访问 Tractor，也可能不需要密码保护；进程所有者的名称足以用于日志记录和访问检查。

Tractor 为管理员提供两种添加密码验证的方法：Engine 上的 PAM 后端，以及用于添加站点自定义质询/响应哈希方案的钩子。

请谨慎处理密码暴露问题，因为用于访问 Tractor 的密码往往与访问其他重要系统的密码相同。实际情况可能是：完全不启用任何密码验证，同时配合站点的良好行为策略（以及对不良行为的惩戒），比薄弱的密码保护更现实；后者会给用户错误的安全感，并可能把重要系统账户暴露给恶意人员。

crews.config 中的 SitePasswordValidator 设置定义从用户界面和脚本发送到 Engine 的密码所采用的验证方案。 使用空字符串 "" 可禁用密码检查， 允许任何有效登录名作为客户端继续访问。任何非空值都会要求提供密码，并指定要应用的验证类型。

 

可插拔认证模块（PAM）

Tractor 提供一种内置方式，可把身份验证决策委托给站点可配置的 PAM 模块，例如那些已经用于验证操作系统桌面用户登录的模块。将 crews.config 中的 SitePasswordValidator 设置改为要使用的 PAM 策略模块名称：

"SitePasswordValidator": "internal:PAM:tractor",

PAM 模块通常定义在 /etc/pam.d 或 /etc/pam.conf 中，你必须根据站点策略提供一个合适的 tractor 模块定义。也可以指定不同的 PAM 模块名称，例如已经满足站点需求的现有模块。比如，要使用 su 身份验证规则，请改用 "internal:PAM:su"。

注意：在 Linux RHEL 6.x 时代的版本上， pam_fprintd.so 模块包含一个缺陷，会导致它在 tractor-engine 每次调用时泄漏文件描述符。由于 PAM 模块被加载到 tractor-engine 进程中，并且 Engine 会随时间执行许多次身份验证，未关闭的管道描述符会在主 tractor-engine 代码不知情的情况下累积，并最终耗尽该 Engine 进程可用的文件描述符限制。虽然许多工作室并不依赖指纹验证，尤其是针对系统服务的脚本化 API 访问，但 fprint 模块会被许多常见 RHEL6 PAM 策略间接调用，包括 login 和 su。在 RHEL 7 时代的发行版中，它已从通用策略中移除。RHEL6 的变通方法是创建自己的 tractor 策略，不包含 system-auth；或者在 crews.config 中指定较不通用的策略，例如 password-auth。

PAM 方案允许 Tractor 管理员快速启用密码检查，但它只适合 LAN 或 VPN 已经为从客户端到 Engine 的凭据传输提供足够安全性的站点。在 PAM 方案中，凭据由客户端应用从用户处收集，但不会在客户端主机上验证（例如运行在 Web 浏览器中的平板 Tractor Dashboard）。编码后的凭据会发送到 Engine，再通过 PAM API 执行验证。编码凭据从客户端传输到 Engine 的过程容易受到某些攻击，因此前一节中提出了相关注意事项。

Tractor 实用脚本的身份验证

Tractor Engine（自 1.5 起）支持质询令牌 URL，使需要身份验证的站点可以直接支持外部 Tractor 脚本。

对于 Python 脚本，发行包中的 tractor.base.TrHttpRPC 模块包含登录支持。使用 TrHttpRPC 模块连接 Engine 很方便，但不是必需的；下文会为使用其他脚本语言或希望编写自己的认证脚本的站点提供细节。

TrHttpRPC 模块提供两个与身份验证相关的方法。PasswordRequired() 方法返回一个布尔值，指示登录监控服务是否需要密码。Login(user, passwd) 方法接受两个参数，即用于登录的用户名和密码。

与 Dashboard UI 类似，Login() 方法会下载名为 trSiteFunctions.py 的文件，其中包含 trSitePasswordHash() 函数。该文件位于 Engine 配置目录中。trSitePasswordHash() 函数定义了在发送给监控服务并交由登录验证器处理之前，用来对用户密码做哈希处理的函数。返回 Python None 表示不需要密码检查。

准备支持密码身份验证的 Python 脚本应先调用 PasswordRequired() 函数，以判断是否需要密码。脚本没有用于请求密码的内置函数，不过 Tractor 实用脚本提供了一些处理方式示例。请参阅 nimby.py 和 jobMonitor.py 的示例。

在外部脚本中实现身份验证

除了对用户密码做哈希处理之外，登录 Tractor 监控服务的外部脚本还需要为监控服务登录 URL 正确格式化质询/认证字符串。需要支持身份验证的脚本应按照以下细节准备质询/认证 URL。

  • 脚本应首先连接到 Tractor 监控服务并请求质询令牌。做法是向监控服务发送 /Tractor/monitor?q=gentoken URL。

  • 该 URL 会返回如下格式的 JSON 字典：

    JSON 示例：{ "challenge": "23767519241324501130125633560" }

  • 脚本必须要么预先知道密码哈希函数，要么从监控服务请求它。一种方法是把哈希函数保存在 Engine 配置目录中的文件里，并使用 /Tractor/config?file=configurationfile URL 请求该文件。例如，随附的示例实用脚本会使用 URL /Tractor/config?file=trSiteFunctions.py 请求一个 Python 文件。该 URL 会把指定文件的内容作为文本交付给脚本。根据所用脚本语言的不同，这些数据可能需要 eval 或 source 到脚本解释器中。

  • 定义哈希函数后，应使用测试密码调用它，以判断是否确实需要密码。如果不需要密码，哈希函数应返回 NULL 或 None 类型的响应。

  • 如果需要身份验证，下一步就是为登录 URL 格式化质询令牌和密码哈希。

  • 首先需要用竖线 | 连接质询令牌与用户密码经过哈希函数处理后的结果。大致形式为 challengestring|hashedpassword。

  • 然后需要把该字符串转换为十六进制表示，其中字符串中的每个字节都要转换为 02x 格式。例如，上面的字符串 challengestring|hashedpassword 会编码为 6368616c6c656e6765737472696e677c68617368656470617373776f7264。

  • 最后，这个十六进制编码的 passwordhash 字符串需要进行 URL 编码，并通过登录 URL 发送给监控服务。该 URL 格式为：

    /Tractor/monitor?q=login&user=user&c=urlencodedchallengestring

  • 如果登录验证通过，监控服务会返回如下所示的 JSON 字典：

    {
  "rc": 0, "login": "ok",
  "host": "192.168.1.1",
  "user": "rdavis",
  "tsid": "4f04f1c9-4ef2488a-0021",
  "crews": ["ValidLogins", "Wranglers"]
}

  • 字典中的 tsid 条目应保留下来，并添加到发送给监控服务的所有 URL 中。以上面的示例为例，每个 URL 现在都应以 ....&tsid=4f04f1c9-4ef2488a-0021 结尾。
