JSON Job Scripting

Downloaded: 2026-06-15T03:49:12+00:00

概要

Tractor 作业文件通常表示为 专用 TCL 作业脚本，并使用 Tractor 作业解析器可识别的额外作业定义运算符。这种格式最初为 Alfred 系统引入，Tractor 仍与几乎所有 Alfred 构造兼容。

Tractor 还支持一组不断演进的 JSON 作业规范格式，本文会对此进行说明。

 

介绍

作业脚本描述要由 Tractor 分派和监控的工作。它们通常由内容创作应用程序中的插件在提交新的渲染作业时创建。因此，下面的语法讨论可能只对开发符合 Tractor 规范的应用程序的人员有用。

不过，有一种非常 生成基本 Tractor 作业的简单方法，可从命令行或 shell 脚本内完成。tractor-spool 命令行工具用于把作业脚本发送到由 tractor-engine 管理的作业队列。你也可以让 tractor-spool 直接生成一个简单作业脚本：

tractor-spool -c prman /home/yoda/new_idea.rib

-c 之后的任何文本都会被包进作业定义，并发送到渲染农场中可用的 tractor-blade 服务器。还有多种 额外命令行选项 可用于修改作业的具体细节。

作业脚本支持一种简单的层级结构方案，可编码依赖关系，而这些依赖关系会决定任务执行顺序。

 

JSON 作业示例

下面是一个简单脚本，用于定义示例任务层级。假设我们需要计算两张阴影贴图，以及一个引用它们的最终帧；在这种情况下，必须先完成阴影贴图，最终帧的渲染才能开始。

{  "TractorJob": "2.0",
   "data": {
      "title": "a very short film",
      "priority": 1.0,
      "spoolcwd": "/private/tmp",
      "spooldate": "2014-03-11 12:06:55",
      "spoolfile": "x.job",
      "spoolhost": "h2o",
      "spooltime": 1394564815,
      "tags": "",
      "metadata": "",
      "comment": "",
      "envkey": [],
      "projects": ["theFilm"],
   },
   "children": [
      {
         "data": {
            "title": "Frame One",
            "commands": [
               {
                 "argv": ["render", "beauty.1.rib"],
                 "service": "PixarRender"
                 "type": "RC",
               }
            ]
         }
         "children": [
            {
               "data": {
                  "title": "Shadow A",
                  "commands": [
                     {
                       "argv": ["render", "light.1.rib"],
                       "service": "PixarRender"
                       "type": "RC",
                     }
                  ]
               }
            },
            {
               "data": {
                  "title": "Shadow B",
                  "commands": [
                     {
                       "argv": ["render", "light.2.rib"],
                       "service": "PixarRender"
                       "type": "RC",
                     }
                  ]
               }
            }

         ]

      }
   ]
}

这个作业脚本描述了一个两级执行树，其中名为 Scene One 的任务会在它依赖的两个任务成功完成后启动自己的命令（render frame.1.rib）（即两个 Shadow任务）。嵌套任务描述定义了（深度优先的）执行顺序；因此会先启动 light.1.rib 的渲染命令，随后启动 light.2.rib 的命令。然后，当二者都完成后，再渲染 frame.1.rib。命令会按照它们在脚本中出现的顺序，作为独立子进程启动。

请注意，这两个阴影命令彼此不依赖，也就是说，它们相互之间没有嵌套关系；因此它们代表一个并行处理机会。在资源充足时，Tractor 通常会同时启动这两个命令，并允许它们并发执行。因此，脚本定义的是一种 命令启动顺序；某些命令在另一个命令启动时可能仍在运行，而这些并发命令可以以任意顺序完成。不过无论如何，某个 Task 都不会启动自己的命令，直到它的 所有 子任务都已成功完成各自的命令。

此示例中的假设如下：

  • 存在一个名为 render 可执行文件，位于当前搜索路径中的某处。
  • 两张阴影贴图图像会在磁盘上创建，并位于 frame.1.rib 期望找到它们的位置（即这些 RIB 文件中的文件系统引用是一致的）。

Tractor 不会 尝试确认这些假设；这是作业生成器的职责。如果确实发生错误，该任务会变为 blocked 状态，相应 UI widget 会转为 error 状态，所有依赖这个出错任务的任务都不会执行（不过不依赖它的其他任务仍可继续）。

关于 Task 需要注意的事项：

  • 标题字符串会显示在 Dashboard 中，因此最好选择简短但描述性强的标题。
  • 子任务列在 -subtasks 字段中；如果没有依赖关系，可以为空或省略。
  • -cmds 字段中的 RemoteCmd 列表描述依赖关系满足后要采取的实际动作；它可以为空或省略；也可以包含多个（以换行分隔的）RemoteCmds，这些命令会按顺序处理。
