实现

Engine（tractor-engine）是一个高性能、多线程的 C++ 应用。它响应 HTTP transaction requests，并维护已 spooled jobs 的数据库。

Blades 是直接的 execution server，由 Python 编写，并提供可由站点扩展的 modules 和 role configuration。

Dashboard 提供 job tracking、visualization 和 control widgets，这些功能基于现代 HTML5/CSS3/JavaScript 框架开发。每个用户的 Dashboard layout 和 preferences 都保存在 engine data area 中按用户划分的区域里。

新的 jobs 通常以 Alfred-format job scripts 的形式 spooled 到 queue 中，engine 会自动处理这些脚本，并在数据库（当前为 PostgreSQL）中存储为 job、task 和 command records。通常通过运行随附的 tractor-spool 命令行脚本把 jobs spooled 到 engine。还随附一个 Python job authoring module，提供用于创建和提交 jobs 的 scriptable API。

 

发布下载内容

Tractor-2.0 下载包包含在 farm 上部署 Tractor 所需的全部 system services、user interfaces、scripting tools 和其他资源。不过，每个下载包都是平台特定的；如果工作室要在多个操作系统上运行 Tractor 组件，则需要为每个平台下载一个 installer。

Tractor 系统由以下高层部分组成：

  • tractor-engine -- 主要的 job queue 管理器 executable
  • tv -- Tractor Dashboard 的 tractor viewer web-app 实现
  • tractor-blade -- 在每个 farm host 上启动 commands 的 remote execution server
  • tractor-spool -- 将 job description files 发送到提交 queue
  • tq -- 用于 job data 查询的 command-line interface 和脚本 scripting module
  • rmanpy -- 完整的 Python 2.7.1 distribution，带额外内置 modules
  • job database -- PostgreSQL 9.3 database server distribution，位于 Tractor install area 内。
  • nimby -- 小型 Qt-based user interface，用于帮助控制把工作站作为 auxiliary farm nodes 使用

注意！并非每台 host 都需要每个 Tractor component，但所有组件都安装在同一个带版本号的 Tractor directory 下，因为它们共享大量资源。实际 Tractor applications 本身几乎是安装包中最小的一部分。

许多工作室会选择在 shared disk 上安装一次，然后通过 network shares 在每台参与 host 上运行。其他工作室会分别在每台 host 上安装软件包，但只运行该 host 特别需要的部分。

 

Tractor-2.0 Bundle 详细信息

新的 Tractor 2.0 packaging 和 installation layout 会把匹配的一组 engine、blade、spooler、user interfaces 和 scripting components 放在同一位置，同时只保留一份共享资源，其中包括多个第三方子系统的预构建版本。Tractor 应该可以直接使用，不需要额外的 external downloads、builds 或 installs。

第三方组件包括完整的 Python 2.7.1，并带有 Qt 和 database support 等内置 extension modules；一个 Tcl interpreter；以及完整的 PostgreSQL 9.3 server，用于替换先前的 job database。这些组件作为便利提供给客户；如果工作室希望，也可以用现场从源码构建的版本替换它们。

  • A PostgreSQL database（PostgreSQL 数据库） 现在随 release 捆绑提供。它替换先前的 job database，并用于存储所有 spooled jobs，以及关于 command launch 和 disposition 的所有收集数据。该数据库支撑多个重要的 2.0 功能，最明显的是 tq query variations。它还以比以前更紧凑的 footprint 存储 job data，并且在某些情况下可能允许一种有用的 load-balancing 形式。在运维上，主 tractor-engine 进程通常会把 database server 作为 subprocess 管理，不需要管理员协助。这包括 start-up 和 shutdown、table creation，以及 versioned schema updates。不过，也有高级选项可控制 database operation 和 administration 的特定方面。只有 tractor-engine 会连接到数据库，因此默认配置只允许来自 localhost interface 的 client connections。
  • Python 更新 -- 许多 Tractor 2.0 工具（包括 engine）现在依赖可用的合规 Python interpreter。Tractor 现在采用与其他 Pixar RenderMan 产品相同的 rmanpy bundling scheme，该机制也用于其他 Pixar RenderMan 产品。这个 Tractor release 中捆绑的版本已更新到 Python 2.7.1，并包含几个重要的内置 extension modules。
    • 更新后的 Python 版本本身包含几项重要功能，例如改进的 Unicode 支持和 bug fixes。
    • 所有基于 Python 的 Tractor tools 和 applications 现在都作为 site-packages modules 交付，位于捆绑的 rmanpy Python distribution 中。共享 utility modules 也位于同一个 Python library area，因此可被基于 rmanpy 的 scripts 轻松 import。
    • rmanpy package 包含 Qt support，Tractor 在重新实现小型独立 nimby desktop UI 时会使用它（同时也移除了一些旧的 X11 和 Tk 平台需求）。
    • 其中有一个 PostgreSQL access module，由 Tractor 内部用来管理 backend data store。对数据库的外部 scripted access 应始终通过 Engine 自身暴露的 access protocols 进行中介，以保护开发人员及其 pipelines 不受内部 database schema 在不同 release 之间变化的影响。
  • Job spooling 作业提交支持 --
    • 另一个扩展包括一个捆绑的 Tcl interpreter，用于解析 Alfred-style job scripts。该扩展实际上被打包为 rmanpy python module，允许 Tractor 基于 python 的 spooler 和 handler system 直接处理 Tcl-formatted alfscript jobs，而不需要额外安装独立 Tcl packages，尤其适用于没有（或不再有）内置 tclsh support 的平台。Tcl 扩展还允许 tractor-spool 脚本在 job 离开用户的 spooling client system 之前直接对新 jobs 做 sanity check，从而在涉及 engine 或 job storage database 之前标记 syntax errors 和其他问题。
    • Tractor rmanpy Python bundle 和上文描述的 PostgresSQL bundle 在 Tractor-2.0 中组合使用，用于提升系统 backend job spooling throughput。带有对 job database 直接访问能力的 parallel persistent intake channels，可以以明显快于先前 release 的速度处理 new jobs 和 expand fragments。
