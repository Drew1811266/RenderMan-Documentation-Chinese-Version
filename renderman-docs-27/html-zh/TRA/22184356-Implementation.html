实现

Engine（tractor-engine）是一个高性能、多线程的 C++ 应用。它响应 HTTP 事务请求，并维护已提交作业的数据库。

Blade 是直接执行服务器，由 Python 编写，并提供可由站点扩展的模块和角色配置。

Dashboard 提供作业跟踪、可视化和控制部件，这些功能基于现代 HTML5/CSS3/JavaScript 框架开发。每个用户的 Dashboard 布局和首选项都保存在 Engine 数据区中按用户划分的区域里。

新的作业通常以 Alfred 格式作业脚本的形式提交到队列中，Engine 会自动处理这些脚本，并在数据库（当前为 PostgreSQL）中存储为作业、任务和命令记录。通常通过运行随附的 tractor-spool 命令行脚本把作业提交到 Engine。还随附一个 Python 作业编写模块，提供用于创建和提交作业的可脚本化 API。

 

发布下载内容

Tractor-2.0 下载包包含在农场上部署 Tractor 所需的全部系统服务、用户界面、脚本工具和其他资源。不过，每个下载包都是平台特定的；如果工作室要在多个操作系统上运行 Tractor 组件，则需要为每个平台下载一个安装程序。

Tractor 系统由以下高层部分组成：

  • tractor-engine -- 主要作业队列管理器可执行文件
  • tv -- Tractor Dashboard 的 tractor viewer Web 应用实现
  • tractor-blade -- 在每个农场主机上启动命令的远程执行服务器
  • tractor-spool -- 将作业描述文件发送到提交队列
  • tq -- 用于作业数据查询的命令行界面和脚本模块
  • rmanpy -- 完整的 Python 2.7.1 发行版，带额外内置模块
  • 作业数据库 -- PostgreSQL 9.3 数据库服务器发行版，位于 Tractor 安装区内。
  • nimby -- 小型基于 Qt 的用户界面，用于帮助控制把工作站作为辅助农场节点使用

注意！并非每台主机都需要每个 Tractor 组件，但所有组件都安装在同一个带版本号的 Tractor 目录下，因为它们共享大量资源。实际 Tractor 应用本身几乎是安装包中最小的一部分。

许多工作室会选择在共享磁盘上安装一次，然后通过网络共享在每台参与主机上运行。其他工作室会分别在每台主机上安装软件包，但只运行该主机特别需要的部分。

 

Tractor-2.0 捆绑包详细信息

新的 Tractor 2.0 打包和安装布局会把匹配的一组 Engine、Blade、spooler、用户界面和脚本组件放在同一位置，同时只保留一份共享资源，其中包括多个第三方子系统的预构建版本。Tractor 应该可以直接使用，不需要额外的外部下载、构建或安装。

第三方组件包括完整的 Python 2.7.1，并带有 Qt 和数据库支持等内置扩展模块；一个 Tcl 解释器；以及完整的 PostgreSQL 9.3 服务器，用于替换先前的作业数据库。这些组件作为便利提供给客户；如果工作室希望，也可以用现场从源码构建的版本替换它们。

  • A PostgreSQL 数据库 现在随发行版捆绑提供。它替换先前的作业数据库，并用于存储所有已提交作业，以及关于命令启动和处置的所有收集数据。该数据库支撑多个重要的 2.0 功能，最明显的是 tq 查询变体。它还以比以前更紧凑的占用空间存储作业数据，并且在某些情况下可能允许一种有用的负载均衡形式。在运维上，主 tractor-engine 进程通常会把数据库服务器作为子进程管理，不需要管理员协助。这包括启动和关闭、表创建，以及带版本的 schema 更新。不过，也有高级选项可控制数据库操作和管理的特定方面。只有 tractor-engine 会连接到数据库，因此默认配置只允许来自 localhost 接口的客户端连接。
  • Python 更新 -- 许多 Tractor 2.0 工具（包括 Engine）现在依赖可用的合规 Python 解释器。Tractor 现在采用与其他 Pixar RenderMan 产品相同的 rmanpy 捆绑机制。这个 Tractor 发行版中捆绑的版本已更新到 Python 2.7.1，并包含几个重要的内置扩展模块。
    • 更新后的 Python 版本本身包含几项重要功能，例如改进的 Unicode 支持和 bug 修复。
    • 所有基于 Python 的 Tractor 工具和应用现在都作为 site-packages 模块交付，位于捆绑的 rmanpy Python 发行版中。共享工具模块也位于同一个 Python 库区域，因此可被基于 rmanpy 的脚本轻松导入。
    • rmanpy 包含 Qt 支持，Tractor 在重新实现小型独立 nimby 桌面 UI 时会使用它（同时也移除了一些旧的 X11 和 Tk 平台需求）。
    • 其中有一个 PostgreSQL 访问模块，由 Tractor 内部用来管理后端数据存储。对数据库的外部脚本化访问应始终通过 Engine 自身暴露的访问协议进行中介，以保护开发人员及其流水线不受内部数据库 schema 在不同发行版之间变化的影响。
  • 作业提交支持 --
    • 另一个扩展包括一个捆绑的 Tcl 解释器，用于解析 Alfred 风格作业脚本。该扩展实际上被打包为 rmanpy Python 模块，允许 Tractor 基于 Python 的 spooler 和 handler 系统直接处理 Tcl 格式的 alfscript 作业，而不需要额外安装独立 Tcl 包，尤其适用于没有（或不再有）内置 tclsh 支持的平台。Tcl 扩展还允许 tractor-spool 脚本在作业离开用户的提交客户端系统之前直接对新作业做健全性检查，从而在涉及 Engine 或作业存储数据库之前标记语法错误和其他问题。
    • Tractor rmanpy Python 捆绑包和上文描述的 PostgreSQL 捆绑包在 Tractor-2.0 中组合使用，用于提升系统后端作业提交吞吐量。带有对作业数据库直接访问能力的并行持久接收通道，可以以明显快于先前发行版的速度处理新作业和 expand 片段。
