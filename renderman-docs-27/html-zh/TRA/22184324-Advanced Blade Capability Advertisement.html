服务键简介

Tractor 使用一种简单的抽象关键字匹配系统，将作业需求与 Blade 能力进行匹配，这个系统称为 服务键（service keys）。

在作业中，需求通过 RemoteCmd -service "keys" 语句指定，其中 "keys" 可以是简单关键字，也可以是更复杂的 表达式。

在 Blade profile 中，能力会列举在 Provides <tractor_profiles.html#provides-svckeys> 列表中。列表里的名称表示这些 Blade 的属性：它们运行某个特定 OS，或安装了某个软件包，或具有快速网络连接，等等。键名是任意的，只有在它们与作业中用于要求这些 Blade 属性的 -service 名称匹配时，才具有站点定义的含义。

因此，一个 Blade profile（在 blade.config 中）可能包含这个条目：

"Provides": ["PixarRender", "RfMRender", "Linux"]

而一个 作业 可能具有如下命令：

RemoteCmd {prman test.rib} -service "RfMRender" RemoteCmd {prman test.rib} -service "PixarRender && Linux"

高级能力注解

已添加几种高级 Blade profile "Provides" 模式，用来支持一些不常见的使用场景。注意，这些注解应放在 blade.config 不是 作业脚本中！

计数型服务键

(max:NN) 后缀。

有时，将 "Provides" 列表视为每台 Blade 主机上的一组计数型资源会很有用。Tractor 键可以添加注解，表示 Blade 应跟踪它们的使用情况，并且只应为仍有未使用容量的这些键通告服务可用性。

注意，这个“计数型键”功能较为微妙，通常只在一些不常见场景中有用。它既与常规 limits处理重叠，也与通用 Blade slot count 概念重叠。只有当单个 Blade 主机被允许并发运行多个命令时，计数这一面才有意义；也就是说，当 Blade 的 slot count 大于 1 时。

这个思路是给 Blade 上的每种服务键类型“分配”一个固定的最大并发数。有时，如果已知使用某个特定键的命令具有较大的资源占用“足迹”，而使用其他键的程序较轻量，就会产生这种需求。例如，工作室可能既有资源密集型的典型 prman 作业，也有轻量的简单 nuke 作业。

"Provides": ["PixarRender(max:2)", "NukeRender(max:4)", "Linux"], "Slots": 4, ...

在此示例中，使用此 profile 的 Blade 一开始会接受任何需要 PixarRender 或 NukeRender 服务的作业。它们总共有四个并发插槽。在任何给定时间，PixarRender 进程不能超过两个，NukeRender 进程不能超过四个。因此，Blade 可能运行 1 个 prman 和 3 个 nuke 进程，或 2 个 prman 和 2 个 nuke 进程，或 4 个 nuke 进程。

因此，此功能允许 Blade 根据哪些键已被先前启动的命令“消耗”，动态通告不同的服务键能力集合。

注意： 这些计数型服务键跟踪的是 Blade 能力，它们在概念上独立于全局 Limit 计数以及 项目或许可 级别的资源共享分配。这些计数型键决定给定的 Blade 是否仍有剩余容量 可用于工作，而 Limits 则在确定哪些作业（如果有）可以分配给那些可用 Blade 时应用。

条件式服务通告

(after:KEY) 后缀。

这个 Blade profile 的 "Provides" 键注解与上文的计数型服务键概念相关。在这种变体中，Blade 上的插槽会被指示为先只分配给一种作业，然后才允许其他“填充”类型用掉可能剩余的容量。一个例子是：一组多核心机器应各自保留给两个 8 线程 prman 调用，剩余容量再由 nuke 作业填充。目标是在空闲时始终留下一个较大的 prman 空位，而不是让轻量作业一点点占用可用插槽，导致永远没有足够大的空位让 prman 任务被调度。

"Provides": ["PixarRender(max:2)", "NukeRender(after:PixarRender)"], "Slots": 6, ...

在此示例中，NukeRender 服务能力要等到两个 PixarRender 进程正在运行后才会被通告。"after" 注解表示给定键只有在命名键已达到其最大计数后才可用。如果某个正在运行的 prman 进程退出，那么 PixarRender 会低于其最大值 2，直到另一个 prman 落到该 Blade 上之前，都不会分配新的 nuke 进程。这些情况下的典型假设是，"after" 键与轻量且运行快速的程序相关。

同样，此功能用于根据当前正在运行的命令，控制某个 Blade 能力是否完全被视为可用。它独立于 Limits，但可以影响某一特定类型正在运行程序的总数。
 

必需键：强制作业服务类型

(R) 后缀。

在 profile 的 "Provides" 列表名称中添加可选的 "keyname(R)" 后缀，会以一种细微方式改变通常的 Blade 分配逻辑。该键会变成 强制必需条件（mandatory）， 作业 必须 在其他键之中请求该键，才能匹配 Blade。

Blade "Provides" 列表中名称的通常处理方式是：只要匹配列表中的 ANY 键，就足以允许匹配的命令在该 Blade 上运行。"(R)" 后缀则指定命令 REQUIRED 至少请求给定键，才能使用该 Blade。

下面是使用该注解的 Provides 列表示例：

"Provides": ["PixarRender", "RfMRender", "DebugEnv(R)"]

对于使用该 profile 的 Blade，只使用 "-service PixarRender" 的 RemoteCmd 将 不再 在这些 Blade 上运行，但 "-service PixarRender,DebugEnv" 会 在那里运行。区别在于，这些 Blade 不会接受只请求 "PixarRender" 的其他常规作业命令。Provides 列表中的 "DebugEnv(R)" 项会让该键成为强制必需条件。

这种细微限制可用于允许调度管理员（wrangler）将作业“引导”到一组特定 Blade 上，这些 Blade 可能为特殊情况而“保留”。如果某个作业在常规渲染农场机器上行为异常，wrangler 可以把 "DebugEnv"（在此示例中）添加到该作业的作业级服务键（例如在 Dashboard 的输入字段中）。该作业重启后的任务随后会去往通告 "DebugEnv" 能力的机器。

这里的 "(R)" 会阻止常规作业也落到调试机器上。即使某些常规作业会匹配 某些 Blade 键，它们并没有请求该 Blade 所 必需 的键。

注意，仅仅把附加键添加到 job 确实 不会 改变通常的分配逻辑：新键会像往常一样成为匹配所必需的条件，因此该作业仍像往常一样只匹配提供该键的 Blade。不过，把 "DebugEnv(R)" 添加到 Blade profile 意味着这些 Blade 将不再匹配只请求 其他 Provides 列表中键的作业；相反，这些 Blade 将 只匹配服务请求至少包含 "DebugEnv" 的作业。
