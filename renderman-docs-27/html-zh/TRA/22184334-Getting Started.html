Tractor 安装位置和服务

Tractor 最常用于各组件作为系统服务运行的工作室，这些服务设计为在其主机重启时自动重启。

软件安装在哪里？如果你在 Engine 主机上本地运行 Tractor 安装程序，并在每个渲染 Blade 节点上再次运行，那么在 Linux 上它会出现在 /opt/pixar/Tractor-2.x。如果你改为从共享文件服务器安装位置运行 Tractor，则必须在服务自动重启之前挂载该安装位置。请参阅 installation 和管理员文档，了解典型服务设置和步骤。

你可以使用命令行命令 “ps -eaf | fgrep -i tractor” 来确认 Tractor 进程是否正在各台机器上运行。你需要 一个 “tractor-engine” 可执行实例在 Engine 主机上运行，并且每个 Blade 节点上都需要一个 “tractor-blade” 进程实例（它会显示为 “rmanpy -m tractor.apps.blade”）。

许可

Tractor 本身需要许可证才能调度作业，但启动不需要。许可证控制 tractor-engine 一次可调度的并发任务数量。Engine 日志会显示找到的许可证数量。你也可以在 Dashboard 中看到该数量：点击左上角的小 Tractor 图标，显示的文本条目之一会是 “# Licenses”，它表示 Engine 能够获取的 Tractor 许可证数量。如果该值为零，则不会发生调度。如果你有有效许可证，请检查 Engine 日志中关于连接 license server的问题消息。

连接到 Engine

首先要确认的是，你确实有路由和防火墙许可，能从各种客户端（包括 Blade）连接到具名 Engine 主机。

通常第一项检查是用 Web 浏览器连接到 Engine，输入一个指向 Engine “web server” 的 URL，例如 “http://tractor-engine.yoursite.com/tractor/”。

Dashboard、Blade 主机和其他 Tractor 客户端需要知道 Engine 主机的名称才能连接。因此，你要么需要在每种情况下指定正在使用的名称，要么需要为 Blade 默认尝试使用的名称（即 “tractor-engine”）设置名称服务器别名（例如在 DNS 中）。从长期看，设置 DNS 别名更容易，因为很多 Tractor 实用程序默认都会尝试这个名称。别名还允许你把 Engine 主机更改到另一台机器或不同地址，而不必更新所有 Blade、Dashboard 和其他客户端。Blade 和其他客户端还需要知道 Engine 监听的端口；这也是可配置的，但在设置 Engine 时使用默认端口 80 会更简单。

如果你有 DNS 条目，并且 Engine 正在端口 80 上监听，那么上面的 Dashboard 测试应该可以工作。在 Blade 主机上，另一个简单的网络健全性测试是运行：

/bin/tq ping

这会尝试连接到 tractor-engine 进程，并执行连接健全性检查。你可以在 Blade 节点或用户桌面的命令行上运行它来确认连接。它会针对不同类型的故障生成各种错误消息。例如，“Unknown host” 表示 “tractor-engine” 无法解析为主机名。

如果你尚未设置 “tractor-engine” 别名，则需要用一个指定 Engine 主机和端口的参数运行测试。如有必要，可以使用主机名或地址。例如：（不可运行的示例！）

tq --engine=my-other-host:80 ping
tq --engine=192.168.0.2:8888 ping


将 Blade 匹配到作业

将 Tractor 工作（命令）发送到 Blade 的基本机制基于 Service Key matching。Tractor 允许不同 Blade 节点宣告不同能力，并非所有任务都能在所有机器上运行。例如，一些任务可能无法在 Windows 上运行，或者一些任务需要安装某个应用软件，比如 RenderMan！

工作室可以开发自己的关键字约定，用这些关键字表示能力。作业会带着表示其需求的关键字提交到 Tractor 队列，而 Blade 会在渲染节点上用表示其能力的关键字启动。当两者匹配时，Tractor 会把命令分配给 Blade。默认情况下，作业会查找名为 “PixarRender” 的关键字，并且所有 Blade 默认都会宣告该 service。

可以在作业创作时指定不同的 Service Keys（如有需要，可按命令指定），Blade 能力则通过 blade.config 文件中的 “Provides” 条目指定。如果没有 Blade 宣告你的作业所需的 service，则不会发生调度。

如果作业已经位于队列中并在 Dashboard 中可见，那么可以右键点击给定作业，并从菜单中选择 “Trace Job”。右下角输出窗格会出现文本，描述应用到下一个恰好请求工作的 Blade 上的匹配测试。通常这个诊断足以确定典型的调度问题。
