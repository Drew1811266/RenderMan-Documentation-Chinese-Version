日志记录（Logging）

Downloaded: 2026-06-15T03:49:12+00:00

日志类型

Tractor 会生成几种类型的日志文件：

  • Engine 运行状态日志

    Engine 会输出有关自身状态和持续活动的日志，例如新作业被提交，或命令被分配给 Blade。这些日志会写入 Engine 命令行上通过 tractor-engine --log=filename 选项指定的文件。

  • Blade 进程日志

    运行在服务器主机上的每个 Blade，都会输出有关 tractor-blade 进程自身状态及其持续活动的日志，例如准备从用户作业启动命令。这些 Blade 日志会写入 Blade 命令行上通过 tractor-blade --log=filename 选项指定的文件。

  • 命令输出日志

    由 tractor-blade 启动的每个命令，例如 prman，也可以生成由 Tractor 捕获的日志输出。其中包括已启动程序写入 stdout 或 stderr 的任何内容。这些是命令输出日志，Tractor 会对它们进行特殊处理。更多详情请参见下方关于命令输出日志的说明。

注意事项：当 Engine 或 Blade 作为 systemd 服务安装时，默认由 journald 处理日志记录，因此不需要 --log 选项。
 

日志轮转（Log Rotation）

Engine 日志：可以使用 Linux logrotate 工具等外部机制进行轮转。Engine 收到 SIGHUP 后，会关闭由其 --log 命令行选项指定的文件，然后重新打开同一文件名继续记录日志。Linux logrotate 工具或类似工具可以根据站点指定的条件轮转日志：先通过 mv 操作移动日志文件（重命名它，同时 Tractor 保持同一个 i-node 打开），然后向 Tractor 发送 SIGHUP，使其创建新的日志文件。注意：如果系统配置对 Engine 日志使用 logrotate，并且同时使用 "compress" 选项，则也应为 Engine 日志处理器添加 "delaycompress" 选项。logrotate 在创建 gz 文件时存在竞态条件，可能会在 Engine 对旧日志的未完成的日志事务结束之前移除旧日志，而不是仅仅重命名它。下面是一个配置示例，例如 /etc/logrotate.d/tractor-engine，假定 tractor-engine 进程本身使用以下选项启动： --log=/var/log/tractor-engine.log：

/var/log/tractor-engine.log {
   size 25M
   rotate 5
   missingok
   postrotate
      /usr/bin/killall -HUP tractor-engine
   endscript
   compress
   compressoptions --best
   delaycompress
}

Blade 日志： tractor-blade.py 自身的诊断输出会定向到 --log 选项指定的文件，或默认定向到 stdout。使用文件时，它们达到约 25 MB 大小时会自动轮转。轮转由 Python 的 logging.handlers.RotatingFileHandler 执行。tractor-blade 命令行选项 --logrotate=NNN 会将轮转阈值设置为 NNN MB。这通常是最简单的方法。

在 少数 需要高级行为的情况下，也可以使用 Python 内置的 "logging.configs.fileConfig" 方案来指定 Blade 的日志配置。tractor-blade.py 选项 --logconfig=FILE 指定一个外部配置，假定它采用 Python 内置 logging 模块期望的格式。下面是一个日志配置文件示例，它实现默认日志记录方案，但重定向到特定文件：

[loggers]
keys=root,tractorblade

[handlers]
keys=hroot,bhrotater

[formatters]
keys=tbfmt

[logger_root]
level=NOTSET
handlers=hroot

[logger_tractorblade]
level=INFO
handlers=bhrotater
propagate=0
qualname=tractor-blade

[formatter_tbfmt]
format=%(asctime)s %(levelname)-7s %(message)s
datefmt=%m/%d %H:%M:%S

[handler_hroot]
class=StreamHandler
level=NOTSET
formatter=tbfmt
args=(sys.stdout,)

[handler_bhrotater]
class=handlers.RotatingFileHandler
level=INFO
formatter=tbfmt
args=('/tmp/b.txt', 'a', 26214400, 5)

由 Blade 启动的应用程序输出，例如渲染进程，不会进行轮转。对于这些命令，每个任务会创建一个新的日志文件。


命令输出日志详细信息

命令输出的保存方式

每个 tractor-blade 启动的单个命令的输出会写入按任务划分的日志文件。Blade 还会过滤命令输出流，以提取进度指示器，用于生成用户界面更新，例如完成百分比进度条。

blade.config 配置档设置 CmdOutputLogging 控制每种 Blade 如何执行这些写入。输出日志可以通过 Tractor 专用 TCP 套接字连接传递到中央 Tractor（Python）日志服务器；也可以使用标准（网络）文件 I/O（例如 NFS）直接写入中央文件服务器上的共享日志区域。

对于已经将中央文件服务器用于其他用途的站点，"直接记录到文件服务器" 方法应视为最佳实践策略，因为它性能更高，通常更稳健，并且输出能以自然方式供其他工具和用户使用。此文件服务器方法确实要求共享网络日志文件位置可由每个 Blade 写入。每种 Blade 配置档所看到的共享/挂载名称可以不同，这取决于该类系统的 OS 和挂载要求。

使用 Tractor 专用日志服务器的替代方法是默认设置，因为它不要求站点在所有 Blade 上配置并挂载中央文件服务器。不过，它要求日志服务器本身必须启动，并指向正确位置来存储日志文件。它还必须有权限在 logserver 主机上未使用的网络端口上开始监听连接。启动服务器的命令通过 SiteCmdLogServerStartup 设置在 tractor.config 文件中指定。如果设置了该项，则 tractor-engine 会在 Engine 启动时将 logserver 作为子进程启动。主机名和端口的选择也必须与 blade.config 中每种 Blade 类型的 CmdOutputLogging 设置协调。

命令输出日志的检索方式

检索并查看输出日志的机制，与上面选择哪种方法把输出文件传递到中央日志位置无关。

Tractor Dashboard 以及其他感兴趣的应用程序可以通过 HTTP 请求检索日志文本。例如，Web 浏览器可以直接获取并显示日志文本。要使其工作，需要一个 HTTP 服务器向请求方提供这些文件。此 Web 服务器可以是 Tractor Engine 本身，也可以是站点已在使用的现有 Web 服务器（Apache 等）。它只需要能通过文档根目录方案访问日志文件。

每个捕获的输出日志文件使用的名称，会根据与命令关联的作业标识符和任务标识符选择。这样文件可以很容易被用户界面引用，也可以在不再需要时清理。

文件名模板字符串可以包含如下替换模式：

%u the job owner's login (userid, string) %J the job id (integer) %T the task id (integer)


将命令输出记录到中央文件服务器

把 Tractor 命令输出记录到中央文件服务器应视为一种最佳实践技术，尤其适用于大型生产站点。下面的 Tractor 日志记录背景信息有助于解释后续设置步骤。

整体性能是让每个 Blade 写入自身命令日志输出的基本动机。与所有日志都通过 TCP 套接字自动回流到公共位置的方案相比（例如 Alfred 命令日志的情况），此方案管理起来没有那么简单。然而，对于非常大量的输出，例如使用每个像素都有 print 语句的着色器进行渲染，将数据写入每个 Blade 的本地磁盘非常快，且不会产生网络流量。使用 "local" 文件操作直接写入高性能网络共享的效果也几乎同样好，并在日志访问和管理方面提供许多额外好处。每个 Blade 看到的日志位置由 CmdOutputLogging 设置在 blade.config 文件中定义。关于此设置的更多详情见下文。

当用户界面向 tractor-engine 请求任务日志时，Engine 实际返回一个（JSON 格式化）URL，UI 可以使用该 URL 自行获取日志。此方法让具体 UI 实现决定如何获取日志，同时也允许 UI 和实际日志服务器之间进行内容协商。例如，在移动设备上查看日志时，这可能很重要。

tractor-engine 传递给 UI 的 URL 形式由 SiteCmdLogRetrievalURL 设置在主 tractor.config 文件中决定。

下面是实现集中式日志记录所需步骤的摘要：

  • 选择一个有大量空间的文件服务器，它可以作为可写目录被每个 Blade 主机挂载。常见网络文件系统方案包括 NFS、DFS 和 SMB。

  • 所选文件服务器共享必须能由每个 Blade 主机上的 Blade 进程所有者写入；这里指的是 Blade 进程 本身的所有者，而不是 Blade 启动的每个应用程序的所有者，后者在某些情况下可能 setuid 为作业所有者。

  • 由于 Blade 进程往往在启动时作为 "守护进程" 启动（通过 init.d、launchd 或 Windows 上的 System Services），文件服务器挂载通常也必须在启动时可用，最好在 Blade 启动之前可用。

  • 每个 Blade 启动时会从中央 Tractor blade.config 文件获取命令日志位置的名称。此文件由 tractor-engine 按需传递给每个 Blade，一个站点应只有一个 blade.config 文件；它包含 Blade 配置档，用于描述每种类型的 Blade 主机的各种配置设置，包括 CmdOutputLogging 位置。

    用于命令日志的网络共享名称在每个 Blade 主机看来不一定必须相同，但对于共享同一个 blade.config 配置档条目的所有 Blade，它必须相同。例如，Linux 机器可能使用自动挂载路径，而 Windows 机器可能使用指向共享的 UNC 路径。例如：

    "CmdOutputLogging": "logfile=/net/spindles/tractor/cmd-logs/%u/J%j/T%t.log"

    或

    "CmdOutputLogging": "logfile=//spindles/tractor/cmd-logs/%u/J%j/T%t.log" 

  • 请注意，建议格式即使在 Windows 上也使用正斜杠，而不是反斜杠。你可以使用标准 Windows 反斜杠表示法，但需要把每个反斜杠都写成双反斜杠。这里的定义会发送给 Blade，并且反斜杠会被当作转义字符（在 Python 中）处理。

  • 注意，管理员可以对 blade.config 进行运行时变更，并通过向 Engine 发送 "reconfigure" 请求，使更新后的版本重新分发给所有正在运行的 Blade。

  • 在启动会使用中央日志位置的 Blade 之前，务必先创建该位置。还应做一些测试，确保服务器主机上的 Blade 所有者可以在那里创建子目录并写入。

  • 然后决定用户界面应如何通过 URL 访问新共享位置中的文件。一个有用的方法是在站点中找到一个现有标准 Web 服务器，它已经对公共日志目录具有读取权限，并可按请求提供这些文本文件。例如，可以在 Web 服务器的主文档根目录下添加指向日志仓库的符号链接。（另见下方 "CORS" 说明。）

  • 然后修改 Engine 的 tractor.config 文件，使其引用共享日志 URL。 SiteCmdLogRetrievalURL 设置是一个 URL 模板，其中包含请求的日志文件占位符，Engine 会从 UI 请求中提供这些值。例如：

    "SiteCmdLogRetrievalURL": "http://internal_web/links/spindles/tractor/cmd-logs/%u/J%j/T%t.log",

注意： logfile 模板 '%u/J%j/T%t.log' 在 logfile 定义和检索 URL 中保持匹配非常重要。


通过标准 Web 服务器提供日志

使用标准 Web 服务器向 Tractor Dashboard 和其他请求方传递这些命令日志有几个好处。首先，它可以把这类文件 I/O 从 Tractor Engine 本身卸载出去。更重要的是，它还允许用户通过通用 Web 浏览器直接浏览日志，只需遍历 Web 服务器提供的作业目录列表即可。显然，底层共享文件服务器本身也允许任意工具和脚本使用普通文件读取操作访问日志。

对 "Inline" Dashboard 请求的 CORS 跨源访问支持

在 Tractor Dashboard 中，可以通过双击作业图中的任务框，或从该任务的右键菜单中选择 "Logs" 来查看命令日志。这会在 Dashboard 界面 "内部" 的新窗格中显示日志。也可以选择名为 "Logs (in new window)" 的菜单项，让日志显示在新的独立浏览器窗口中。

使用 Web 服务器而不是 tractor-engine 来提供这些日志文件时，有一个技术细节：可能需要对 Web 服务器配置做一个小改动，才能让它与你浏览器的 "同源安全策略" 设置成功配合。这些策略通常很有用，可在某个站点的 JavaScript 试图在你不知情的情况下访问完全不同网站的资源时保护隐私。在某些情况下，如果你 确实 希望两个网站协作，例如 tractor-engine 和你的日志 Web 服务器，就需要在日志 Web 服务器上明确指定哪些内容可共享。

这项技术称为跨源资源共享，或简称 CORS；大多数 Web 服务器都提供启用方式。基本思路是：Dashboard 由 tractor-engine 提供给浏览器的 HTML 和 JavaScript 内容组成，有时该 JavaScript 会向另一个 Web 服务器（日志服务器）发出 HTTP（XHR）请求。只有日志服务器允许这种共享时，浏览器才会允许该 "跨站点" 事务。以 Linux 上的 Apache 为例，可以向 /etc/httpd/conf.d 中的新文件或现有文件添加类似下面的文本：

<Directory /var/www/html/tractor-logs>
   Header set Access-Control-Allow-Origin *
</Directory>

当然，你应使用集中式日志区域的实际路径。你也可以使用比 "*" 更严格的设置；详情请查阅 Web 服务器文档。
