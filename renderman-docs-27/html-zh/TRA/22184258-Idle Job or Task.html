Idle Job or Task

Source: https://rmanwiki-27.pixar.com/space/TRA/22184258/Idle+Job+or+Task
Downloaded: 2026-06-15T03:49:12+00:00

调度系统用户常见的问题是： 为什么这个作业（或任务）没有运行？ 作业可能不运行有许多原因：网络或硬件故障、系统配置错误，或者只是正常调度规则阻止了作业执行。

要理解作业如何以及何时变为活动状态，跟踪作业任务在系统中的生命周期可能会有帮助。

作业必须已经提交。

首先，你必须已经按下某个应用内按钮或发出提交作业的命令。作业必须先由 Engine 处理并加载。

要验证 Engine 是否已收到作业，请检查它是否在 Dashboard 中可见。如果看不到该作业：

  • 确保没有作业过滤器或客户端搜索框阻止作业显示，
  • 检查 Dashboard 是否指向与提交该作业的实体相同的 Engine，并且
  • 检查 Engine 日志文件中是否在作业被提交时出现过 “job Loaded” 消息。日志文件中相邻的错误消息可能表明作业文件存在语法错误。

你也可以在命令行上查看最近提交的作业，例如 “tq jobs mine --sort jid”。

作业必须能够运行。

验证作业 没有处于以下状态：

  • 未暂停，
  • 未延迟，且
  • 没有按照其 afterJids 列表所指定的那样等待其他作业完成。

作业必须有就绪任务。

验证作业确实有就绪任务。提交作业后，至少会有一个可运行的就绪任务（除非作业作者能够编织出具有循环依赖的作业！）。就绪任务会是作业图上的叶节点，不过并非所有叶节点都一定就绪。可以在命令行上用 “tq tasks state=ready and jid=<jid>” 列出作业的就绪任务。

提交时的剩余任务会处于 blocked 状态；一旦其所有子任务成功完成（或被手动跳过），某个任务就会变为就绪。对于串行子任务，任务可能还必须等待其“较早”的同级任务完成后才能变为就绪。

Blade 必须正在请求工作。

验证 Blade 确实正在请求工作（并且请求正确的 Engine:port）。参与的 Blade 会在 Dashboard 的 Blades 标签页中可见。如果一个都没有，请确保 Blade 正在联系正确的 Engine，并且 Blade 和 Engine 主机的防火墙中打开了正确端口。Engine 的默认端口是 80 (TCP)，Blade 的默认端口是 9005 (TCP)。（注意，Blade 和 Engine 上必须打开 9005 (UDP)，进度消息才能报告给 Engine；不过这不会影响任务是否在 Blade 上启动。）

你可以在 Dashboard 中运行 tracer，验证某个 Blade 是否正在请求工作。这会报告哪些作业已被考虑用于调度。如果相关作业显示在 trace 中，也会显示它未被选中的原因。如果相关作业没有显示，可能是因为选择了更高优先级作业的任务，或因为其他调度规则会阻止该作业在该 Blade 上运行。

还要考虑，如果 Blade 在某个间隔内发生若干任务失败，它们可能不会请求工作。这样的 Blade 会在 Blade 列表的 Note 列中标记为 error accrual hiatus。 Blade 的配置档案页面 中的 RecentErrorThrottle 设置管理这个暂停计算的参数。

调度规则必须允许考虑该作业。

有几条调度规则决定任务是否在给定 Blade 上运行：

  • 发出请求的 Blade 的 service keys 必须匹配任务的 service key expression，
  • 发出请求的 Blade 的指标必须满足任务的指标需求，
  • 任务不得受 limits 限制。这里的 limits 可能由作业、任务或任务当前命令的 limit tags 表示，并且
  • 任务不得受 sharing limits 限制。这里的 limits 是与作业项目关联的那些 limits。

即使某个任务满足这些需求，也可能存在一个更高优先级作业的任务同样满足这些需求，并因此被作业选中。

还要注意，每次任务分配和完成都会影响 limit 计数器，这会影响哪些任务有资格响应下一个 Blade 请求。
