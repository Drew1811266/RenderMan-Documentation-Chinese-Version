Limits Overview（限制系统概述）

Tractor Limits system 可用于监控并强制执行 farm-wide restrictions，以控制某些 limited resources。它的做法很简单：维护与 site 自行选择的 abstract keywords 相关联的 counts。commands 启动时递增这些 counts，commands 退出时递减。Sites 可以指定：只有当前 count 低于给定 threshold 时，才允许启动带有特定 limit keywords 的新 commands。

考虑一个 example：我们有一个名为 "AwesomeUnicorn" 的 application，已安装在全部 100 个 renderfarm nodes 上。不过，我们只有 15 个 license seats，因此需要避免同时启动超过 15 个 concurrent instances。一旦已有 15 个在运行，就必须等一些退出后才能继续启动，最高不超过 15 的 limit。来自 job queue 的其他类型 commands 可以在剩余 idle farm nodes 上运行。Tractor 会维护 application names 在启动时的 tally，以及其他任意 limit tag（资源限制标签） names，这些 names 可由 job script 附加到每个 command。这种计数称为 SiteMax global limit（全局资源限制），因为它适用于来自任意 job、在 farm 任意位置运行的 command keywords。

基本 limit counting policy 由 limit.config setting SiteMaxCounting 控制。目前它有两个可能 settings；在默认"perInvocation"（按调用计数） mode 下，每次 command dispatch 都会递增 limit counts。另有一种名为 "perHost" 的 alternate global counting scheme，它只在带有特定 tag 的 第一个 command 落到给定 farm host 上时递增 count。随后在同一 host 上运行的类似 commands 不会递增该 limit 的 count。这个设计用于支持某些 application licensing scenarios：在这些场景中，一个 license 支持同一 host 上运行 unlimited instances 的 application。

Tractor 还会为每个 limit 维护 BladeMax tally，并按 blade hostname 建立索引。此值可用于限制任意单个 blade 上可运行的 concurrent invocations 数量。例如，我们可能判断 AwesomeUnicorn app 非常强大（或 CPU intensive），因此同一时间每个 blade 上只应运行一个 instance。也可以指定 per-blade exceptions。

类似地，每个 limit 还会维护一个单独 tally，并按 job owner name 为每个 limit tag 建立索引。此 OwnerMax value 可用于把 individual users 限制在消耗给定 limit tag 的某个 concurrent invocations 数量内。例如，我们可能希望指定：任何 user 都不应同时使用超过 5 个 AwesomeUnicorn licenses。如下所示，site 可以指定 general owner limit，也可以为特定 users 指定若干 exceptions。

limit definition 还可以指定 JobMax value，用于约束每个 individual job（单个作业项目） 在 queue 上可并发启动并消耗该特定 limit tag 的数量。这是 site-wide setting，会影响所有使用该 limit 的 jobs。

注意：除了 limits.config definitions 中的 JobMax value， 每个 job 自身还有一个 maxActive attribute，它以另一种方式对 job 在 farm 上的 "footprint" 提供 per-job control。每个 job 的 maxActive attribute 会指定该 job 同时 active tasks 数量的简单上限。这是一种通用 active task count restriction，独立于 limit policies 或 command type。默认情况下，每个 job 的 maxActive attribute 设置为 zero，表示 "unconstrained"，并使用本页描述的常规 site-wide Limit policies 控制 resource usage。maxActive attribute 是额外 constraint，用于进一步减少 job 的 blade usage。

Adaptive Farm Allocations（自适应 Farm 分配） 是在 Tractor 2.0 中引入的。这个 limit sharing mechanism 提供了一种方式，可把给定 site-wide limit 的 flexible percentages 分配给不同 types of jobs。例如，来自 project "X" 的 jobs 获得 farm 的 10%，来自 show "Y" 的 jobs 获得 60%，其他 jobs 可以使用剩余部分。下面会更详细描述这种 sharing。
 

limits.config 文件（限制配置文件）

limits.config 文件以 JSON notation 指定一个 limit definitions dictionary。每个 limit definition 由必需的 global maximum concurrent count 和 owner maximum concurrent count，以及 optional modifiers 组成。必须在 site override directory中创建此文件。

对于 SiteMax、OwnerMax 或 BladeMax，某个值 -1 表示允许 unlimited（无限制数量） 数量的 invocations 调用实例。

{
   "Limits":
   {
      "prman":    {"SiteMax": 100, "OwnerMax": -1},

      "thingy"   {  "SiteMax": 25,
                    "SiteMaxCounting": "perHost",
                    "OwnerMax": 5,
                    "OwnerExceptions": {"bob": 2, "alice": 10},
                    "BladeMax": 2,
                    "BladeExceptions": {"gigantor": 8, "myphone": 0}
                  },

      "aparatus": {"SiteMax": -1, "OwnerMax": 2}
   }
}

在 Job Scripts 中指定 Limit Tags

Tractor jobs 可以显式声明它们会消耗的 limit tags。通常使用 Job 或 RemoteCmd 的 -tags list parameter 指定。若在 job level 指定，这些 tags 会用于扩展每个 command 自己的 list（如果有）。

RemoteCmd "/bin/AwesomeUnicorn mybest.corn" -tags "w00t spoon" -service "AUserver"

此 example 会递增名为 "AwesomeUnicorn"、"w00t" 和 "spoon" 的 limit tags。

Blade Triggered Limit Tags（由 Blade 触发的限制标签）

有时，不管正在运行的 job type 是什么，只要使用某种类型的 blade，就希望递增某个 limit counter。如果 administrator 只是想知道给定 profile 中有多少 blades 正在使用，就可能出现这种需求。不过，此 feature 通常用于下面描述的 "limit sharing" allocations 上下文。

要让 blade assignment 触发 limit increment，请向 limit definition 添加 TaggedBlades value；它接受一个 list，包含应触发 counting 的 profile names 或 host names。
 

Adaptive Farm Allocations（自适应 Farm 分配）

Limit Sharing 提供了一种方式，可把给定 limit 的 flexible percentages 分配给不同 types of jobs。

默认情况下，limit count 是简单的 "global" tally：与任何已启动 command 相关联的 limit tags 会在 launch 时递增，并在 exit 时递减。如果某个 particular limit tally 已达到允许的 maximum count，那么使用该 tag 的 commands 在某些先前 commands 退出之前都不允许启动。

limit sharing 变体会使这些 limit counts 被 归属 到某个具名 project，并由 system 在这些 projects 之间强制执行 usage ratio。

因此，sharing 所需的 ingredients 包括：

  • 添加 Job -projects attribute，用于指定每个 job 的 affiliation。未关联的 jobs 会归属到名为 "default" 的 project。

  • 添加 Shares dictionary 到 limits.config 中的某个 limit definition，用来描述该 limit 按 project 分配的 allocations。

  • 确保 shared limit 在 dispatching 期间生效，方法是把它的 limit tag name 添加到 job scripts，或者把 TaggedBladeskeyword 添加到 limit definition 本身，使所有分配到具名 blades（profiles）的 assignments 都被 tally。

未指定 project affiliation 的 jobs 会归属到名为 "default"的 project。Project affiliation 可以在 Dashboard 中添加或修改。limit definition 中命名的 "Shares" 代表该 limit usage 的 100%。如果 fractional named allocations 的总和不是 1.0，则所有剩余 capacity 会自动分配给 "default" project。"Shares" definition 也可以显式定义 "default" allocation 的大小。如果给定 limit definition 中所有 allocations 的总和高于或低于 1.0，则整组 values 会按比例缩放，使新的 total 为 1.0。
 

Sharing Example（共享分配示例）

位于 limits.config 中的某处 ...

{
  "Limits":
  {
     "prman": {
         # a limit in the classical style
         "SiteMax": 100, "OwnerMax": -1
     },
     "theWholeFarm": {
        "SiteMax":  1000,
        "TaggedBlades": "Linux64", # for example, all blades in this Profile
        "Shares": {
           "Rocket": {"nominal": .65, "cap": .75},
           "Pawns":  {"nominal": .35, "reserve": 0.10}
        }
     },
     "someOtherLimitName": {
        "SiteMax":  234,
        "Shares": {
           "lookdev": {"nominal": .5},
           "research": {"nominal": .25, "cap": 0.5},
           "default": {"nominal": .25}
        }
     }
  }
}

假设我们当前有两个 shows 正在 production 中。我们会使用 annotations spool jobs，为每个 job 给出 project affiliation(s)。

Job -projects {Pawns} -title "pawn_hair_test1" ... or Job -projects {Rocket} -title "rktpkt" ...

参考上面的 example，我们创建了一个名为 "theWholeFarm" 的 limit tag name，用来抽象表示所有使用 "Linux64" blade profile 的 render farm blades。

Nominal Allocations（名义分配） -- 在该 limit 内，我们创建了两个 Shares， 表示把 theWholeFarm 分配给每个 project 的 allocation。Allocations 以 limit 的 SiteMax count 的 fraction 给出。如果 allocations 总和大于 1.0，会发生 rescaling。

当 tasks 被分配给 blades 时，每个 project 的 tally 会相应递增。theWholeFarm limit tag 的 usage 始终是其 shares 中 counts 的总和。

Nominal Maximum Utilization（名义最大利用率） -- 一旦某个 project 的 count 达到该 share 指定的 maximum，就不会再 dispatch 该 project 的额外 tasks，直到先前的 task 退出。

Dynamic Overflow（动态溢出） -- 当某个 share 未充分利用时，其他 shares 可选择 overflow 超出其 nominal（名义） allocation，直到一个更大的预定义 cap（上限） value。

Reserved Allocations（保留容量分配） -- share 可以选择保留 farm 的一部分；某些 blades 会保持 idle，直到该 project 需要它们。
