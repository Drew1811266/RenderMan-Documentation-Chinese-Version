限制系统概述

Tractor Limits 系统可用于监控并强制执行整个渲染农场范围的限制，以控制某些受限资源。它的做法很简单：维护与站点自行选择的抽象关键字相关联的计数。命令启动时递增这些计数，命令退出时递减。站点可以指定：只有当前计数低于给定阈值时，才允许启动带有特定限制关键字的新命令。

考虑一个示例：我们有一个名为 "AwesomeUnicorn" 的应用程序，已安装在全部 100 个渲染农场节点上。不过，我们只有 15 个许可证席位，因此需要避免同时启动超过 15 个并发实例。一旦已有 15 个在运行，就必须等一些实例退出后才能继续启动，最高不超过 15 的限制。来自作业队列的其他类型命令可以在剩余空闲农场节点上运行。Tractor 会维护应用程序名称在启动时的计数，以及其他任意 limit tag（资源限制标签） 名称，这些名称可由作业脚本附加到每个命令。这种计数称为 SiteMax global limit（全局资源限制），因为它适用于来自任意作业、在渲染农场任意位置运行的命令关键字。

基本限制计数策略由 limit.config 设置 SiteMaxCounting 控制。目前它有两个可能设置；在默认 "perInvocation"（按调用计数） 模式下，每次命令分派都会递增限制计数。另有一种名为 "perHost" 的替代全局计数方案，它只在带有特定标签的第一个命令落到给定渲染农场主机上时递增计数。随后在同一主机上运行的类似命令不会递增该限制的计数。这个设计用于支持某些应用程序许可场景：在这些场景中，一个许可证支持同一主机上运行该应用程序的无限实例。

Tractor 还会为每个限制维护 BladeMax 计数，并按 blade hostname 建立索引。此值可用于限制任意单个 Blade 上可运行的并发调用数量。例如，我们可能判断 AwesomeUnicorn 应用程序非常强大（或 CPU 密集），因此同一时间每个 Blade 上只应运行一个实例。也可以指定逐 Blade 例外。

类似地，每个限制还会维护一个单独计数，并按 job owner 名称为每个限制标签建立索引。此 OwnerMax 值可用于把单个用户限制在消耗给定限制标签的某个并发调用数量内。例如，我们可能希望指定：任何用户都不应同时使用超过 5 个 AwesomeUnicorn 许可证。如下所示，站点可以指定通用所有者限制，也可以为特定用户指定若干例外。

限制定义还可以指定 JobMax 值，用于约束每个 individual job（单个作业） 在队列上可并发启动并消耗该特定限制标签的数量。这是全站设置，会影响所有使用该限制的作业。

注意：除了 limits.config 定义中的 JobMax 值，每个作业自身还有一个 maxActive attribute，它以另一种方式对作业在渲染农场上的 "footprint"（占用规模）提供逐作业控制。每个作业的 maxActive 属性会指定该作业同时活动任务数量的简单上限。这是一种通用活动任务计数限制，独立于限制策略或命令类型。默认情况下，每个作业的 maxActive 属性设置为 zero，表示 "unconstrained"（不受约束），并使用本页描述的常规全站 Limit 策略控制资源使用量。maxActive 属性是额外约束，用于进一步减少作业的 Blade 使用量。

Adaptive Farm Allocations（自适应渲染农场分配） 是在 Tractor 2.0 中引入的。这个限制共享机制提供了一种方式，可把给定全站限制的灵活百分比分配给不同类型的作业。例如，来自项目 "X" 的作业获得渲染农场的 10%，来自制作项目 "Y" 的作业获得 60%，其他作业可以使用剩余部分。下面会更详细描述这种共享。

limits.config 文件（限制配置文件）

limits.config 文件以 JSON 表示法指定一个限制定义字典。每个限制定义由必需的全局最大并发计数和所有者最大并发计数，以及可选修饰项组成。必须在 站点覆盖目录中创建此文件。

对于 SiteMax、OwnerMax 或 BladeMax，值 -1 表示允许 unlimited（无限制） 数量的调用。

{
   "Limits":
   {
      "prman":    {"SiteMax": 100, "OwnerMax": -1},

      "thingy"   {  "SiteMax": 25,
                    "SiteMaxCounting": "perHost",
                    "OwnerMax": 5,
                    "OwnerExceptions": {"bob": 2, "alice": 10},
                    "BladeMax": 2,
                    "BladeExceptions": {"gigantor": 8, "myphone": 0}
                  },

      "aparatus": {"SiteMax": -1, "OwnerMax": 2}
   }
}

在作业脚本中指定限制标签

Tractor 作业可以显式声明它们会消耗的限制标签。通常使用 Job 或 RemoteCmd 的 -tags 列表参数指定。若在作业级别指定，这些标签会用于扩展每个命令自己的列表（如果有）。

RemoteCmd "/bin/AwesomeUnicorn mybest.corn" -tags "w00t spoon" -service "AUserver"

此示例会递增名为 "AwesomeUnicorn"、"w00t" 和 "spoon" 的限制标签。

由 Blade 触发的限制标签

有时，不管正在运行的作业类型是什么，只要使用某种类型的 Blade，就希望递增某个限制计数器。如果管理员只是想知道给定配置档中有多少 Blade 正在使用，就可能出现这种需求。不过，此功能通常用于下面描述的 "limit sharing"（限制共享）分配上下文。

要让 Blade 分配触发限制递增，请向限制定义添加 TaggedBlades 值；它接受一个列表，包含应触发计数的配置档名称或主机名称。
 

Adaptive Farm Allocations（自适应渲染农场分配）

Limit Sharing 提供了一种方式，可把给定限制的灵活百分比分配给不同类型的作业。

默认情况下，限制计数是简单的 "global"（全局）计数：与任何已启动命令相关联的限制标签会在启动时递增，并在退出时递减。如果某个特定限制计数已达到允许的最大计数，那么使用该标签的命令在某些先前命令退出之前都不允许启动。

限制共享变体会使这些限制计数归属到某个具名project，并由系统在这些项目之间强制执行使用比例。

因此，共享所需的组成部分包括：

  • 添加 Job -projects 属性，用于指定每个作业的项目归属。未关联的作业会归属到名为 "default" 的项目。

  • 向 limits.config 中的某个限制定义添加 Shares 字典，用来描述该限制按项目分配的份额。

  • 确保共享限制在分派期间生效，方法是把它的 limit tag 名称添加到作业脚本，或者把 TaggedBlades 关键字添加到限制定义本身，使所有分配到具名 Blade（配置档）的分配都被计数。

未指定 project 归属的作业会归属到名为 "default" 的项目。项目归属可以在 Dashboard 中添加或修改。限制定义中命名的 "Shares" 代表该限制使用量的 100%。如果具名分配份额的总和不是 1.0，则所有剩余容量会自动分配给 "default" 项目。"Shares" 定义也可以显式定义 "default" 分配的大小。如果给定限制定义中所有分配的总和高于或低于 1.0，则整组值会按比例缩放，使新的总和为 1.0。

共享分配示例

位于 limits.config 中的某处 ...

{
  "Limits":
  {
     "prman": {
         # a limit in the classical style
         "SiteMax": 100, "OwnerMax": -1
     },
     "theWholeFarm": {
        "SiteMax":  1000,
        "TaggedBlades": "Linux64", # for example, all blades in this Profile
        "Shares": {
           "Rocket": {"nominal": .65, "cap": .75},
           "Pawns":  {"nominal": .35, "reserve": 0.10}
        }
     },
     "someOtherLimitName": {
        "SiteMax":  234,
        "Shares": {
           "lookdev": {"nominal": .5},
           "research": {"nominal": .25, "cap": 0.5},
           "default": {"nominal": .25}
        }
     }
  }
}

假设我们当前有两个制作项目正在制作中。我们会使用注解提交作业，为每个作业给出项目归属。

Job -projects {Pawns} -title "pawn_hair_test1" ... or Job -projects {Rocket} -title "rktpkt" ...

参考上面的示例，我们创建了一个名为 "theWholeFarm" 的限制标签名称，用来抽象表示所有使用 "Linux64" Blade 配置档的渲染农场 Blade。

Nominal Allocations（名义分配） -- 在该限制内，我们创建了两个 Shares， 表示把 theWholeFarm 分配给每个项目的份额。分配以限制的 SiteMax 计数分数给出。如果分配总和大于 1.0，会发生重新缩放。

当任务被分配给 Blade 时，每个项目的计数会相应递增。theWholeFarm 限制标签的使用量始终是其各 share 中计数的总和。

Nominal Maximum Utilization（名义最大利用率） -- 一旦某个项目的计数达到该 share 指定的最大值，就不会再分派该项目的额外任务，直到先前的任务退出。

Dynamic Overflow（动态溢出） -- 当某个 share 未充分利用时，其他 share 可选择溢出超出其 nominal（名义） 分配，直到一个更大的预定义 cap（上限） 值。

Reserved Allocations（保留容量分配） -- share 可以选择保留渲染农场的一部分；某些 Blade 会保持空闲，直到该项目需要它们。
