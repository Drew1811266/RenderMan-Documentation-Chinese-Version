Tractor 2 功能

Source: https://rmanwiki-27.pixar.com/space/TRA/22184354/Tractor+2+Features
Downloaded: 2026-06-15T03:49:12+00:00

Tractor-2.0 为 核心 Tractor 系统 新增了重要功能，该系统建立在 1.0 系列版本的基础上。新增和改进范围很广：既有面向调度管理员的全新命令行与脚本接口，也有简单的用户界面变化；既包括将整个作业数据库升级到新的高性能、高能力格式，也包括新的全工作室资源共享与分配控制。

以下是一些重点：

  • 改进的 Tractor 产品布局 -- 单一发布目录、每个平台单一下载包、捆绑的子系统更新 -- 新的 Tractor 2.0 打包和安装布局会把匹配的一组 Engine、Blade、提交器、用户界面和脚本组件放在同一位置，同时只保留一份共享资源，其中包括多个第三方子系统的预构建版本。

 

  • Tractor 查询工具 -- 这里引入 tq： Tractor 查询 命令行工具和模块。tq 基于经过 Pixar 工作室验证的工具，是从终端 shell、Python 脚本或 Dashboard 的新标签页查询实时 Tractor 数据的最佳方式。

 

  • 自适应农场资源分配 -- 这是一种新方式，可使用 Tractor 灵活的 *Limits* 系统在人员或项目之间动态分配抽象资源。如果两部影片同时制作，可以把 60% 的渲染农场分配给其中一部，30% 分配给另一部，剩余 10% 留给其他项目。如果某个影片项目空闲，其他项目可以临时扩展自己的份额；当所有项目都处于活动状态时，再收缩回名义水平。

 

  • 调度层级 -- 一种简单方式，可把大范围作业组织成按站点定义的降序优先级组。默认层级名称为： 默认名称为 admin、rush、default、batch。 也可以自行创建！

 

  • 自定义菜单操作 -- 添加站点定义的 Dashboard 菜单项，用于调用集中管理的脚本，并由用户当前的列表选择作为参数。

 

  • Job Authoring API（作业编写 API） -- 新的 tractor.api.author 模块允许 Python 脚本轻松创建 Job、Task 和 Command 对象，并按照依赖关系需求把它们链接起来。随后 Job.spool() 方法会把生成的作业发送到 tractor-engine 的作业队列。

  • 简单 Engine 发现 -- 面向小型工作室的新零配置通告能力，允许 tractor-blades 和其他 Tractor 工具在本地网络中找到 tractor-engine，而不需要手动修改名称服务（DNS/LDAP）配置。在管理员已经在站点名称服务数据库中创建主机名别名 tractor-engine 的工作室里，tractor-engine 会自动禁用这种 SSDP 风格的流量。

 

  • 检查点-恢复支持 -- 对作业脚本、调度和 Dashboard 的新扩展，增加了与增量计算相关的有用能力。Tractor 还支持通用的“从上一个检查点重试”机制。这两项功能都与新的 RenderMan 19 RIS 检查点和增量渲染功能集成。

 

  • Blade 自动更新 -- 新的 tractor-blade 补丁管理系统允许管理员把渲染农场固定到特定的 Blade 补丁版本，并自动把新版本推送到整个渲染农场。过期的 Blade 会使用新的模块版本重启自身。

 

  • PAM（可插拔认证模块）支持 -- Engine 新增的可选内置 PAM 支持，会把密码验证直接委托给 Engine 主机上的操作系统。对于 LAN 已经提供足够凭据传输安全性的工作室，这一替代方案可以很容易地启用密码支持。

 

  • 权限隔离 -- 新的 EngineOwner 设置位于 tractor.config 中，用于指定 tractor-engine 应该以哪个登录身份运行。这个设置很重要，因为它允许 Engine 在获取可能需要的受保护网络端口后立即放弃 root 权限。随后 Engine 的日常正常操作会在指定登录名关联的限制下进行。

 

  • 动态服务键能力通告 -- 新增了几个 Blade 配置档 "Provides" 模式，以支持一些高级用例。例如，Blade 可以根据之前启动的命令已经消耗了哪些服务键，动态通告不同的一组服务键能力。

 

  • 资源使用情况跟踪 -- CPU、RSS 和 VSZ rusage 进程指标现在会记录到每个已启动命令的作业数据库中。目前支持 Linux 和 OSX tractor-blades。

 

  • 命令重试历史 -- 现在每一次命令启动尝试都会在作业数据库中创建唯一的跟踪记录。因此，可以使用 tq 工具查看给定任务上的重试历史。

 

  • 配置文件加载 -- 简化的覆盖系统有助于减少杂乱，并更清楚地显示工作室中哪些文件已经相对原始出厂设置被修改。

 

  • 任务并发节流 -- 每个作业都有新的 maxActive 属性，用于限制该作业中允许同时运行的最大任务数量。这个对作业在渲染农场上的占用规模的可选控制，在某些情况下可为少量作业提供有用的临时调度管理控制，尤其是当修改站点级 Limits 设置并不合适时。

  • Blade 自动错误节流 -- 新的 Blade 配置档设置，如果 Blade 在给定时间间隔内遇到过多错误，可阻止它们接收新工作。

 

  • 作业提交改进 -- 作业处理升级包括更快的处理速度、更好的错误检查，以及必需子系统的捆绑。并行化的作业接收和数据库暂存机制，在大量作业同时提交，或大量 expand 任务并行运行时，可以显著减少积压。提交器捆绑的自包含 Tcl 解释器简化了站点安装要求，并可在作业交付给 Engine 之前执行客户端错误检查。新的 JSON 作业提交格式也将受支持（但目前仍需等待相关变更，因此不可用）。
