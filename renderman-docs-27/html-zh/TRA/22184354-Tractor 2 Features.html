Tractor 2 Features

Source: https://rmanwiki-27.pixar.com/space/TRA/22184354/Tractor+2+Features
Downloaded: 2026-06-15T03:49:12+00:00

Tractor-2.0 为 核心 Tractor 系统 新增了重要功能，该系统建立在 1.0 系列版本的基础上。新增和改进范围很广：既有面向 wranglers 的新命令行与脚本接口，也有简单的用户界面变化；既包括将整个 job database 升级到新的高性能、高能力格式，也包括新的全工作室资源共享与分配控制。

以下是一些重点：

  • 改进的 Tractor 产品布局 -- 单一发布目录、每个平台单一下载包、捆绑的子系统更新 -- 新的 Tractor 2.0 打包和安装布局会把匹配的一组 engine、blade、spooler、用户界面和脚本组件放在同一位置，同时只保留一份共享资源，其中包括多个第三方子系统的预构建版本。

 

  • Tractor 查询工具 -- 这里引入 tq： tractor query 命令行工具和模块。tq 基于经过 Pixar 工作室验证的工具，是从终端 shell、Python 脚本或 Dashboard 的新标签页查询实时 Tractor 数据的最佳方式。

 

  • 自适应农场资源分配 -- 这是一种新方式，可使用 Tractor 灵活的 *Limits* 系统在人员或项目之间动态分配抽象资源。如果两部影片同时制作，可以把 60% 的 farm 分配给其中一部，30% 分配给另一部，剩余 10% 留给其他项目。如果某个 show 空闲，其他项目可以临时扩展自己的份额；当所有项目都处于活动状态时，再收缩回名义水平。

 

  • 调度层级 -- 一种简单方式，可把大范围 jobs 组织成按站点定义的降序优先级组。默认 tiers 名称为： 默认名称为 admin、rush、default、batch。 也可以自行创建！

 

  • Custom 菜单操作（Custom Menu Actions） -- 添加站点定义的 Dashboard 菜单项，用于调用集中管理的脚本，并由用户当前的列表选择作为参数。

 

  • Job Authoring API（作业编写 API） -- 新的 tractor.api.author 模块允许 Python 脚本轻松创建 Job、Task 和 Command 对象，并按照依赖关系需求把它们链接起来。随后 Job.spool() 方法会把生成的 job 发送到 tractor-engine 的 job queue。

  • 简单的 Engine 发现 -- 面向小型工作室的新简单 zero-config 通告能力，允许 tractor-blades 和其他 Tractor 工具在本地网络中找到 tractor-engine，而不需要手动修改 nameserver（DNS/LDAP）配置。在管理员已经在站点 nameserver 数据库中创建主机名别名 tractor-engine 的工作室里，tractor-engine 会自动禁用这种 SSDP-style 流量。

 

  • Checkpoint-Resume（检查点-恢复）支持 -- 对 job scripting、dispatching 和 Dashboard 的新扩展，增加了与增量计算相关的有用能力。Tractor 还支持通用的 retry from last checkpoint 机制。这两项功能都与新的 RenderMan 19 RIS checkpoint 和 incremental rendering 功能集成。

 

  • Blade Auto-Update（Blade 自动更新） -- 新的 tractor-blade 补丁管理系统允许管理员把 farm pin 到特定 blade patch version，并自动把新版本推送到整个 farm。过期 blades 会使用新的模块版本重启自身。

 

  • Pluggable Authentication Module（PAM，可插拔认证模块）支持 -- Engine 新增的可选内置 PAM 支持，会把密码验证直接委托给 engine host 上的操作系统。对于 LAN 已经提供足够凭据传输安全性的工作室，这一替代方案可以很容易地启用密码支持。

 

  • 权限隔离 -- 新的 EngineOwner 设置位于 tractor.config 中，用于指定 tractor-engine 应该以哪个 login identity 运行。这个设置很重要，因为它允许 engine 在获取可能需要的受保护 network ports 后立即放弃 root 权限。随后 engine 的日常正常操作会在指定 login name 关联的限制下进行。

 

  • 动态 Service Key 能力通告 -- 新增了几个 blade profile "Provides" 模式，以支持一些高级用例。例如，blades 可以根据之前启动的 commands 已经 consumed 了哪些 keys，动态通告不同的一组 service key capabilities。

 

  • Resource Usage Tracking（资源使用情况跟踪） -- CPU、RSS 和 VSZ rusage process metrics 现在会记录到每个已启动 command 的 job database 中。目前支持 Linux 和 OSX tractor-blades。

 

  • Command 重试历史 -- 现在每一次 command launch attempt 都会在 job database 中创建唯一的跟踪记录。因此，可以使用 tq 工具查看给定 task 上的重试历史。

 

  • 配置文件加载 -- 简化的 override 系统有助于减少杂乱，并更清楚地显示工作室中哪些文件已经相对原始 factory settings 被修改。

 

  • Task 并发节流 -- 每个 job 都有新的 maxActive 属性，用于限制该 job 中允许同时运行的最大 tasks 数量。这个对 job 在 farm 上 footprint size 的可选控制，在某些情况下可为少量 jobs 提供有用的临时 wrangling 控制，尤其是当修改站点级 Limits 设置并不合适时。

  • Automatic Blade Error Throttle（Blade 自动错误节流） -- 新的 blade profile 设置，如果 blades 在给定时间间隔内遇到过多错误，可阻止它们接收新工作。

 

  • Job Spooling（作业提交）改进 -- Job processing 升级包括更快的处理速度、更好的错误检查，以及必需子系统的捆绑。并行化的 job intake 和 database staging 机制，在大量 jobs 同时 spooled，或大量 expand tasks 并行运行时，可以显著减少 backlog。spooler 捆绑的自包含 Tcl interpreter 简化了站点安装要求，并可在 job 交付给 engine 之前执行 client-side error checking。新的 JSON job spooing 格式也将受支持（但目前仍需等待相关变更，因此不可用）。
