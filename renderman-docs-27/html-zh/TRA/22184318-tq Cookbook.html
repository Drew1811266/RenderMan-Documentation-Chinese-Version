tq Cookbook

Source: https://rmanwiki-27.pixar.com/space/TRA/22184318/tq+Cookbook
Downloaded: 2026-06-15T03:49:12+00:00

可以通过无数方式使用 tq 命令行工具 来查询和管理农场状态。借助强大的 且 覆盖范围可按需要扩展的 搜索子句语法，操作既可以非常精确，也可以覆盖很大范围。

本节提供若干 tq 可能有用的场景。
 

作业队列
 

队列大小

有多种方式可以评估剩余工作量。下面的命令会列出所有尚未调度的任务：

% tq tasks blocked or ready

这里 blocked 和 ready 别名会分别展开为 state=blocked 和 state=ready。

wc 可用于统计它们。使用 --noheader 或 --nh 是为了不把表头包含进计数。

% tq tasks --noheader blocked or ready | wc -l

对于某些站点，结果集可能因为数据库中的配置限制而被截断。这是为了保护 Engine、数据库服务器和客户端，避免极大结果集带来的内存、CPU 和带宽利用率压力。可以使用 --limit 标志覆盖这个限制，或更谨慎地扩大它。

% tq tasks --noheader blocked or ready --limit 100000 | wc -l

这里我们把结果集限制为 100,000 条记录。对于某些队列，这可能仍然不够大。可以把限制设得更高；但对于这类查询，可以使用作业中维护的任务状态计数器。下面的命令会列出包含 blocked 和 ready 任务的作业及其数量。

% tq jobs --nh blocked or ready -c numblocked,numready

配合一些选项和 shell-fu，可以把结果通过管道传给 paste -sd+ | bc 来求和。--nocolor 确保特殊字符不会干扰 bc。--zeros 会让零值显示为 0 而不是空白；当使用 + 连接值形成 bc 的算术表达式时，这一点是必需的。--delimiter + 会让 blocked 和 ready 计数器以 + 分隔，而 paste -sd+ 会用 + 把多行连接起来。

% tq --nocolor --zeros jobs --noheader --delimiter + -c numready,numblocked ready or blocked | paste -sd+ | bc

注意，某些选项出现在 jobs 子命令之前，另一些出现在之后。选项位置可通过内置帮助验证。

% tq help usage % tq help jobs

仍需注意结果集大小可能被自动截断，因此可能需要使用 --limit 选项以确保计算准确。
 

取消删除作业

已删除的作业可以从归档中恢复。第一步是发出 undelete 命令。

% tq --archives jobs undelete jid=1234

注意，--archives 标志 必须 使用，因为只能 undelete 已删除的作业，而 deleted 作业只存在于归档中。

要让该作业运行，必须 restart 作业或 retry 特定任务。

% tq restart jid=1234

这里不需要 --archives 标志，因为该作业已不再被视为 deleted。

资源占用者

有时存在供应有限的特殊计算资源，你可能想了解哪些用户正在使用这些资源。

假设有一些高性能 Blade 可以做 GPU 渲染，且这些 Blade 有一个特殊 profile name gpu。可以按如下方式列出所有带该 profile 的 Blade：

% tq blades profile=gpu

下面的查询会列出所有 active 任务：

% tq tasks state=active

要只获取正在这些 GPU Blade 上运行的任务：

% tq tasks state=active and profile=gpu

（注意：如果忘记 state=active 部分，最终会得到曾经在 gpu Blade 上运行过的所有任务。）

tq 没有内置聚合函数，但可以在命令行上使用 uniq -c（Linux 和 macOS）轻松执行这种聚合。下面的命令匹配与上面相同的任务列表，但只显示任务的 owner。--noheader 避免表头行被计入；--columns owner 使每行只显示 owner 属性；--sortby owner 按 owner 对结果排序，以便 uniq -c 正常工作。

% tq tasks --noheader -state=active and profile=gpu -c owner -s owner | uniq -c

   398 ringo
    10 john
     2 george
     1 paul

现在你可以看到谁在使用 gpu Blade，便可以采取任何适当操作，例如重新调整作业优先级、暂停作业、retry 任务，或调查某些任务是否运行时间超出预期。
 

命令历史记录

Tractor 会跟踪农场上每个命令的执行，包括 Blade 名称、开始时间、停止时间，以及进程的内存和 CPU 利用率。每次执行都表示为一个独立的 invocation。Invocation 由三个属性唯一标识： jid, tid和 iid。 iid 是 invocation id，即调用编号，从 1 开始。

 

查找重试任务

下面的命令会找到特定作业中所有由 retry 产生的 invocation。Retry 可能由用户、通过 API 的脚本，或 Engine 自动发起。搜索字符串周围使用双引号，是为了避免 > 符号被 shell 解释。argv 值太长，无法在默认列宽中显示，因此使用 ... 缩短它。

% tq invocations "jid=52 and iid > 1"

       jid  tid  cid  iid blade      starttime   stoptime    rcode argv
========== ==== ==== ==== ========== =========== =========== ===== ========================================
        52    1    1    2 c2221-8002 01/31|16:40 01/31|16:40     1 expandtest.py --exp...-time=100 --random
        52    4    4    2 c1929-8000 01/31|16:40 01/31|16:40     1 expandtest.py --exp...-time=100 --random

农场管理
 

排空农场

有时需要关闭全部或部分农场，以执行硬件维护或软件升级等管理活动。假设某站点需要升级机架 3 和机架 6 的电源，且这些机架中主机的名称分别以 r3 和 r6 开头。

要排空这些机架上正在运行的任务，我们首先会使用 nimby 命令阻止 Blade 请求更多工作，如下所示：

% tq nimby "name like ^r3 or name like ^r6"

管理员可以使用下面的命令轮询这些 Blade 上正在 active 运行的任务。当命令不返回结果时，他们就知道可以安全关闭这些 Blade。

% tq tasks "active and (blade like ^r3 or blade like ^r6)"

管理员也可以不等待任务完成，而是选择强制 retry 正在这些 Blade 上运行的任务。

% tq retry "active and (blade like ^r3 or blade like ^r6)"

前面的 tq tasks 命令可用于轮询系统，确保这些机架上没有剩余 active 任务，然后即可继续维护。

维护完成后，可以使用与下线时类似的命令让 Blade 重新上线：

% tq unnimby "name like ^r3 or name like ^r6"

一些正则表达式爱好者可能会很快指出 ^r[36] 应该匹配机架 3 和机架 6，这是正确的。不过，搜索子句语法要求包含方括号的表达式放在引号中，以免被解释为搜索子句列表的分隔标记。因此，上面的命令（以及类似的其他命令）也可以写成这样：

% tq unnimby "name like '^r[36]'"
