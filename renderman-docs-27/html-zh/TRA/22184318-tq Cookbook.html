tq Cookbook

Source: https://rmanwiki-27.pixar.com/space/TRA/22184318/tq+Cookbook
Downloaded: 2026-06-15T03:49:12+00:00

可以通过无数方式使用 tq command-line tool 来查询和管理 farm 的状态。借助强大的 并且 覆盖范围可按需要扩展的 search clause syntax，操作既可以非常精确，也可以覆盖很大范围。

本节提供若干 tq 可能有用的场景。
 

Job Queue（作业队列概述）
 

Queue Size（队列大小说明）

有多种方式可以评估剩余 work 的数量。下面的 command 会列出所有尚未 scheduled 的 tasks：

% tq tasks blocked or ready

这里 blocked 和 ready aliases 会分别展开为 state=blocked 和 state=ready。

wc 可用于统计它们。使用 --noheader 或 --nh 是为了不把 headers 包含进计数。

% tq tasks --noheader blocked or ready | wc -l

对于某些站点，result set 可能因为 database 中的 configuration limit 而被截断。这是为了保护 engine、database server 和 client，避免极大 result sets 带来的 memory、cpu 和 bandwidth utilization 压力。可以使用 --limit flag 覆盖这个限制，或更谨慎地扩大它。

% tq tasks --noheader blocked or ready --limit 100000 | wc -l

这里我们把 result set 限制为 100,000 records。对于某些 queues，这可能仍然不够大。可以把 limit 设得更高；但对于这类 queries，可以使用 job 中维护的 task state counters。下面的 commands 会列出包含 blocked 和 ready tasks 的 jobs 及其数量。

% tq jobs --nh blocked or ready -c numblocked,numready

配合一些 option 和 shell-fu，可以把 result pipe 到 paste -sd+ | bc 来求和。--nocolor 确保特殊字符不会干扰 bc。--zeros 会让 zero values 显示为 0 而不是空白；当使用 + 连接 values 形成 bc 的 arithmetic expression 时，这一点是必需的。--delimiter + 会让 blocked 和 ready counters 以 + 分隔，而 paste -sd+ 会用 + 把多行连接起来。

% tq --nocolor --zeros jobs --noheader --delimiter + -c numready,numblocked ready or blocked | paste -sd+ | bc

注意，某些 options 出现在 jobs subcommand 之前，另一些出现在之后。Option 位置可通过 built-in help 验证。

% tq help usage % tq help jobs

仍需注意 result set sizes 可能被自动截断，因此可能需要使用 --limit option 以确保计算准确。
 

取消删除 Job

已删除的 job 可以从 archives 中恢复。第一步是发出 undelete command。

% tq --archives jobs undelete jid=1234

注意，--archives flag 必须 使用，因为只能 undelete 已删除的 job，而 deleted jobs 只存在于 archives 中。

要让该 job 运行，必须 restart job 或 retry 特定 tasks。

% tq restart jid=1234

这里不需要 --archives flag，因为该 job 已不再被视为 deleted。

Resource Hoarders（资源囤积者）

有时存在供应有限的特殊 computing resources，你可能想了解哪些 users 正在使用这些 resources。

假设有一些 high-performance blades 可以做 GPU rendering，并且这些 blades 有一个特殊 profile name gpu。可以按如下方式列出所有带该 profile 的 blades：

% tq blades profile=gpu

下面的 query 会列出所有 active tasks：

% tq tasks state=active

要只获取正在这些 GPU blades 上运行的 tasks：

% tq tasks state=active and profile=gpu

（注意：如果忘记 state=active 部分，最终会得到曾经在 gpu blades 上运行过的所有 tasks。）

tq 没有 built-in aggregation functions，但可以在 command-line 上使用 uniq -c（Linux 和 macOS）轻松执行这种 aggregation。下面的 command 匹配与上面相同的 tasks 列表，但只显示 task 的 owner。--noheader 避免 heading lines 被计入；--columns owner 使每行只显示 owner attribute；--sortby owner 按 owner 对 results 排序，以便 uniq -c 正常工作。

% tq tasks --noheader -state=active and profile=gpu -c owner -s owner | uniq -c

   398 ringo
    10 john
     2 george
     1 paul

现在你可以看到谁在使用 gpu blades，便可以采取任何适当操作，例如重新调整 jobs 优先级、暂停 jobs、retry tasks，或调查某些 tasks 是否运行时间超出预期。
 

Command History（命令历史记录）

Tractor 会跟踪 farm 上每个 command 的执行，包括 blade name、start time、stop time，以及 process 的 memory 和 cpu utilization。每次 execution 都表示为一个独立的 invocation。Invocation 由三个 attributes 唯一标识： jid, tid和 iid。 iid 是 invocation id 调用编号，从 1 开始。

 

查找 Retried Tasks 重试任务

下面的 command 会找到特定 job 中所有由 retry 产生的 invocations。Retry 可能由 user、通过 API 的 script，或 engine 自动发起。Search string 周围使用 double quotes，是为了避免 > sign 被 shell 解释。argv value 太长，无法在默认 column width 中显示，因此使用 ... 缩短它。

% tq invocations "jid=52 and iid > 1"

       jid  tid  cid  iid blade      starttime   stoptime    rcode argv
========== ==== ==== ==== ========== =========== =========== ===== ========================================
        52    1    1    2 c2221-8002 01/31|16:40 01/31|16:40     1 expandtest.py --exp...-time=100 --random
        52    4    4    2 c1929-8000 01/31|16:40 01/31|16:40     1 expandtest.py --exp...-time=100 --random

Farm Management（农场管理说明）
 

Draining the Farm（排空农场流程）

有时需要关闭全部或部分 farm，以执行 hardware maintenance 或 software upgrades 等 administrative activities。假设某站点需要升级 rack 3 和 rack 6 的 power supplies，并且这些 racks 中 hosts 的名称分别以 r3 和 r6 开头。

要让这些 racks 上正在运行的 tasks 排空，我们首先会使用 nimby command 阻止 blades 请求更多 work，如下所示：

% tq nimby "name like ^r3 or name like ^r6"

Admins 可以使用下面的 command 轮询这些 blades 上正在 active running 的 tasks。当 command 不返回结果时，他们就知道可以安全关闭这些 blades。

% tq tasks "active and (blade like ^r3 or blade like ^r6)"

管理员也可以不等待 tasks 完成，而是选择强制 retry 正在这些 blades 上运行的 tasks。

% tq retry "active and (blade like ^r3 or blade like ^r6)"

前面的 tq tasks command 可用于轮询 system，确保这些 racks 上没有剩余 active tasks，然后即可继续 maintenance。

Maintenance 完成后，可以使用与下线时类似的 command 让 blades 重新上线：

% tq unnimby "name like ^r3 or name like ^r6"

一些 regular expression 爱好者可能会很快指出 ^r[36] 应该匹配 rack 3 和 rack 6，这是正确的。不过，search clause syntax 要求包含 square brackets 的 expressions 放在 quotes 中，以免被解释为 search clause list 的分隔标记。因此，上面的 command（以及类似的其他 command）也可以写成这样：

% tq unnimby "name like '^r[36]'"
