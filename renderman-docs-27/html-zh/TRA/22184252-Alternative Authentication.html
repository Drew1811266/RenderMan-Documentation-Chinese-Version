这里讨论不使用 PAM 来管理自定义身份验证的替代方法。

自定义挑战-响应哈希函数

这种方法在下文称为 "Scheme #1"，可以为传输中的凭据提供更好的保护，但需要自定义编码，才能与站点密码数据库正确集成。

背景： 远程登录有两种基本密码方案，应根据站点需要权衡它们的收益和风险。

Scheme #1 - 对哈希密码使用挑战/响应

  • 最佳实践级别的基础保护
  • 不在网络上传输密码
  • 通常需要自定义账号/密码数据库
  • 需要在服务器和客户端上实现匹配的哈希

此方法的优点是不会通过网络发送可恢复的密码，并可防止重放攻击（即有人捕获一次成功登录的网络流量，稍后重放该序列以获取访问权限）。主要缺点是需要在服务器端创建并存储一个特殊的 "account" 数据库。它也很可能需要编写代码，在服务器和每种客户端上实现匹配的哈希函数。

该方案的工作方式如下：在使用系统之前，每个用户密码的哈希版本（见下文）会存储到服务器端数据库中，该数据库可以是磁盘上的简单文件。登录期间，一个新的一次性随机字符串（称为 challenge，挑战值）会发送到 UI，服务器也会保留该字符串的副本。UI 提示用户输入登录名和密码，然后 UI 依次执行两次哈希操作。第一次哈希与在服务器上创建用户账号条目时使用的哈希相同。然后第一次哈希的输出会与服务器的随机 challenge 字符串组合，并再次哈希。第二次哈希的输出会连同用户登录名的明文一起发送到服务器进行验证。服务器通过重新计算第二次哈希来验证登录，其输入包括它之前发送给该客户端的一次性 challenge，以及它自己在密码数据库中已有的该用户预哈希条目。服务器将自己的计算结果与 UI 发送来的结果比较，如果匹配则授予访问权限。

用于创建初始密码数据库的哈希，以及客户端的第一次哈希，是一个单向的加密质量函数，应用于由用户登录名、其选择的密码，以及永久的全站 "realm" 字符串组合而成的字符字符串。realm 字符串可以是公司名称、网站名称或其他本地唯一字符串，它有助于确保攻击者不能使用预计算哈希密码的查找表轻易破解密码数据库。例如，如果用户 "baker" 选择密码 "Cherry3.417"，则哈希函数的输入可能是 "baker|Cherry3.417|example.com"。使用 MD5 或更强函数进行哈希后，此输入字符串会转换为一组新的输出字符。哈希不是随机的；给定相同输入时，它总会生成可重复的输出。良好的哈希函数永远不应是“可逆的”，也就是说，不能通过对哈希输出运行某个函数来恢复输入。此外，良好的哈希会让两个不同输入字符串映射到相同输出的可能性极低。

初始哈希确保用户密码不会以可恢复形式存储在磁盘上，也不会以明文形式通过网络发送。哈希值只能用于与另一个哈希计算结果比较。第二个基于 challenge 的哈希确保传输的字符串本身并非“等同于密码”，也就是说，它不会匹配从服务器密码数据库文件中复制出来的字符串。由于 challenge 字符串每次登录尝试都不同，第二个哈希还有助于确保不能通过重放之前捕获的登录事务来获得登录访问权限。

这里的主要困难在于创建、维护并访问用于服务器端密码哈希的存储方案。LDAP 可能是此用途的方便候选。其次，有时也很难找到并正确实现一个在服务器和所有可能客户端上表现完全一致的哈希算法。

Scheme #2 - 基本的 "/etc/passwd" 服务器端方法

  • 哈希仅在服务器上执行
  • 密码会以编码但可恢复的形式从客户端发送
  • 经典但早已过时的 Unix 传统

如果已有简单的 /etc/passwd 风格密码哈希数据库，此方法更容易实现。缺点是网络嗅探可用于恢复密码，而该密码很可能也是用户的主桌面/系统密码。如果客户端与服务器之间的网络连接本身可信，例如使用 SSL，则此方案更安全。

流程大致如下：提示用户输入登录名和密码后，对话框的客户端处理器会以某种方式编码密码，然后将登录请求发送到服务器。服务器接收请求并解码消息，从而恢复用户刚刚输入的明文密码和用户名。随后服务器对密码运行单向哈希，并将其与此前存储的该用户密码哈希版本比较（当然使用相同算法哈希）。主要弱点来自传输中的密码编码必须可恢复这一事实。还应注意，客户端编码本身可能在源码可见的上下文中实现，例如网页内很容易看到的 JavaScript 函数，因此攻击者很容易看到所用编码算法。

注意，此方案不会因为由客户端而不是服务器在发送密码之前执行单向哈希而得到改进。这样做会使传输中的哈希变成“等同于密码”，就像未哈希的密码存储在服务器上一样，使重放攻击变得非常容易。不过，如果重放攻击不是问题，那么让客户端执行哈希或许可以让 Tractor 使用普通系统 /etc/passwd 数据库，而无需通过网络传输这些密码。然而，尤其对于 /etc/passwd，Dashboard 客户端代码需要外部方式获取每个 passwd 条目中已有的逐用户 "salt" 值，才能计算正确的 crypt-style 哈希。对于少量用户，这些值也许可以以某种自动化方式直接内置到 trSiteDashboardFunctions 脚本中。

启用 Tractor 密码验证

  • 密码验证使用一 对 站点定义函数实现：一个从 Dashboard 客户端的 Web 浏览器发送凭据，另一个在 tractor-engine 服务器上验证凭据。

  • 这对函数协同实现验证方案，示例包括上面概述的两种方法。

  • 客户端代码必须插入位于 Tractor "config" 目录中的 trSiteDashboardFunctions.js 文件。该文件内有一个名为 trSitePasswordHash 的函数，必须修改它，使其返回适当的密码哈希或编码。实际的哈希或编码计算可以在此文件中完成，也可以根据各站点的插件可用性和外部代码约定放在其他位置。如果 trSitePasswordHash 返回 JavaScript 值 null，则 Dashboard 会忽略密码处理。否则，该函数应返回新计算出的字符串，结果会连同用户输入的登录名一起发送给 Engine。

  • 收到登录凭据字符串后，Engine 会调用外部验证程序，并通过 stdin 将这些字符串作为输入传入。要调用的程序在 crews.config 文件中用字典键 SitePasswordValidator 指定。

  • config 目录中提供了一个用 Python 编写的示例验证器脚本。下面是调用它的 crews.config 条目示例：

    "SitePasswordValidator": "python ${TractorConfigDirectory}/trSiteLoginValidator.py"

  • 被调用程序的退出码决定密码和用户名组合是否有效。程序应以 0（零）退出表示成功，以非零退出表示不允许登录。

  • 示例程序 trSiteLoginValidator.py 展示了如何以 /etc/passwd crypt() 的风格对明文密码计算单向哈希，然后将该哈希与从 Unix 调用 "getpwnam()" 取得的值比较；该调用可以访问简单的 /etc/passwd 或 NIS（yp）条目。 注意 这个简单示例假定 Dashboard 客户端函数 trSitePasswordHash() 只是把传入的明文密码作为输出返回，这是为了便于说明；真实实现应强加密密码后再传输，或切换到使用 Tractor 专用密码数据库的 challenge/response 方案（即上文的 "Scheme #1"）。
