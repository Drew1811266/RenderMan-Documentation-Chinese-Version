图像预览

Source: https://rmanwiki-27.pixar.com/space/TRA/22184332/Image+Preview
Downloaded: 2026-06-15T03:49:12+00:00

此功能允许用户在 Tractor Dashboard 界面中通过上下文菜单（右键）选择 “Preview Output Image”，直接显示某个任务创建的图像。图像可以由浏览器自身直接显示，某些情况下也可以通过运行在用户本地桌面系统上的外部应用显示。无论哪种方式，都需要一些站点配置。

摘要 -- 要启用关于图像预览的站点特定策略，管理员应检查并很可能修改配置文件 “trSiteDashboardFunctions.js” 中的默认预览处理器 -- 请查找名为 “trSiteOpenPreviewLink” 的回调函数。

背景

在考虑通过 Tractor Dashboard 进行图像预览时，需要记住几个系统层面的注意事项：

  • 图像文件位于哪里？由作业启动的应用可能运行在用户桌面上，也可能运行在农场中某处。这些应用运行时，会把输出写到哪里？写到它们恰好运行的机器的本地磁盘？写到中央文件服务器？这些文件是否通过某种文件共享“全局可见”？
  • 正在使用哪些图像文件格式？常规输出图像能否由浏览器内置的 “img” 支持直接显示，还是需要特殊预处理或外部显示程序？
  • 谁在使用 Dashboard 界面？想看预览图像的人是否始终是艺术家本人，并且他们在自己的桌面工作站上工作，Tractor Dashboard 浏览器也运行在同一台机器上？如果协调人员、TD 或管理员会从不同用户的作业访问图像，并且从不同网络位置、不同设备访问 Tractor 的 Web 界面，预览方案可能需要不同的配置方案。
  • 默认情况下，浏览器中的安全功能会阻止像 Tractor Dashboard 这样的网页应用启动外部图像显示程序或任何其他进程。这样做会造成严重安全漏洞，因为这意味着随机网页中嵌入的 JavaScript 可以对你的本地机器“做事情”，例如运行可执行文件或读写文件。在评估这类访问的实际价值与风险之间的权衡时，显然需要考虑站点特定因素。

预览方案

通过基于 Web 浏览器的 Tractor Dashboard 显示图像有两种基本方法。最简单也最安全的一种，是让 Web 浏览器自己直接从网络上的 Web 服务器获取图像，就像获取 Web 上的任何其他图像一样。另一种方法是在每个用户的桌面（或其他 Web 浏览设备）上安装自定义图像查看插件，然后指示该插件使用自己的自定义机制来查找、读取并显示图像像素。

第一种情况下，Tractor Dashboard 中的 JavaScript 代码会让 Web 浏览器从 Tractor Engine 或本地网络某处的另一个 Web 服务器 请求图像。浏览器获得图像的 URL 后，会通过 http 在单独的浏览器窗口中自行加载图像。这种方法要求所有可预览的渲染必须生成浏览器可以原生显示的输出图像；例如要注意，TIFF 图像在大多数浏览器中并不被广泛支持。该方案还要求渲染器必须把输出图像写到给定 Web 服务器可读取的位置，也就是说，从 Web 服务器的角度看，渲染结果必须位于 Web 服务器的文档树内。指定预览图像名称的作业脚本必须小心使用输出图像的完整网络路径 （从 Web 服务器的角度看）。 在资产存储于中央文件服务器的站点中，很可能已经在使用这样的网络路径，并且 Engine 使用的 chaser/preview 路径会与艺术家在自己工作站上使用的路径相同。

可以通过向 tractor.config 中的 SiteURLMap 配置表添加条目，使网络挂载在 tractor-engine 的 Web 文档根目录中“虚拟”出现。表中的每一对路径都表示一种映射：从相对于 Engine 文档根目录的引用，映射到文件系统中的其他位置。常见的新增映射会采用简单的 “duplicate” 形式：

“/net/assets/previews”, “/net/assets/previews”,

它会把对 “/tractor/net/assets/previews” 中图像的 http 请求映射到文件系统位置 “/net/assets/previews”。

在 浏览器插件 方法中，Tractor Dashboard JavaScript 代码会要求用户的浏览器打开作业脚本中指定的 chaser URI。这些 URI 通常采用如下格式：

scheme_name:scheme_parameters

这种机制的一个常见示例是，网页中包含类似 “mailto:myfriend@example.com” 的 “mailto” URI 链接。当有人点击该链接时，浏览器会为 “mailto” scheme 使用特殊处理器，通常会要求底层操作系统启动用户偏好的邮件应用。

也可以通过这种 URI scheme 机制调用其他内置或第三方浏览器扩展。Tractor 只是调用浏览器的 “open” 方法打开作业的 chaser 命令中提供的 URI，并假定站点管理员已经协调好作业脚本生成和可用浏览器插件，从而在用户桌面上调用期望的应用。这类插件可以执行许多任意操作，包括读取浏览器主机上的本地文件；这当然可能非常有用，也很容易被滥用。

因此，使用 URI 处理器方法时，预览渲染可以把临时图像写到用户本地桌面上的某个位置，然后 Dashboard + 插件组合可以让功能完整的原生显示应用启动，并直接在桌面上显示图像。在这种情况下，不一定需要全局共享文件，不过如果可用，启动的应用当然也可以使用它们。插件方法的缺点是，你需要为每个工作站平台或其他查看 Tractor Dashboard 网页的设备编译浏览器特定版本的插件。另外，如果你离开工位时尝试预览某个渲染，本地设备上的插件必须能够以某种方式自行访问输出图像，无论该图像位于 Web 的何处。
