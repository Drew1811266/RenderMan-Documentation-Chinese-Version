配置文件（Configuration Files）

blade.config 配置文件指定服务器配置档，每个配置档可应用于一个或多个渲染农场主机。其思路是使用一个集中式配置（策略）文件控制所有机器，并允许一个配置档在必要时应用到多台机器。新服务器可以上线，并使用其匹配到的配置档设置，以简单的即插即用方式加入渲染农场。

blade.config 文件通常会完整复制到站点覆盖目录，然后在那里自定义配置。它包含一个字典，其中有两个顶层键条目：

  • ProfileDefaults - 定义所有配置档的默认值。这里给出的基础值用于初始化 BladeProfiles 列表（见下文）中的每个条目。每个配置档定义可以覆盖这些默认值中的部分或全部。

  • BladeProfiles - 是各个配置档的有序列表，这些配置档可应用于一个或多个渲染农场主机。配置档中的各个条目是对 ProfileDefaults 字典中指定值的覆盖值。
     

配置档匹配（Profile Matching）

  • 启动时，每个 tractor-blade 服务器会自检自身，以找出一些基本特征：

    • 它的主机名和 IP 地址

    • 它的 CPU 核心数

    • 它的 GPU 数量和类型

    • 它的物理内存和磁盘大小

    • 其他基础平台信息

  • 然后，每个 Blade 会从 Engine 请求整个 blade.config 文件。

  • 每个 Blade 会按顺序遍历 BladeProfiles 列表。

  • 与 Blade 基本特征（主机名、平台等）匹配的第一个配置档会 “胜出”：Blade 停止搜索配置档列表，并将该配置档应用为自己的运行定义。

  • 请求工作时，服务器的配置档名称会发送给 Engine，使分配器能决定哪些作业匹配该主机的访问限制，以及哪些命令匹配该服务器的能力。

由于采用 “第一个匹配项胜出”策略，只适用于少数机器的配置档定义应放在 BladeProfiles 列表靠前位置，而兜底通用配置档应放在末尾，确保专用 Blade 会匹配到适当配置档。
 

必需配置键（Required Keys）

BladeProfile 数组中每个配置档对象（字典）都会指定若干 “必需”键。大多数必需键在 ProfileDefaults 字典中指定，并由各个配置档应用覆盖值。下面是一个参考示例：

{

    "ProfileDefaults":
    {   "ProfileName": "default",
        "Provides": ["pixarRender", "pixarNRM"],
        "Hosts": {"Platform": "*"},
        "Access": {
            "Crews": ["*"],
            "NimbyCrews": ["*"],
            "NimbyConnectPolicy": 1.5
        },
        "Capacity": {
            "MaxSlots": 1,    # 0 -> use number of system CPUs
            "MaxLoad":  1.5,  # CPU load avg, normalized by cpu count
            "MinRAM":   1.0,  # gigabytes
            "MinDisk": 20.0,  # gigabytes
        },
        "UDI": 1.0,
        "RecentErrorThrottle": [5, 30, 120],
        #"CmdOutputLogging": "logfile=/fileserver/tractor-logs/%u/J%j/T%t.log",
        "CmdOutputLogging": "logserver=${TRACTOR_ENGINE}:9180",
        #"VersionPin": "@env(TRACTOR_ENGINE_VERSION)",
        # "TaskBidTuning": "immediate",
        "TR_EXIT_STATUS_terminate": 0,
        "SiteModulesPath": "",
        "DirMapZone": "nfs",
        "EnvKeys": [
            {   "keys": ["default"],
                "environment": {},
                "envhandler": "default"
            },
        ]
    },


    "BladeProfiles":
    [
        {   "ProfileName": "BigLinux",
            "Hosts": {
                "Platform": "Linux-*",
                "MinNCPU": 8,      # must have at least 8 CPUs
                "MinPhysRAM": 16,  # must have at least 16 GB
            },
            "UDI": 11,
            "EnvKeys": [
                "@merge('shared.linux.envkeys')"
            ]
        },

        {   "ProfileName": "OtherLinux",
            "Hosts": {"Platform": "Linux-*"},
            "EnvKeys": [
                "@merge('shared.linux.envkeys')"
            ]
        },

        {  "ProfileName": "our_next_profile",
             ...
        }
    ]
}


配置键名术语（Key Terms）

  • ProfileName - 配置档条目的名称；每个配置档都有一个站点定义的抽象标题，应该描述对站点有意义的内容，例如 "Linux_machines"、"8core_rack" 或 "server_of_last_resort"。

  • Hosts - 一个字典，用来回答：此配置档条目应用到哪些主机？

    • 当每个服务器遍历配置档条目时，它们会用自己的主机信息测试 Hosts 字典中的模式，以确定该条目是否适用。

    • Hosts 字典中的每个条目都表示一项测试，并且它们

    • 必须全部通过，配置档才会应用到该 Blade。匹配使用通配符风格字符串比较完成。

    • 字典项 "Name" 指定的模式会与服务器的所有主机名、别名和点分四组 IP 地址比较。"Name" 条目可以是列表，也可以是单个条目。每个条目都可以包含通配符风格字符。注意！这里输入的名称和地址应是 Engine 看到的值，其次才是 Blade 主机自身看到的值；使用 VPN 隧道或其他路由的主机，Engine 和 Blade 上可见的数据可能不同。

    • "Platform" 给出的模式会与每个服务器上由以下 Python 表达式生成的字符串匹配：

      Python 表达式：platform.platform() + '-' + platform.architecture()[0]

    • "NCPU" 指定的值会与主机报告的 CPUs 数量进行精确比较。

    • 使用 "MinNCPU" 可测试主机是否具有至少给定数量的 CPUs。

    • 使用 "MinPhysRAM" 可测试主机是否至少安装了给定 GB 内存。

    • 使用 "PhysRAM" 可测试主机是否正好安装了给定 GB 内存（四舍五入到最近的整数）。

    • 使用 "MinNGPU" 可测试主机是否至少具有给定数量的 GPU 卡。关于过滤，请参见 blade.config 中的 "GPUExclusionPatterns"。

    • 使用 "GPU.label" 可测试主机是否包含特定类型的 GPU。这些是通配符匹配模式，例如 ["*Quadro*K620*", "NVIDIA*"]。

    • 使用 "PathExists" 可测试主机是否存在特定目录或文件。这些可以是已安装的应用程序，也可能是不同文件服务器访问挂载点。这里的值应为路径字符串列表，所有路径都必须存在才会匹配。 

  • Access - 服务器访问限制：哪些作业可以使用此服务器？

    • 这里定义的模式会由分配器与作业特征匹配。

    • 例如 {"Crews": "Project-A"} 只允许由 "Project-A" 用户组成员提交的作业在给定服务器类型上运行，该用户组在 crews.config 文件中定义。

    • 另一个例子 {"Crews": "bob"} 只允许用户 "bob" 提交的作业在给定服务器类型上运行，例如 bob 的桌面机。你可以把用户名指定为用户组，而无需在 crews.config 文件中定义该用户组。

    • 用户组定义可以是单个值或值列表。列表会以方括号定义：{"Crews": ["Project-A", "Project-B"]}。

    • 如果未定义，默认行为是允许所有有效用户访问。

    • 操作顺序：

      1. 作业提交时会带有某些特征：user、priority、metadata 等。

      2. 当作业变为 active（活动，已载入可分配队列）时，加载器会把作业特征与所有当前服务器定义上的 Access 字段进行比较。它会预先准备一张映射，说明哪些服务器类型可以运行每个作业。

      3. 如果定义文件发生变化，发送给 tractor-engine 的信号会导致该文件重新加载，并重新处理访问映射。类似地，仍在使用旧定义的服务器（通过其请求中的序列号检测）会被告知重新获取新文件。

      4. 当服务器请求任务时，会用服务器类型通过访问映射过滤掉某些作业。

    • Access 字典对谁可以修改使用该配置档的 Blade 的 NIMBY 设置提供特殊访问限制。有两个基本控制项："NimbyCrews" 列表定义哪些用户组有权限更改设置，前提是变更请求通过 Engine 路由（默认方式）。"NimbyConnectPolicy" 设置控制脚本是否可以直接连接 Blade 来更改 nimby 状态，还是必须通过 Engine 间接提出请求；使用 1.5 或更大值可只允许经过 Engine 身份验证的 nimby 请求。

  • Provides - 一个抽象服务键名数组，精确列出 Blade 可以接受哪些命令类型。例如，某个特定软件工具可能只安装在所有渲染农场机器的一小部分上，因此可在这些 Blade 的 Provides 列表中添加一个代表该应用程序的名称，表示它安装在那里。想调用该工具的作业需要把相同键名添加到其 RemoteCmds -service 表达式，表示它们需要提供该能力的 Blade。实际使用的键名是任意的，只需要配置档定义和作业之间对这些名称达成约定。例如，按约定，安装了 Pixar 的 RenderMan 渲染器的 Blade 通常用 "PixarRender" 键表示。比较时会忽略键名拼写的大小写。

    • 如果整个作业通过了上面的 Access 测试，则会检查各个 ready（就绪）命令，判断它们是否匹配服务器的 Provides 服务键列表，例如："Provides": ["pixarRender", "pixarNRM"]。

    • Provides 列表中的键名通常只是简单关键字。不过，也有一些高级服务键注解可在不常见情况下应用。

  • Capacity - 一个约束字典，会使服务器停止请求新工作。

    • "MaxSlots": (int) - 服务器上的最大并发命令数。特殊值 '0'可用于自动把 MaxSlots 约束为主机上发现的 CPUs 数量（这是默认值）。

    • "MaxLoad": (float) - 当 loadavg 低于此阈值时继续请求任务。

    • "MinRAM": (float GB) - 当可用 RAM 大于此最小值时继续请求任务。

    • "MinDisk": (float GB) - 当可用磁盘空间大于此最小值时继续请求任务。此键接受两种值格式：

      "MinDisk": 3.75,

      或：

      "MinDisk:[3.75, "/scratch"],

      第二种形式会命名要检查的磁盘。

  • "EnvKeys" - 环境键字典列表。

[
   {
      "keys": ["default"],
      "environment": {
          "REMOTEHOST": "$TR_SPOOLHOST"
     },
     "envhandler": "default"
   },
   {
      "keys": ["prman-*"],
      "environment": {
          "RMANTREE": "/opt/pixar/RenderManProServer-$TR_ENV_RMANVER",
          "PATH": "$RMANTREE/bin:$PATH"
      },
      "envhandler": "rmanhandler"
   },
   {
      "keys": ["JOB=*", "TASK=*", "SCENE=", "PROD=*"],
      "environment": {
      },
      "envhandler": "productionhandler"
   },
]

每个字典都是一个具名 environment（运行环境），并定义一个 environment keys 列表，该列表会与待启动作业传入的 envkeys 匹配。

该 environment 条目包含对命令运行时默认运行环境变量的覆盖值。 envhandler key 包含 Python 运行环境处理器的名称，该处理器会在命令执行前对其进行操作。envhandler 可通过把模块添加到 SiteModulePath 条目定义的目录来扩展。更多详细信息请参阅 envhandler 文档。

  • SiteModulesPath - 定义 Blade 可访问的目录，其中包含附加 Python 运行环境处理器。更多详细信息请参阅 envhandler 文档。
     

  • CmdOutputLogging - 已启动命令输出日志的目标位置。关于此设置的详细信息，请参阅单独的 日志说明。
     

  • VersionPin - 指定一个特定 tractor-blade “模块版本”，使用给定配置档的 Blade 应运行该版本。这些主机会被 "pinned" 到 Blade 软件的特定版本。此设置通常只在 ProfileDefaults 部分中给出，作为一种向整个渲染农场推送 Blade 更新的方式，把所有机器固定到统一的 Blade 模块版本。

    当管理员重新加载 blade.config 时，Blade 会检查其配置档中的这个设置。如果 Blade 发现自己没有运行正确的模块版本，它会从 Engine 下载指定模块，并使用新模块重启自身。Blade 只会获取预打包版本的核心 tractor-blade Python 模块本身；它不会更新整个 Tractor 安装区域，也不会更新随附的 "rmanpy" Python 解释器或其他 Tractor 工具；要升级这些其他组件，Blade 主机上需要完整安装。

    具名模块补丁包必须已经存在于站点 tractor-engine 主机上，通常由管理员放置在那里。每个完整 Tractor 产品安装都包含该版本的 "factory" Blade 模块包；其他 Blade 模块包可能通过客户支持渠道提供。这些包是自包含的 zip 格式模块，使用 ".pyz" 文件扩展名。

    当 Blade 请求固定模块下载时，Engine 会在两个位置查找。首先查找 Engine 的 "--configdir" 目录，也就是 Engine 主机上包含其他站点修改过的配置覆盖的目录。其次查找已安装产品目录中与当前版本匹配的随附包。
     

  • TaskBidTuning - 设置为 "immediate" 时，Blade 会在先前任务成功退出后立即请求新工作，或在新任务已启动且仍有额外槽位容量时立即请求新工作。否则，Blade 会在请求之间更保守地等待（minsleep 秒）。"immediate" 模式会使 Blade 很快轮询所有等待中的快速运行命令。
     

  • TR_EXIT_STATUS_terminate - 控制发出 "TR_EXIT_STATUS nnn" 指令的脚本是最终自行以给定退出状态码覆盖值退出，还是在没有及时退出时由 tractor-blade 主动终止。使用 0 表示等待，使用 1 表示主动终止。
     

  • RecentErrorThrottle - 类似 [5, 30, 120] 的设置表示：同一 Blade 在 30 秒内出现 5 个错误，将导致该 Blade 的新任务分配暂停 120 秒。此上下文中的 "error"（错误）是指 Blade 启动的命令因崩溃而以非零退出状态码或意外信号退出。当列表第三个参数给出 -1（替换上例中的 "120"）时，达到错误阈值的 Blade 会改为把自己置于 "nimby" 状态；在此状态下，它们不会接收新任务分配，直到管理员通过 Dashboard 或 tq 手动清除 nimby 状态。注意，此设置可以放在 "ProfileDefaults" 字典中应用到所有 Blade，也可以在特定配置档中单独放置不同设置（或对默认值的覆盖）。
     

  • DirMapZone - 定义目录映射中用于跨平台支持的区域。Blade 会用默认 DirMapZone 初始化（Windows="unc"，Linux 和 Mac="nfs"）。Blade 配置档中的值会覆盖内部默认值。
     

  • UDI - Universal Desirability Index（通用期望指数）

    • 这是一个抽象的站点自定义评分值，用于在作业已提交且有 Blade 等待工作时帮助选择 “更理想”的服务器。

    • 目前，UDI 只在存在空闲 Blade 时适用；如果所有服务器都忙碌，则任务会分配给下一台可用的 Blade。

  • NIMBY - 将 NIMBY 添加到 Blade 配置档，会定义 Blade 可以拾取哪些作业。设置包括：

    • 1 - 将只接受从该 Blade 主机提交的本地作业。

    • 0 - 将接受来自任何人的作业。

    • username - 将只接受来自指定用户的作业。

    注意，上文描述的 "Access" 字典中的 "NimbyCrews" 设置控制哪些用户（如果有）允许通过 Dashboard 或其他经过会话验证的客户端动态更改此设置。还要注意，当在特定配置档定义中提供任何 "Access" 覆盖时，需要指定完整的 Access 键值对集合；未指定的键将不会从 ProfileDefaults 中的 Access 值继承。
     

  • GPUExclusionPatterns -- 从匹配和计数考虑中排除某些 GPU 类型。注意，此关键字只在 ProfileDefaults 字典中有效，并且 GPU 过滤会在任何单个配置档的匹配测试之前执行。背景：给定配置档可以基于 "Hosts" 子句中的若干条件匹配特定主机，这些条件可包括 GPU 的数量和类型。有些主机包含 “无关”的虚拟 GPU 或性能不足的 GPU，应始终在配置档匹配流程之前排除。使用这里的 "GPUExclusionPatterns" 列表枚举要在计数和匹配中跳过的 GPU 厂商/型号。注意，"GPUExclusionPatterns" 仅限 ProfileDefaults 块（此处）使用；它在各个配置档定义中会被忽略，因为 GPU 计数在匹配之前发生。列表中的每个条目都是一个简单通配符风格模式；不含 '*' 或 '?' 的模式会被视为 "*TEXT*"。"GPUExclusionPatterns": ["QXL", "Standard VGA"],
