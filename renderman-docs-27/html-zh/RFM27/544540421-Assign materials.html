Shader Assignments（着色器分配）

gpuCache 在 Maya 中是单个 shape（形状），因此无法像平常一样分配 materials（材质）。但我们可以使用 RenderMan 的 Dynamic Rules Editor（动态规则编辑器） 在 render-time（渲染时）分配材质，但有以下限制：

  • 所有被引用的 materials（材质）必须存在于场景中。

  • 不支持 per-faceset materials（按 faceset 的材质）。

  • 任何会影响 raytracing core（光线追踪核心）的 attribute（属性）都会被渲染器忽略。这包括 group membership definitions（分组成员关系定义）、selective lighting（选择性照明）等。

  • 任何 rule change（规则更改）都需要重启 IPR 才会生效。

  • 这些限制将在后续版本中解除。

RLF Editor overview（RLF 编辑器概览）

打开 Dynamic Rules Editor（动态规则编辑器），看看它的工作方式。

你可以点击 shelf（工具架）中的双尖括号图标，或点击 gpuCache attribute editor（属性编辑器）中的 Edit RLF File（编辑 RLF 文件） 按钮位于 gpuCache 的 attribute editor（属性编辑器）中。shelf 按钮会编辑场景中的所有 alembics，而每个节点上的按钮只会编辑与该节点关联的 RLF。

Rules management（规则管理）

左侧按钮负责所有 rule（规则）和 file management（文件管理）任务：

  • Add new rule（添加新规则）：创建一条新的空规则

  • Delete selected rule（删除所选规则）：点击规则编号选择一条或多条规则，然后按此按钮。

  • Duplicate selected rule（复制所选规则）：点击规则编号选择一条或多条规则，然后按此按钮。

  • Load RLF file（加载 RLF 文件）：用磁盘上的文件替换编辑器内容。（仅 scene editor，场景编辑器）

  • Save RLF file（保存 RLF 文件）：将编辑器内容保存到磁盘。

  • Auto-save changes（自动保存更改）：当图标为橙色时自动保存到磁盘。（仅 node editor，节点编辑器）

Rules order（规则顺序）

规则会按照编辑器中的顺序求值。若要更改顺序，请点击规则编号并拖动到新位置。

Rule controls（规则控制）

Enable/disable（启用/禁用）：当 toggle（开关）为绿色时，会对该规则求值。

Stop/continue（停止/继续）：当图标为叉号时，如果此规则匹配，规则求值会停止。向下箭头图标表示即使此规则匹配，也会继续求值后续规则。

Material（材质）：材质对应的 shading group（着色组），它会引用 surface（曲面）和 displacement shaders（置换着色器）。

Attributes（属性）：插入或覆盖 attribute values（属性值）。点击 new（新建）/edit（编辑） 图标以打开 attribute dialog（属性对话框）。

  • 点击 + 图标，并右键点击 attribute field（属性字段）以选择一个受支持的 attribute（属性）。

  • 像平常一样选择一个或多个 attributes（属性），然后点击 - 图标将其删除。

  • 具有固定可接受值列表的 attributes（属性）也可以通过右键点击来设置。

  • 保存窗口时，settings（设置）会自动保存。

Match type（匹配类型）：有 3 种 matching engines（匹配引擎）：

  • exact：这是最快的方式，并且只匹配完整路径。

  • glob：基于 glob matching（glob 匹配），使用以下 tokens（标记）：

    • *：任意数量的字符

    • ?：单个字符

    • **：hierarchy（层级）中的任意深度

  • re： re2 regular expression（正则表达式）。这是最复杂、最精细，也可能最慢的匹配方法。

Path expression（路径表达式）：你可以双击字段来输入表达式，也可以从 alembic node list（Alembic 节点列表）拖放。

若要拖放，必须先在 alembic node list（Alembic 节点列表）中选择节点。

Notes（备注）：当事情变复杂时，可在此字段中写备注...

一条规则可以分配：

• material（材质）

• attributes（属性）

• 或两者都分配！

About attributes and node types（关于属性和节点类型）

在 RenderMan 中，每个 shape（形状）都是一个 instance（实例）。这意味着附加到特定 shape 的 attribute（属性）会影响该 shape 的所有 instances（实例）。

如果你想把 attribute（属性）分配给 instance（实例），请创建一个匹配该 instance transform（实例变换）的表达式。这对于类似 maxspeculardepth, maxdiffusedepth, relativepixelvariance等 classic attributes（经典属性）尤其重要。

In-scene shading（场景内着色）

Dynamic rules（动态规则）使用类似 shell 的 glob expressions（glob 表达式）编写。更多信息请参见下方的 Dynamic Rules CookBook（动态规则指南）。

这是我们的起点：gpuCache 节点使用默认材质渲染：

为 archive（归档）的一部分分配 base material（基础材质）

我们准备了一些可供分配的 basic materials（基础材质）。先把 hull_srf 材质分配给整个 teapot（茶壶），并写下第一条规则：

  • 在 node list（节点列表）中，选择我们要分配的 hierarchy（层级）的顶层节点：Rolling_Teapot。

  • 点击 Add Rule（添加规则）（蓝色 + 图标）来创建新规则，或使用初始的空规则

  • 双击 expression field（表达式字段）进入编辑模式。此字段是 >_ 列。

  • 转到 Alembic node list（Alembic 节点列表），并用鼠标中键将所选节点拖入 expression field（表达式字段）。

    • 请注意，拖放之前必须先在列表中选择一项。（你的窗口也必须 docked（停靠），不能在 Windows OS 上浮动；切换焦点会导致无法用鼠标中键拖动。）

  • 将表达式从 /Rolling_Teapot_Grp/Rolling_Teapot 编辑为下面这样：

    /**/Rolling_Teapot

    • ** 表示匹配任意嵌套深度（包括没有嵌套），而 * 只会匹配 / 分隔符之间的字符

    • 按 Enter 验证。

  • 右键点击 Payload（载荷） 字段（带 sliders 图标），然后选择 hull_srfSG。

  • 启动 IPR 进行测试。

Match sub-strings in a hierarchy（在层级中匹配子字符串）

现在我们要为 teapot（茶壶）上的 rivets（铆钉）着色。我们需要更复杂的表达式。

  • Duplicate the first rule（复制第一条规则）

    • 右键点击规则编号（本例中为 0），然后选择 Duplicate Rules（复制规则） （位于 “-” 减号图标正下方的 copy 图标）。

  • 扩展前一个表达式，以匹配任何包含单词 "rivet" 的 object（对象）。

    /**/*rivet*/*

    • 第二个 ** 表示 rivet object（铆钉对象）可以嵌套在 teapot 下的任意层级

    • * 如果节点名称中出现 rivet，则 rivet * 会匹配。"rivet"、"Arivet" 和 "Brivet2" 都会匹配。

  • 在该模型上，还有一些 screws（螺丝）可以使用相同材质。复制表达式，让它们也能匹配。

    /**/*screw*/*

Rule execution order（规则执行顺序）

我们来为 teapot 的 rings（环）分配材质。

使用相同工作流：复制规则，将 "rivet" 替换为 "ring" 并将 Payload 设置为 ring_srf。但这并不完全奏效，因为如你在 node list（节点列表）中所见，有些 shapes（形状）名称中同时包含 "rivet" 和 "ring"，而 rings（环）上的 rivets（铆钉）现在使用了错误的材质。

如果结果不正确，我们可以更改 rule execution order（规则执行顺序） （你的顺序可能与图中我的顺序不同）：如果先匹配 rivet，匹配引擎在找到匹配后会停止，随后只有仅包含 "ring" 的 shapes 才会匹配第二条规则。

  • 将光标放在规则编号上，然后用鼠标左键拖动 rule 1（rivets）到 rule 0（ring）上方。

  • 渲染以检查结果。

当然，我们只是浅尝辄止，建议你阅读 Regular Expressions（正则表达式） 以了解更复杂的用法。

如果你已完成 shading（着色），请选择 disk/save RLF（RenderMan Look File）图标。导入 asset（资产）时可以使用它，每次分配 shading nodes（着色节点）。这也可以成为你的 asset publishing scheme（资产发布方案）的一部分。

Self-contained Asset Shading（自包含资产着色）

如果你想导入已经完整着色的 assets（资产），有 2 个选项：

  • 当 RfM 在与 archive 相同的位置找到同名 RLF 文件时，它会自动应用该文件。

    • 这在 pipeline（流程管线）中并不总是很灵活，但对于归档资产可能很有用。

RLF 文件格式不包含 material descriptions（材质描述），因此你必须确保：

  • 所有被引用的 materials（材质）都存在于你的场景中。

  • Output All Shaders（输出所有着色器） 已在 render globals（渲染全局设置）的 Advanced（高级）部分中勾选。

  • 如果 gpuCache 节点的 RLF File Name（RLF 文件名） 字段包含指向 RLF 文件的路径，RfM 会将其应用到该 gpuCache 节点。

    • 这样就很容易用脚本进行 look assignment（外观分配）和更新。

请注意，你仍然可以在场景中使用 Dynamic Rule Editor（动态规则编辑器）覆盖自动 RLF assignments（RLF 分配）！

在这种情况下，scene RLF rules（场景 RLF 规则）会先于 archive 的规则执行，因此有机会覆盖它们。

Notes about RLF export（关于 RLF 导出的说明）

当你在场景中引用多个 RFL files（RFL 文件）时，必须确保所有 shading nodes（着色节点）都有唯一名称。如果多次定义相同名称，渲染器会使用排在最前面的那个，你会得到意外结果。

你可以通过点击 RLF Editor（RLF 编辑器）中的  图标导出 RLF 文件，也可以编写一个简单的 Python 脚本，将场景当前的 RLF 数据保存到文件：
