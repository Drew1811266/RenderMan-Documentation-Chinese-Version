Configuring RfK with Custom Ops（使用自定义 Ops 配置 RfK）

要配置 RfK 使用 custom Ops（自定义 Ops），必须构建一个配置文件，用于指定你的 Ops 以及它们应被求值的顺序。该文件：

$KATANA_RESOURCES/RfK/config.xml

会从 KATANA_RESOURCES 路径中某个条目的相对任意位置读取。结果会按照读取顺序合并。

当前配置支持指定额外的 "terminal Op"（终端 Op）实例，这些实例会添加到场景中，并对渲染以及 live render（实时渲染）更新观察可见。

config.xml 文件格式为：

<rfkConfig format="1">
    <!-- Ops which affect the scene as seen at ribgen time -->
    <renderTerminalOps>
        <op type="SomeOpName" insertWhen="beforeVstructExpansion"/>
        <op type="AnotherOpName" insertWhen="last" needsSystemArgs="true"/>
        <op type="AnOpWithArgs" insertWhen="last">
            <attr type="GroupAttr">
                <attr name="taco" tupleSize="1" type="IntAttr">
                    <sample size="1" time="0" value="1 "/>
                </attr>
                <attr name="cheese" tupleSize="1" type="StringAttr">
                    <sample size="1" time="0">
                        <str value="salsa"/>
                    </sample>
                </attr>
            </attr>
        </op>
   </renderTerminalOps>

   <!-- Ops which are evaluated during live render updates -->
   <liveRenderTerminalOps>
        <op type="SomeOpName" insertWhen="beforeVstructExpansion"/>
   </liveRenderTerminalOps>
</rfkConfig>

Ops 可以作为 renderTerminalOps 或 liveRenderTerminalOps 添加。每种类型内的 Ops 格式相同，唯一差别是 Katana 对这些 Ops 求值的时机。

renderTerminalOps

此处添加的 "op" 元素会影响渲染时所看到的场景。

liveRenderTerminalOps

此处的 "op" 元素会添加到 Katana 进程内的 live render observer（实时渲染观察器）。如果需要以 live render 更新特有的方式操作场景，可以在这里完成。

一个 "op" 元素有两个必需属性：

  • type 是包含已注册 Op 类型名称的字符串；insertWhen 是一个字符串，用于指示该 op 应插入到 RfK 自身添加的 terminal ops 中的哪个位置。有效字符串值列表见下表。

一个 "op" 元素可以有以下可选属性：

  • needsSystemArgs 的值为 "yes"、"true" 或 "1" 时，会把 "system" GroupAttribute 添加到 Op 的参数中。它包含当前帧和快门信息。

一个 "op" 元素也可以包含类型为 "attr" 的子元素。它应当是 FnAttribute::GroupAttribute 的 XML 序列化内容，可通过 attr 的 getXML() 方法从 Python 或 C++ 生成。

Valid insertWhen Options（有效的 insertWhen 选项）

Option（选项）

Description（说明）

first

位于所有其他 terminal Ops 之前

beforeVstructExpansion

在 virtual struct connections（虚拟结构连接）展开之前，即调用 Op PRManResolveVirtualStructConnections 之前

afterVstructExpansion

在 virtual struct connections（虚拟结构连接）展开之后，即调用 Op PRManResolveVirtualStructConnections 之后

last

将你的 Op 附加到所有其他 Op 之后。

Supported Configuration Options（支持的配置选项）

除了 renderTerminalOps 和 liveRenderTerminalOps 钩子外，RfK 插件还会识别几个其他专用选项。前两个是简单的配置/扩展选项。最后两个允许把非标准信息从 args 文件传递到 custom terminal Ops（自定义终端 Ops）。

shaderPathRecursionLimit
搜索着色器搜索路径时，默认会向下递归 3 层后停止在该路径中的搜索。可以通过 config.xml 中的 shaderPathRecursionLimit 选项配置此行为。将该值设为 0（零）会在搜索着色器路径时禁用递归。

config.xml 中的示例条目：

<!-- Turn off shader search path recursion →
<int name="shaderPathRecursionLimit" value="0"/>

shaderPostProcessFunctionName
着色器加载并配置后，可以使用一个函数来修改该着色器的 "info" 属性。shaderPostProcessFunctionName 指向一个 attribute function（属性函数），该函数接收一个 group attribute（现有的 info 属性）并返回一个新的 info 属性。

config.xml 中的示例条目：

<int name="shaderPostProcessFunctionName" value="MyFunc"/>

hintPassthroughNames
可以使用 shader parameter info 机制（PRManAddShaderParameterInfoOp）和 hintPassthroughNames 列表，将 Non-standard（非标准）parameter hints 从 args 文件传递到 Op。 

outputTagPassthroughNames
可以使用 shader parameter info 机制（PRManAddShaderParameterInfoOp）和 outputTagPassthroughNames 列表，将 Non-standard（非标准）的每着色器、每输出 tags 从 args 文件传递到 Op。
