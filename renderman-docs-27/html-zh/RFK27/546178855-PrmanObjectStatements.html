PrmanObjectStatements

Source: https://rmanwiki-27.pixar.com/space/RFK27/546178855/PrmanObjectStatements
Downloaded: 2026-06-15T03:49:12+00:00

PrmanObjectStatements 会分配到一个特定 location 或一组 locations，并在转换为 prman 时设置一个 AttributeBegin/AttributeEnd block（块）的状态。此节点可用于为对象分配 subdivision approximation（细分近似）以渲染平滑表面、为 displacement（置换）渲染应用 displacement bound（置换边界），以及更改 visibility flags（可见性标志）和 memberships（成员关系）。


在 RenderMan 中，可以通过不同方式选择对象及其实例的渲染方式。这些设置在该节点中方便地分为：

Section（部分）

Description（说明）

attributes

可随 instance（实例）变化的 Attributes（属性）（例如 lighting subset 或 camera visibility)

primAttributes

被视为 "prototype"（原型）属性的 per-primitive attributes（按图元属性）（在下表中称为 "Primitive Variables"）

primvars

属于 primitive（图元）定义的 constant per-primitive attributes（常量按图元属性）。这些属性不能按 instance 改变。

traversal

用于 scene traversal（场景遍历）的 RfK plugin controls（插件控制）。


这些设置列在下面的表格中。注意，这些表格目前并不与 PrmanObjectStatements UI 中参数的呈现方式完全对齐。

Traversal

当 RfK 遍历 Katana scene graph 时，它会并行处理 locations，并将它们传递给 RenderMan 进行渲染。默认情况下，从 Katana 取得的每个新的 child location 都会生成一个新线程进行处理。这个默认行为不太可能为任意给定 scene 提供最佳性能；不过 PrmanObjectStatements 提供了用于调优特定 pipeline（例如你所在工作室的标准 Op Chain）或多种 scene 类型（例如许多 instances、许多 textures、许多 primitives，或三者都有）中的效率和 parallelism level 的设置。

int

0

强制此 location 及其 child locations 的 traversal 使用 serial processing。

Force Serial

第一个工具是用于控制 scene 中 parallelism depth（并行深度）的 attribute： forceSerial。

例如，考虑一个 scene：

/root/world/geo/pine_tree
/root/world/geo/elm_tree
/root/world/geo/oak_tree

其中每个 tree location 都包含 branches、leaves、roots 等。

理想情况下，应在足够高的层级并行 traversal，避免让每片 leaf 或每个 branch 都在不同线程中处理。在 RfK 中，我们通过在希望停止并行处理的 location 上将 "prmanStatements.traversal.forceSerial" 设置为 True。当 traversal 期间遇到该 attribute 时，RfK 会在该 location 切换为 serial，并且所有 child locations 都会在单个线程中运行。

在上面的示例中，将 "/root/world/geo/*_tree" 上的 "forceSerial" 设为 true，意味着每棵 tree 会在不同线程上处理，但每棵 tree 的所有内容会在其各自线程中串行处理。你需要评估自己的 scene setup，判断在哪个层级强制 serial processing 才合理。

每个线程都有自己的 Geolib runtime，其中包含 cooked locations 的 per-thread caches。因此，任何 Op 所需的所有 locations 都必须按线程 cooked。继续上面的示例：如果有三个线程分别查看 "/root/world/geo" 下的不同 locations，你会看到 "/root" 被 cooked 三次，"/root/world" 被 cooked 三次，"/root/world/geo" 也被 cooked 三次。如果以 serial 运行，或在 "/root" 处将 parallel traversal 强制为一个线程，那么只有一个 Geolib runtime 会 cook 该 location，因此你只会看到该 Op 在 "/root" 处打印一次。

Evict Here

我们用于提高 parallel traversal efficiency 的第二个工具是 evictHere 设置。 

默认情况下，当一个 Katana scenegraph location 被处理并转换为 Rix scenegraph 时，该 location 的 siblings 会从线程的 runtime memory 中 evicted。若已知这些 siblings 最好保留在 cache 中供后续访问，则将此 flag 设为 false 可防止在该 location 进行 sibling eviction。

Flatten Instance Source

最后一个工具是 flattenInstanceSource 设置。

此设置并不会改变 RfK 对 Katana scene 的 parallel traversal 行为，而是在构建 instances 时改善 RenderMan 的 traversal。默认情况下，RenderMan 会将一个 instance source 扁平化为 group 中每个 geometric primitive 对应的独立 object prototype。随着 instances 数量增加，或 instance source 内部 primitives 数量增加，这可能变得昂贵。将此 option 设为 Yes 会告诉 RenderMan 将这个 instance source location 保持为单个 object prototype。

此设置会覆盖 prmanGlobalStatements.plugin.flattenInstanceSources 的 global setting（全局设置）；该 global setting 可以在所有 instance source locations 上启用此设置。
