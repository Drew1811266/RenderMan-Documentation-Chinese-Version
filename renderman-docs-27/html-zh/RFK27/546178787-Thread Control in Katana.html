设置 Traversal Thread Count（遍历线程数）

当 RfK 将 Katana scene graph（场景图）转换为用于渲染的 Rix scene graph 时，会并行执行，并在 location 分支处生成新线程。默认值是 "all on"，意味着它会在 scene 中每个子 location 上 fork；这对大多数场景未必理想，但允许作者更深入地控制 parallelism（并行度）。控制层级有多种，需要一些调整，才能得出适合你的 pipeline 以及后续各个 scene 的配置。 

  1. prmanGlobalStatements.plugin.traversalMethod: 全局 traversal（遍历）设置。当设为 'Serial' 时，它会完全禁用并行 scene traversal。通常不建议这样做，但它对计时对比和 reentrant diagnostics（可重入诊断）很有用（也就是说，我在串行和并行 traversal 中都能看到这个问题吗？）。

  2. prmanGlobalStatements.plugin.maxTraversalThreads: 请求用于 traversal（遍历）的线程数：这是开始调优的首要位置。默认 max 设置为 4，但根据机器配置和 scene layout，此级别的性能可能提升一倍，也可能减半。对技术用户来说，这个值用于初始化 RfK 的 TBB task scheduler。将它设置为 TBB 默认值 '0' 不建议。根据经验，将它设为 '0' 时我们见过明显减速，可能是因为与正在使用的其他 TBB sessions 冲突。

  3. prmanStatements.traversal.forceSerial: 用于按 location 对 scene 或 asset 进行精细调优的设置。强烈建议在 component 或 group 层级将它设为 true，使 traversal 不会尝试一直并行化到 gprim 层级。更多关于 location-level thread tuning 的细节，请参见 PrmanObjectStatements。

性能说明

请理解，某些分支以串行方式处理可能会改善 time to first pixel（开始发送像素前所需的时间），而且这高度依赖 scene；同时也要理解，对所有内容都使用并行 traversal 可能代价较高。


设置 Render Thread Count（渲染线程数）

目前在 RfK 中有三种设置 render threads（渲染线程）数量的方法。前两种会向 prman 的命令行参数添加 "-t:#"；按照 RenderMan 对 prman 参数的定义，这会覆盖 RIB 中的任何线程规格（第三种选项）。

设置 thread count（线程数）的方法， 按优先级顺序：

  1. Katana preferences（偏好设置）： interactiveRenderThreadsOverride 和 interactiveRenderThreads3D（等同于在 batch mode（批处理模式）中使用 "--threads3d" 命令行参数）。

  2. Katana attribute（属性）： renderSettings.renderThreads.

  3. PRMan option（选项）： prmanGlobalStatements.options.limits.threads。

该 attribute（属性）会直接转换为 RIB：

由于它在优先级列表中最低，如果 (1) preferences（偏好设置）或 (2) render settings（渲染设置）启用了线程值，这个 thread option（线程选项）就会被覆盖。

特殊值

在所有情况下，如果给 prman 的值为 0（zero），则会使用所有 CPU 进行渲染。如果指定负值（-N），则 prman 会使用除 N 个 CPU 之外的所有 CPU 进行渲染。
