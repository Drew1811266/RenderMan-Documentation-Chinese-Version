Stats in Katana

Source: https://rmanwiki-27.pixar.com/space/RFK27/637632529/Stats+in+Katana
Downloaded: 2026-06-15T03:49:12+00:00

RenderMan diagnostics are available in Katana as live stats and offline JSON reports. Access to more advanced configuration is also available with the use of an environment variable or through a specialized OpScript.

Live Stats

The PrmanLiveStats Tab displays a live overview of metric data from an active Katana render.

It displays several key metrics in real time, including process memory, elapsed wall clock time, and render progress. The window will automatically connect and disconnect as renders start and stop. When a render completes or is canceled, its data remains in the live stats view until the next render starts or a new scene loads.

See Live Stats Overview for detailed information about the contents of this window.

JSON Report

To generate an end-of-render diagnostic report, set the PrmanGlobalStatements JSON Report Filename attribute to the desired output filename. (prmanGlobalStatements.options.statistics.jsonFilename).

See the Diagnostic Reportdocumentation for detailed information on the configuration and interpretation of this report.

Custom Configuration

For more control over diagnostic output, RfK can be directed to use a configuration file using the environment variable $RMAN_STATS_CONFIG_PATH. Additionally, RfK supports customization of the configuration file search path and/or the configuration file name via specialized attributes:

prmanGlobalStatements.stats.configPath (default: “$RMANTREE/etc")

prmanGlobalStatements.stats.configFile (default: "stats.ini")

If set, these will take precedence over other options. The “stats” attributes are not exposed in PrmanGlobalStatements; they must be set via AttributeSet or OpScript. Below is an OpScript that exposes these two attributes as user args with defaults as listed above. Copy and paste into your Katana scene and modify the user attributes configPath and configFile name as needed.

RfK will search for a configuration file with the following precedence:

  1. The prmanGlobalStatements.stats options take precedence over all other options (see OpScript below).

  2. $RMAN_STATS_CONFIG_PATH/stats.ini if the environment variable is set.

  3. $RMANTREE/etc/stats.ini

      <port name="out" type="out"/>

        <string_parameter name="location" value="/root/world/location"/>

Troubleshooting

Normally, the stats system does most of its work quietly behind the scenes. However, for those situations when things aren't working as expected, there are built-in mechanisms to get diagnostics on the diagnostics system.

First, understand that you cannot disable stats collection. This functionality is embedded in the renderer and remains “always on”. However, you can control how the diagnostic data is delivered and presented.

The stats.ini file used to configure the stats system includes flags to increase verbosity during config file loading and subsequent processing. If you are not explicitly using a custom stats.ini file, the default configuration will come from $RMANTREE/etc/stats.ini.

In your config file, set the stats debug logLevel to 5:

# Stats processing log level.
# Range is 0 (none) to 5 (debug output). Default is 3 (warnings)
logLevel 5

You should see an uptick of RZ0000 output to the Render Log, which will help you verify it's picking up the stats.ini file you expect. It will also tell you whether or not the WebSocket stats server is enabled for live stats.

RZ0000 {INFO}   : SessionConfig - Loaded stats config from file: '/path/to/stats.ini'.
RZ9000 rfk_statsserver_<renderboot_PID>
RZ0000 {INFO} : Session - Created stats session: 'Stats Session Name'.
RZ0000 {DEBUG} : Session - WebSocket stats server not enabled
RZ0000 {DEBUG} : Session - Loading listeners​

If you do not see these log messages after turning up the log level verbosity, you are likely picking up a different stats.ini configuration file. Check for an RMAN_STATS_CONFIG_PATH environment variable, or path/filename values from the prmanGlobalStatements.stats attributes.

Please refer to the general Troubleshooting document for further help diagnosing stats system issues.
